## 1. 命题与边界

### 1.1 Feed Mode 命题

将首页从「分区仪表盘」改为 **沉浸式 Feed / Timeline**，且时间轴在叙事上明确定位为 **AI 工作台时间轴**：展示与「用户输入 → AI 处理 → 可消费产物」相关的 **里程碑时点**，而不是通用社交时间线。

底层对象模型（File / Session / Turn / Asset / Contact）**不改变**，仅改变 **首页聚合与导航隐喻**。

### 1.2 目标（Demo 要证明什么）

| 目标 | 说明 |
|---|---|
| 双线叙事 | 同一实体链上能读出 **资产进入系统的时点** 与 **AI 完成某阶段处理的时点**（见 §3.0）。 |
| 异步可见 | 长任务（如会议：入库 → 转录/解析完成）拆成 **多条时间轴项**，而非仅在详情页可见。 |
| 可下沉汇总 | Files / Workspace（或等价汇总页）作为二级入口，承接浏览与管理，而非首页主叙事。 |
| 与主 Demo 对照 | 便于和产品、研发讨论：分区首页 vs 时间线首页，哪一种更符合 Eureka 3.0 心智。 |

### 1.3 非目标

| 非目标 | 说明 |
|---|---|
| 不替代 DEMO_SPEC 主链路证明责任 | 深度 Session 工作台、Skill 细节仍以 `agentic-flow-demo.html` 为主。 |
| 不接后端 | Mock / 本地状态即可。 |
| 不做终极视觉稿 | 「沉浸式」仅表达密度与层级倾向（焦点、分组、筛选），不定稿组件像素。 |
| 不一次性解决所有排序算法 | 可先约定规则（见 §5），复杂优先级后置。 |
| **AI 主动推送** | 未来可能有 AI 主动向工作台投递的内容；**当前 Spec 不展开**（见 §3.4）。 |

## 3. AI 工作台时间轴：展示内容归类

时间轴上出现的每一条 **item** 表示一个 **有意义的时间点**（通常对应一次状态跃迁或产物就绪），而不是「每一次后台心跳」。

### 3.0 核心范式：同一文件的两段时点（已定）

以 **文件类输入**（录音 / 图片 / Markdown 等）为例，工作台时间轴至少区分两类记录：

| 时点类型 | 含义 | 典型文案方向 |
|---|---|---|
| **入库 / 创建时点** | 文件上传成功，在系统中形成可追溯的 File（资产主体已创建） | 「已上传」「已进入文件库」 |
| **AI 处理完成时点** | 例如转录（ASR）完成，或用户触发后的 **解析管线完成**（与 DEMO_SPEC 中 Meeting 门禁一致：未解析前只有入库项；解析完成后新增一条） | 「转录完成」「解析完成」「已生成摘要 / Notes」 |

**结论**：同一 `file_id` 上可以存在 **多条** 时间轴 item，分别对应 **创建时间点** 与 **解析（或转录）完成时间点**；二者不可合并为一条，否则无法表达异步间隔与状态。

> 实现提示：`thread_key = file_id`（或等价关联键）可将多条 item 在 UI 上做成「同一卡片线程」或「相邻成组」，但 **数据上仍是多条时点记录**（见 §6）。

### 3.1 建议的事件维度（字段）

每条时间轴项至少包含：

| 字段 | 说明 |
|---|---|
| `occurred_at` | 排序主键：该里程碑实际发生时刻（入库时刻 vs 解析完成时刻分别填写）。 |
| `milestone_kind` | 如 `file_ingested` \| `ai_transcribe_done` \| `ai_parse_done` \| …（具体枚举随管线细化）。 |
| `actor` | `user` \| `agent` \| `system`（用于区分「用户触发」与「AI 产出」）。 |
| `surface` | 业务域：flash / meeting / upload / conversation / workspace… |
| `status` | 若该条为进行中占位（可选）：pending / running / done / failed。 |
| `refs` | `file_id` / `session_id` / `asset_id` 等，点击跳转详情。 |
| `thread_key`（可选） | 同一 File / Session 的多条里程碑折叠展示（见 §6）。 |

### 3.2 用户侧里程碑（示例）

- 文件上传成功（生成 §3.0 入库类 item）  
- 打开会议详情并点击「开始解析」（若产品决定该点击也单独占一条，可为 `user_triggered_parse`）  
- 新建 Ask / 发送 turn（是否进工作台轴：见 §5.5）  

### 3.3 AI 侧里程碑（示例）

- 转录完成（相对入库为第二条 item）  
- 解析完成、结构化产物写入 Workspace（可一条汇总或拆条，见 §5.2）  
- Flash：结构化完成 → Todo / Idea 写入  

### 3.4 延后：AI 主动推送

未来可能存在 **AI 主动推送** 到工作台的内容（无需用户当刻上传即出现一条）。**当前版本 Spec 不展开**：不设枚举、不设 Demo 交互；仅在架构上保留「时间轴 ≠ 仅被动跟随用户操作」的可能性。

### 3.5 AI 生成的 Asset：要不要每条都上时间轴？

文件解析 / Flash 结构化完成后，系统往往在 Workspace 里产生 **多条 Asset**（Note、Todo、Idea、Contact 候选等）。是否在时间轴上为 **每一个 Asset** 再记一条「生成记录」，与 §3.0 的 **管线级里程碑**（如「解析完成」）存在粒度冲突，建议按下表分层决策：

| 策略 | 含义 | 适用 |
|---|---|---|
| **A. 仅管线里程碑** | 时间轴只保留「解析完成」类汇总一条（或入库 + 解析两条），**不**为每个 Asset 单独占一行 | 默认推荐：轴上清爽，明细进 Workspace / 文件详情 |
| **B. 管线 + Asset 拆条** | 除里程碑外，每个新生成 Asset **各一条** item，`refs.asset_id`，可与 `thread_key=file_id` 同组折叠 | 强调「产物可追溯」、或运营需按条跳转时 |
| **C. 混合** | 里程碑一条 + 仅对「需用户确认」类（如 Contact 候选）单独拆条 | 与 DEMO_SPEC 中待确认交互对齐 |

**原则**：若同一解析批次内 Asset 数量多，**B** 易造成时间轴刷屏；优先用 **A** 或 **C**，必要时在汇总卡片内展示「本次生成 3 Notes · 2 Todos」并跳转详情。

### 3.6 用户手动创建的 Asset：要不要上时间轴？

**建议：要。** 工作台时间轴若只反映 AI 产出、不反映用户在建模空间里手动新建的内容，会显得「Workspace 与时间轴脱节」。

| 情形 | 建议 |
|---|---|
| 用户在 Workspace / 入口 **手动新建** Note、Todo、Idea 等 | 生成一条时间轴 item：`milestone_kind` 如 `asset_created_manual`，`actor: user`，`refs.asset_id` |
| 用户在文件详情里 **手动编辑** 仅更新内容、不产生新实体 | 可不新增轴项，或合并为「最近编辑」（后续版本再议） |

与 **§3.5 策略 B** 的关系：AI 批量生成的 Asset 是否拆条可按策略取舍；**用户手动创建**通常一次一条，拆条成本低、叙事清晰，宜默认展示。

### 3.7 文件解析生成的 Note vs Ask Agent 生成的：时间轴要不要对称？

容易产生误解：**若文件解析产生的 Notes 不在时间轴逐条展示，Ask Agent 生成的 Notes 是否也必须不展示？**

结论：**不必对称**；两类来源在产品心智与「 burst 规模」上不同，可采用 **有原则的差异化**（见下表）。

| 维度 | 文件解析管线产出 | Ask Agent（conversation）产出 |
|---|---|---|
| **典型形态** | 一次解析 **批量** 落多条 Asset | **按轮次** 与用户对话，产出常与当前 turn 绑定 |
| **用户在场感** | 用户未必当场阅读每一条 | 用户通常 **在场** 看完本轮回复后再保存 |
| **时间轴默认** | §3.5 **策略 A**：管线里程碑 + 摘要，**明细进详情 / Workspace** | **至少要有会话级可见性**：否则「我刚问完 Agent」在工作台轴上 **不可追溯**，体验断层 |

**Ask Agent 在工作台轴上的最低建议**：

1. **会话级条目（推荐必有）**：例如「Ask · {会话标题或首句摘要} · 已生成 / 可保存 N 条结果」，`refs.session_id`，点击回到 Session / 历史。  
2. **是否再拆到「每条 Note 一条轴记录」**：可选；单次会话 turn 数通常少于「整文件解析一批十几条」，拆条 **刷屏风险更低**。若与文件解析 **观感一致**，亦可采用「会话一条 + 卡片内摘要」，与 §3.5 A 对齐。

**一句话**：文件解析用汇总里程碑，主要是因为 **单次批量太大**；Ask Agent 更应保证 **会话活动在轴上可追溯**，再决定在会话内是否逐 Asset 拆条。

## 5. 实现前需讨论的议题（开放问题）

以下问题 **不影响撰写本 Spec**，但应在编码加深前逐项对齐。

### 5.1 排序与时间戳

- **同一 File 多条里程碑**：每条 item 自有 `occurred_at`，按时间倒序排列；无需合并。  
- **同一 Session 多条更新**（非 File 管线）：仍可按「每条输出时间」或「聚合为一条卡片」取舍。  
- **异步任务进行中**：是否插入「进行中」占位卡，还是用详情页 / 推送表现？  
- **跨时区 / 同日折叠**：是否强制「今天 / 昨天」分组？

### 5.2 粒度：一条卡片对应什么？

- **同一文件**：入库一条 + 转录/解析完成至少一条（§3.0）已定为 **多条 item**。  
- **解析产生的多个 Asset**：采用 §3.5 **策略 A / B / C** 之一；默认倾向 **A**（汇总里程碑）或 **C**（仅待确认类拆条）。  
- **用户手动新建 Asset**：建议每条一条（§3.6）。  
- **Ask Agent**：至少会话级一条（§3.7）；是否逐 Note 拆条可选。  
- Flash 多轮 turn：每轮一条还是 Session 级一条？

### 5.3 用户 vs Agent 的视觉分工

- 左右气泡（对话感）vs 统一卡片左对齐（通知流）vs 双轨（用户在上、Agent 在下）？  
- 「沉浸式」是否包含 **全屏单任务聚焦**（例如解析中时弱化其他条目）？

### 5.4 与 Files / Workspace 的去重

- 文件已在 Feed 出现，文件库里是否 **重复展示**？若重复，如何标注「已在时间线」？  
- Workspace 资产：仅在 Feed 里摘要跳转，还是 Feed 只放里程碑、明细只在 Workspace？

### 5.5 会话类（conversation）的位置

- HOME_LOGIC_SPEC：conversation 不进 Files 列表。与 **§3.7** 一致：Feed 上建议至少有 **Ask 会话级** 时间轴项，避免「问完即消失」；若需与「系统通知类」区分，可用 `surface: conversation` + 固定模板文案。是否再拆 turn 级见 §3.7。

### 5.6 性能与分页（产品层）

- 时间线无限滚动时，是否按「日期 + 懒加载」还是「Session 线程折叠」控制信息量？

### 5.7 未读与打断

- Feed 是否需要 **未读红点 / 聚合摘要条**（与沉浸式「纯时间序」可能冲突）？

## 6. 线程化（可选进阶）

若同一 `thread_key` 下事件密集，可提供：

- **折叠**：默认一条「会议 · Product Design Sync · 3 条更新」，展开见子时间线。  
- ** Pin**：用户钉选某 Session（后续版本）。

Demo 第一期可 **不做线程**，仅在 Spec 中预留字段。
