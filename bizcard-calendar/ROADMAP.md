# BizCard Calendar 能力提升路线图

## 目标与价值

通过手机端授权多方日程管理 App（如 Google Calendar），将用户日程同步到 BizCard 中呈现，供 Agent 读取。Agent 在拥有日程上下文后，可更好地完成提醒、任务管理与后续跟进。

**本目录**：存放与 Calendar 能力拓展相关的需求、设计与技术方案，与现有 `bizcard-3.0` 的日历展示/交互（Reminders 为主）形成递进关系。

---

## 第一步：日历拓展为可用的「日程容器」

### 目标
- 将日历从「仅承载 Reminder（时间点）」拓展为可承载**日程（Event）**的容器。
- 日程为**区间型内容**：有开始时间、结束时间（如 14:00–15:00），而不仅是提醒时间点。

### 内容模型（最小集）
- **Event**
  - `id`
  - `title`
  - `start`（ISO 或 `YYYY-MM-DDTHH:mm`）
  - `end`（同上）
  - `allDay`（boolean，可选）
  - `source`（本地 / 待接入的第三方，如 `google`）
  - `calendarId`（归属哪个日历，为多日历预留）
  - `attendees`（参会人，可选，数组或字符串；来自第三方或用户填写）
  - `location`、`description`（可选）
- **Reminder**（现有）
  - 保持现有字段；在 UI 上可与同一天的 Event 一起展示（如日历格内：事件块 + 提醒点）。

### Event 卡片与 Notes 的统一

**Notes 本质上是「历史发生的 meeting」**；用户未必事先把该会议同步到 BizCard 日历，但会议一旦发生并被记录（录音/笔记），就可以用**同一套 Event 卡片**在日历上承载、用**同一套 Event 详情页**打开——即**当前的 Notes 详情页**在概念上就是「历史 Event（会议）」的详情。

- **Event 卡片**可承载两类内容：
  1. **来自日历的日程**：用户同步的 Google 等日程，或本地创建的日程（有明确 start/end，可有参会人、地点等）。
  2. **来自 Notes 的会议**：用户没有预先在日历里创建，但实际发生了并被 BizCard 记录下来的会议；以「历史 Event」的形式落在日历上，时间可由会议发生时间推导，详情即当前 **Notes 详情页**（含内容、Briefing、关联 Reminders 等）。
- **统一容器**：无论是「先有日程、后补笔记」还是「先有会议记录、在日历上以 Event 展示」，都用同一 Event 卡片 + 同一详情页；Notes 详情页即为该 Event（会议）的详情视图，无需为 Notes 再做一套独立卡片或详情。
- **实现上的衔接**：第一步将日历拓展为「可展示 Event + Reminder」的容器时，把现有 Notes（Meeting）视为一种 Event 数据来源，在日历/列表中用 Event 卡片展示，点击后进入现有 Notes 详情页（或与「来自日历的 Event」详情合并为同一页，按来源/类型做少量区块差异）。

### 设计要点
- 日历视图中需区分「事件块」（区间）与「提醒点」（点）。
- 日视图 / 周视图：事件以时间段条形式展示；提醒可挂在时间轴或列表区。
- 数据层：Event 与 Reminder 分离存储，便于后续只对 Event 做「多方日历同步」，对 Reminder 做「基于日程的智能安排」。

### 详情页策略：Reminder 与 Event 是否复用同一详情页

**结论：复用同一套详情页，按类型切换展示与操作（推荐）。**

同一日历上同时存在 **Event 卡片**与 **Reminder 卡片**；点击后进入同一详情页容器，按类型展示不同区块与操作。

- **Reminder 模式（保持简单）**
  - 展示：标题、时间点、来源、完成态、关联笔记（若有）。
  - 操作：标记完成、编辑、删除。
  - 不做复杂扩展，满足「待办/提醒」的查看与完成即可。

- **Event 模式（可做得更完整）**
  - 展示：
    - 标题、开始–结束时间、全天标识；
    - **参会人**（若有，来自同步或用户填写）；
    - 来源/日历、地点、描述（若有）；
    - **Briefing 区块**：用于帮助用户**回溯**该日程——可来自会前/会后的笔记摘要、Agent 生成的要点或待办回顾，用户进入详情即可快速「过一遍」上下文。
  - 操作：编辑、删除；可选「为该日程添加提醒」入口。
  - Briefing 的数据来源与生成时机可在第一步或第三步与 Agent 能力一起约定（如关联某条 Meeting Note、或由 Agent 在会前/会后生成）。

- **好处**：用户心智统一（点日历上任一项 → 同一详情）；Event 详情更「有信息量」（参会人 + Briefing 回溯），Reminder 保持轻量。

**建议**：第一步采用「一个详情页、按类型展示 Reminder / Event」；日历与列表上用卡片样式区分 Event 与 Reminder，详情内用同一容器 + 类型分支，Event 模式预留参会人与 Briefing 区块。  
**与 Notes 的衔接**：承载「历史会议」的 Event 卡片，点击后进入的详情即为**当前 Notes 详情页**（或与其合并后的同一详情页）；不另做一套「Notes 专属」卡片或详情，Event 卡片即可承载来自日历的日程与来自 Notes 的会议。

### 产出
- Event 数据模型与 API 约定（或本地 schema）。
- 日历容器 UI：支持展示「某日事件区间 + 当日 Reminders」的 demo 或高保真。
- 与现有 Reminder 的共存规则（同一天内的展示优先级、折叠规则等）。

## 第二步：接通多方日历（App 端设计与实现）

### 目标
- 用户在手机端授权一个或多个日历来源（如 Google Calendar、Outlook 等）。
- 将授权来源的日程拉取到 BizCard，在统一日历容器中展示。
- App 端需要完成：授权流程、账号管理、同步策略与错误态。

### 设计要点（App 端）
- **授权入口**：设置页或日历页内的「连接日历」类入口。
- **账号与来源管理**：
  - 展示已连接的日历账号（如 Google 账号邮箱）。
  - 支持断开、重新授权。
  - 多账号时可选「同步哪些日历」（若第三方 API 支持多日历）。
- **同步策略**：
  - 全量 + 增量（如按 syncToken / 变更时间）。
  - 同步频率与后台策略（前台刷新、后台定期、推送驱动等）。
- **权限与隐私**：
  - 明确向用户说明「读取日程用于 Agent 提醒与跟进；在本 App 内对日程的修改、删除会同步回该日历」。
  - 接入 Google 等需写回删/改时，需请求**读写日程**权限；仅做只读展示时可只请求读权限。
- **错误与状态**：
  - 授权过期、网络失败、限流等的提示与重试。
  - 单来源失败不影响其他已连接来源的展示。

### 第三方日程的增删改策略（以 Google Calendar 为例）

在 App 内呈现用户 Google 日程时，对**来自第三方（如 Google）的 Event** 采用「只读同步 + 删改回写」策略，不在第三方侧新增事件：

| 操作 | 行为 | 说明 |
|------|------|------|
| **增（Create）** | 仅在**本地日历**创建 | 用户在我们 App 里「新建日程」只写入 BizCard 本地日历，不向 Google 发起创建。避免把本地/Agent 产生的日程灌回 Google，保持 Google 为「用户主动管理的日程」来源。 |
| **删（Delete）** | 允许，并**回写 Google** | 用户在 App 内删除来自 Google 的某条日程时，同步调用 Google API 删除该事件，保证 Google 与 App 一致。 |
| **改（Update）** | 允许，并**回写 Google** | 用户在 App 内编辑来自 Google 的某条日程（如改时间、标题）时，同步调用 Google API 更新该事件，保证 Google 为唯一真相源。 |

- **只读来源**：若某连接仅用于「只读展示」（例如只读权限或产品暂不开放写回），则在该连接下不提供删/改入口，或仅提供「从本端隐藏」等本地操作。
- **本地 Event**：`source` 为本地的 Event，增删改仅落本地，不涉及第三方 API。
- **冲突**：若用户在 Google 端与我们端同时修改同一事件，以「最后写入」或「以 Google 为准」等策略在技术方案中明确（建议第二步技术设计时定稿）。

### 技术要点（概要）
- OAuth 2.0（或各厂商 SDK）完成授权。
- 统一将第三方事件映射为内部 Event 模型（含 `source` / `calendarId`）。
- 冲突与去重：同一事件在多端编辑时的策略（以某端为准或最后写入为准，需产品约定）。

### 产出
- App 端「连接日历」流程的交互与视觉规范（可落在本目录下的 design 或 prd 文档）。
- 多日历数据模型与同步接口设计（可落在 tech 或 api 文档）。
- 隐私说明与权限文案草案。

## 第三步：根据用户日程安排提醒与跟进

### 目标
- Agent 可读取用户日程（Event）上下文。
- 根据「何时有空」「何时有会」等，智能安排提醒时间或跟进建议（如「会后 15 分钟提醒你发纪要」）。
- 将生成的 Reminder 写回 BizCard 日历，与现有 Reminder 列表/日历视图一致展示。

### 设计要点
- **Agent 侧**：
  - 输入：用户自然语言 + 当前/未来一段时间的 Event 列表（可脱敏或摘要）。
  - 输出：建议的提醒时间、或直接创建 Reminder；可选「关联到某 Event」（如「会议 A 的跟进」）。
- **产品规则**：
  - 提醒与事件的关联方式（仅时间关联 vs 显式关联到某 Event）。
  - 用户对「自动安排」的接受度与开关（例如仅建议 vs 自动写入）。
- **日历侧**：
  - 若 Reminder 关联到 Event，在 UI 上可展示「某会议下有 1 个跟进提醒」。
  - 不影响现有「按日 / Timeline」的 Reminder 展示逻辑。

### 技术要点（概要）
- Agent 服务可访问「已同步的 Event」接口（鉴权、范围、频率控制）。
- Reminder 模型可扩展字段：如 `linkedEventId`（可选）。
- 若需「建议时间」：需简单规则或模型，结合 Event 区间、偏好设置等。

### 产出
- 「基于日程的提醒/跟进」产品规则（可单独 prd 或在本 roadmap 下细化）。
- Agent 使用日程的接口与数据格式约定。
- Reminder 与 Event 关联的 UI 与交互说明（可放在 bizcard-3.0 或本目录）。
