# PRD（迭代）：Reminder 详情页（现阶段）

| 属性 | 内容 |
|------|------|
| 状态 | 迭代草案 |
| 版本 | v0.10 |
| 定位 | 以「个人向待办」为核心，统一 Reminder 详情展示与可交付交互，避免引入多角色/协作/分享类边缘问题 |

---

## 1. 产品原则（与旧版差异）

### 1.1 个人待办：责任主体永远只有一个

- Reminder 是**当前用户自己的待办**；系统**不提供**向他人通知、分配、共享待办的能力。
- 因此 **取消「关联人 / 参与人」作为独立数据与 UI 模块**；不再在 Reminder 上维护多联系人关系。

### 1.2 人名、speaker、谁让谁做——进入「文案与转写引用」，不进入「关联人」

- 从 transcript 解析出的人名（如「冯总」「小B」）、以及 speaker1/2 等，**可出现在 Reminder 的标题、说明/备注，或 Source 卡内「转写引用（Transcript excerpt）」**中，用于**解释与回溯**。
- 若用户需要把某人沉淀为独立 Contact，应走**名片夹/会议 Contacts 等既有入口**（本 PRD 不在 Reminder 内提供联系人管理）。
  - Speaker 占位符属于**会议级映射**：在会议内完成“占位 speaker → 具体联系人/显示名”的配置后，Reminder 文案里出现的同占位符应**联动显示为对应人名**（见 1.4、3.4）；占位符的**书写形态**以产品/实现统一为准（如 `Speaker1`、或会议侧使用的 A/B 命名等，不对命名风格做强约束）。

### 1.3 从会议解析出的 Reminder：建议先「Suggested」再「Accept」

- AI 从 transcript 提取的候选项在 Meeting 侧以 **Suggested** 呈现，用户 **Accept** 后进入个人生效待办（具体状态机见你们 Meeting PRD；本页只要求详情页能承载「已生效待办」展示）。

### 1.4 与会议主流程的关系（先匹配声纹/说话人，再读 Reminder 文案）

整体链路是：**录音上传 → ASR 转写 → AI 解析**；解析产物之一可以是 Reminder。

- **进入会议详情后的首步能力（可跳过）**：在会议里提供**说话人/声纹与 Contact 的手动对应**能力（你方 UI 上可呈现为 *Identify participants* 一类工作流；具体页面形态与分步不约束）。在这一步中，用户把会议内识别出的**占位说话人**匹配到**具体联系人**；并可持续沉淀**谁参加了这场会**的参会人信息。用户**可跳过**该步继续浏览会议，但**未完成匹配时，占位符不会自动变成真人名**。
- **对 Reminder 的约束**：从会议解析出来的待办，**默认可带 speaker 占位符**（如随解析模板输出的 `Speaker123` 体系）。一旦在同一场会议内，对应占位符完成了“→ 联系人/显示名”的映射，Reminder 侧**凡展示该占位符处都应随之替换**；若用户先看到 Reminder、会议侧尚未匹配，则继续显示占位符，并依赖 **3.4** 里“补映射/选择联系人”的路径。

## 2. 目标与范围

### 2.1 目标

- 在 Reminder 详情页完成：**内容、时间、状态**的查看与编辑（视实现为只读/可编）。
- 管理 **Attachment**（Image / MD文档 分类；上传入口在底部 `+`）。
- 提供 **Source** 单卡可回跳（按来源类型）；当且仅当待办来自**可引用的会议转写**时，在同一来源卡片内展示**转写引用**（不单独截“音频层”原文）。
- 对**非从转写句段直接抽取/无可引用证据**的待办：**不展示转写引用**（只展示标题/说明即可）。

### 2.2 本期范围（In Scope）

- Header + **仅 Attachment** Tab（或未来扩展，但本期不开放 Contacts Tab）。
- Source 三种来源展示（手动 / Ask Agent / 会议解析）。
- 转写引用：在同一来源卡片内以轻量样式展示：**Speaker（或用户名） + 时间码 + 原文片段**（不做二次跳转；返回会议后在会议页完成定位/播放）；**视觉层级**见 **3.3**（须弱于 Reminder 主内容区）。
- 底部悬浮条：录音 / Ask Agent / `+`；本期 `+` **仅 Upload 关联文件**。

### 2.3 不在范围（Out of Scope）

- Reminder 内联系人列表、`?` 匹配、Select Contacts 快捷添加等**已全部移除**。
- 多角色待办、协作、分享、提醒给他人。
- 复杂日程：重复/多时区/全天块等（本期以 DDL 单值为主）。

## 3. 创建来源（Source）与展示

Reminder 可来源于：手动创建 / Ask Agent / 会议音频解析。

### 3.1 展示规则

- **只展示一张主来源卡片**。
- 手动：纯展示，不可回跳。
- Ask Agent：可「查看会话」回跳。
- 会议解析：可「回到会议」回跳；**转写引用（若存在）放在同一张来源会议卡内**。

> Ask Agent 创建且同时存在会议 context：Source 以 **session** 为主，会议为次级信息（不拆第二张来源卡）。是否展示转写引用取决于该待办是否绑定可引用转写句段；无绑定则不展示。

### 3.2 转写引用出现条件

- **展示**：待办从 transcript 的明确句段抽取，且**存在**可用于回溯的可引用转写信息（**至少含文本**；谁在说、起止时间等以实际实现为准，可为空但弱化展示）。  
- **不展示**：
  - 手动创建 / 纯规则推断 / 合并摘要型生成，导致**没有对应可引用句段**；  
  - 或仅具备会议来源但**无可引用句段/证据不足**（实现可选择隐藏，避免“看起来像有证据但其实没有”）。

### 3.3 信息架构与视觉层级（Source 轻、主内容重）

**目的**：Source 与转写引用属于**回溯与出处**，不应在视觉上与「待办本身」同级，避免用户误以为来源卡片才是主阅读对象。

**层级约定（自上而下由重到轻）**：

1. **Reminder 主内容区**（标题、说明、时间等主字段）：使用页面内**最重**的正文层级——例如白底、清晰实线或等价分隔、正常标题/正文字重与主色，作为用户编辑与扫读的第一焦点。
2. **Source 单卡**：整体须**明显轻于**上述主内容区——例如浅灰底、弱边框（虚线或低对比实线均可）、**不使用**与主字段同级的强阴影或大号粗体标题；回跳按钮为**次级**（小号或幽灵/描边样式），不与主操作按钮同级。
3. **同卡内的转写引用**：在 Source 卡内部再降一级——**Speaker（或用户名）+ 时间码**使用更小字号与更低对比的辅助色；**原文片段**字重与颜色须**弱于** Reminder 说明正文（仍须可读），仅作佐证，不参与与标题抢注意力。

**实现提示**：Demo 见 `reminder-detail-demo.html`；设计令牌可与 Meeting 详情对齐，但 Reminder 详情内须遵守本节层级，不得把 Source 做成与 `.field` 主区块同质感的「第二张主卡片」。

---

### 3.4 Speaker 占位符的联动与快速替换

**问题背景**：AI 从会议总结/转写中抽取待办时，文案**默认可带**会议侧的 speaker 占位符；占位符是否已替换，取决于**第 1.4 节**的会议内匹配（或 **3.4 下文** 的补映射）是否已发生。占位符的**具体写法**不约束（数字编号、A/B 命名等，以实现为准）。

**联动规则（不约束字段/存储方式）**：

- **会议级别映射**：同一会议内，需存在一个**稳定的**“占位符说话人 → 具体联系人/显示名”的映射结果；其输入主要来自 **1.4**（会议详情中的说话人与 Contact 对应/参会人识别流程），**允许用户跳过**该步（此时映射为空，文案保持占位符）。
- **渲染联动**：Reminder 展示标题/说明/转写引用时，凡出现需要随映射替换的占位符，应**始终**使用会议侧**最新**映射；用户在会议中更新对应关系后，派生的 Reminder **应自动反映变化**（不要求用户逐条手动改待办文本）。
- **与命名示范的关系**：下文用 `Speaker1` / `Speaker\d+` 仅为**举例**；**实际占位符体系以解析与会议侧展示为准**（也兼容 A/B 等形态）。
- **覆盖范围**：联动至少覆盖
  - Reminder 标题/说明中的占位符
  - 转写引用的 speaker 展示（若有）
  - 转写引用正文中出现的占位符（若有）

**未映射时的补全（UX）**：

- 当占位符仍未被映射时：
  - **优先入口**：将 Reminder 标题/说明中仍显示的占位符渲染为**可点击 token**，点击后打开 **Contact 选择器**，把该占位符绑定到**某个联系人/显示名**（效果上**补齐或修正** 1.4 那一步在会议里本应完成的对应关系；是否允许在 Reminder 里写回会议侧，以实现为准，但**展示必须联动**）。
  - **可选入口**：在 Meeting 详情的 Reminder 列表中，也允许用同样方式对占位符进行点击补全（更适合“在列表里快速改完”）。
  - 选择联系人后，更新**会议级映射**并触发同会议派生内容联动更新（Reminder 多处以同一规则重绘即可）。
- 作为兜底：用户也可以直接编辑 Reminder 标题/说明，将占位符替换成具体人名（**仅改当前条文案**的路径，是否应反向影响会议级映射，以实现为准，本页不对写入策略做约束）。

## 4. 页面结构（Meeting-style）

### 4.1 Header

- 状态：未完成 / 已完成。
- **Source** 单卡（见第 3 节，含**可选**转写引用，见 3.2；视觉层级见 3.3）。
- **Reminder 内容**：标题 + 说明（人名/上下文尽量写在这里，而非“关联人”区）。
- **时间**：DDL。

### 4.2 Tab

- 仅 **Attachment**（子 Tab：Image / MD文档，无数据不展示子分类）。

- **Upload** 只通过底部 `+` → `Upload 关联文件`；上传后自动切到 **Attachment** 并落表。

## 6. 底部悬浮条

- **录音**：补充语音说明（与实现绑定）。
- **Ask Agent**：单 Reminder 单 session；用于改标题/时间/说明等（以可用能力为准）。
- **`+`**：本期**仅**「Upload 关联文件」。

## 7. 验收标准

1. 详情页**无** Contacts/关联人模块；人名与上下文出现在标题/说明，或 Source 卡内的**转写引用**中（若存在）。
2. 会议来源：转写引用（若存在）**与来源会议同卡**展示；**无可引用转写信息**的待办**不**展示转写区。
3. Attachment 分类与上传路径符合第 5、6 节。
4. Source 不重复、单卡可回跳（有入口时）。
5. Source 与转写引用的展示符合 **3.3**：主内容区视觉最重，Source 卡轻于主内容，转写引用轻于 Source 卡内会议行文案。
6. 若待办文案包含会议侧 speaker 占位符：在会议内完成/更新**占位说话人 → 联系人**对应后，Reminder 侧展示应联动更新（见 1.4、3.4）；用户在 **1.4 可跳过** 的前提下，应仍能通过 **3.4** 的 **Contact 选择器**在 Reminder/列表中补全映射；占位符的**具体写法**不以此验收条约束。
