## 1. 功能概述

硬件 2.0 引入 **蓝牙耳机连接功能**：卡片作为手机与蓝牙耳机之间的**音频桥梁**，用户通过卡片录制耳机通话/会议音频。

**核心价值**：用户戴着蓝牙耳机参加线上会议/通话时，卡片在后台完成**双方向录音**——对方声音（手机下行）和我的声音（耳机麦克风上行）——全部在卡片侧编码存储。

**App 职责**：通过 BLE 控制链路引导用户完成三条蓝牙连接的建立、状态展示、异常恢复；**不直接参与音频传输或录音编码**。

---

## 2. 蓝牙链路架构

| # | 链路 | 连接关系 | 类型 | 作用 |
|---|------|----------|------|------|
| ① | 控制链路 | 手机 App ↔ 卡片 | BLE | 遥控指令（扫描耳机、指定 MAC、电量上报等）；**不传音频** |
| ② | 下行音频 | 手机系统 ↔ 卡片（`Eureka-Audio`） | 经典蓝牙（HFP/A2DP） | 手机将会议/通话音频路由到卡片，供录音截取 |
| ③ | 桥接音频 | 卡片 ↔ 用户蓝牙耳机 | 经典蓝牙 | 卡片将下行音频转发至耳机（用户听对方）；耳机麦克风上行（用户声音）进卡片录音 |

**录音落点**：会议/通话场景下，**录音编码与文件落点在卡片侧**。手机通过 `Eureka-Audio` 提供下行音频源；耳机麦克风上行经耳机→卡片链路至卡片。

**强接管**：耳机已连接时，录音**一律使用耳机麦克风**（含闪念场景），滑块误拨不切换麦克风来源。仅在耳机未连接时，才使用侧边滑块对应麦克风。

## 3. 绑定模型

| 关系 | 基数 | 说明 |
|------|------|------|
| 账号 → 卡片 | **1 : N** | 一个账号可绑定多张卡片 |
| 卡片 → 账号 | **1 : 1** | 一张卡片只能绑定一个账号，解绑后方可绑定其他账号 |

## 4. 页面架构与路由

### 4.1 二态路由

App 设备页采用**二态路由**——扫描页与设备控制台之间根据 BLE 连接状态自动切换。

| 进入条件 | 落地页面 |
|----------|----------|
| 有 BLE 连接中的卡片 | 直接进入该卡片的**设备控制台** |
| 无 BLE 连接 + 无经典蓝牙活跃 | **扫描页**（纯净态） |
| 无 BLE 连接 + 有卡片经典蓝牙仍在 | **扫描页** + 顶部第三态横幅（→ §11） |

### 4.2 页面职责

| 页面 | 职责 | 不负责 |
|------|------|--------|
| **扫描页（Search Devices）** | 发现设备、展示已绑定卡片列表、绑定新卡片 | 耳机模式配置、链路管理 |
| **设备控制台** | 单张卡片的全链路管理（BLE、耳机模式 Toggle、引导链） | 切换到其他卡片（需返回扫描页） |

### 4.3 页面跳转

| 来源 | 动作/条件 | 目标 |
|------|-----------|------|
| 扫描页 | 点击「连接」→ BLE 成功 | 设备控制台 |
| 扫描页 | 点击「Bind」→ 绑定成功确认 | 设备控制台 |
| 设备控制台 | BLE 断开（无经典蓝牙活跃） | 扫描页（纯净态） |
| 设备控制台 | BLE 断开（经典蓝牙仍在） | 扫描页 + 第三态横幅 |
| 设备控制台 | 用户点击「返回」 | 扫描页（BLE 连接保持，再次进入仍进管理页） |
| 设备控制台 | 用户点击「解绑」并确认 | 扫描页 |

## 5. 扫描页（Search Devices）

### 5.1 页面结构

```
[NavBar] Search Devices                         [刷新]

[横幅区 · 仅第三态可见]（→ §11）

[Scan Hero]  扫描动画 + "Scanning Nearby Devices"

[My Devices]  ── 已绑定卡片列表
  Card A  · SN-xxx  · 已连接 / 未连接  [管理 / 连接]
  Card B  · SN-xxx  · 未连接           [连接]

[+ 添加新的 BizCard]

[Available Devices]  ── 附近可绑定卡片（扫描结果）
  Card C  · SN-xxx  · 可用    [Bind]

[Before you start]  ── 操作提示
```

### 5.2 交互规则

| 操作 | 行为 |
|------|------|
| 点击 My Devices 中的「连接」 | 发起 BLE 连接 → 成功弹窗确认 → 进入设备控制台 |
| 点击 Available Devices 中的「Bind」 | 绑定流程 → 绑定成功弹窗（含隐私说明）→ 确认后进入设备控制台 |
| 曾绑定卡片自动回连 BLE | 卡片状态实时刷新 + Toast「Card SN-xxx 已自动连接」→ 延迟 ~2s 自动跳转管理页（用户操作可取消跳转） |
| 点击「刷新」 | 触发重新扫描附近设备 |
| 点击「+ 添加新的 BizCard」 | 进入全局扫描，展示 Available Devices |

### 5.3 空态

无已绑定卡片时，仅展示 Scan Hero + Available Devices（扫描结果）+ Before you start。

## 6. 设备控制台

### 6.1 页面结构（单页纵向）

采用**单页 + 耳机模式 Toggle**，不用并列 Tab。

1. **设备头部（Device Header）**：设备名、SN、电量、BLE 连接状态
2. **卡片连接区**：BLE 连接状态展示 / 连接按钮
3. **耳机模式区**：Toggle（兼经典蓝牙能力开关）+ 展开引导链
4. **辅助信息区**：帮助、解绑、固件版本

### 6.2 线框状态清单

| 状态 | 描述 | 关键 UI 表现 |
|------|------|-------------|
| **W1** | BLE 未连接 | 主 CTA = 连接卡片；Toggle 禁用灰显 |
| **W2** | BLE 已连，Toggle OFF | Toggle 可用；说明文案引导开启 |
| **W3** | Toggle ON，引导 Eureka | 引导链 ③ 高亮；Bottom Sheet 引导去系统蓝牙 |
| **W4** | 自动回连耳机中 | 引导链 ④ 进行中；可取消/扫描其他 |
| **W5** | 手动扫描耳机 | 扫描按钮 + 耳机列表 |
| **W6** | 全链路就绪 | 绿点 + 耳机名称；可更换耳机 |
| **W7** | BLE 断开 | 回到连接卡片主逻辑；弱提示经典链路可能仍在 |

### 6.3 本页不承载

- 开始录音 CTA
- 录音页逻辑
- 录音列表断链展示（沿用现有版本）

## 7. 耳机模式 Toggle 与引导链

### 7.1 Toggle 门禁

| 场景 | Toggle 状态 |
|------|------------|
| BLE 未连接 | **禁用**（灰显不可点） |
| BLE 已连接 | **可操作** |
| BLE 中途断开 | **禁用**；页面回到连接卡片逻辑 |

Toggle ON = 打开卡片侧经典蓝牙能力；Toggle OFF = 关闭经典蓝牙能力。

### 7.2 引导链（Toggle 展开区）

仅当 **Toggle ON 且 BLE 已连接** 时展示。每步 4 种状态：`未就绪 / 进行中 / 已就绪 / 异常`。

| 步骤 | 内容 | 门禁 |
|------|------|------|
| ① | 手机系统蓝牙已开启 | 未开 → 引导去系统设置 |
| ② | App ↔ 卡片 BLE 控制链路已连接 | 本页上部已满足 |
| ③ | 手机 ↔ 卡片 `Eureka-Audio` 已连接 | 未连 → Bottom Sheet / 前往系统蓝牙 |
| ④ | 卡片 ↔ 蓝牙耳机已桥接 | 优先自动回连 → 超时/失败则扫描列表 |

### 7.3 耳机模式就绪判定

- **耳机模式 = ready**：仅当 `Eureka-Audio` = connected **且** 卡片↔耳机 = connected
- **耳机模式 = not ready**：其余所有情况
- **不设「准备中」聚合状态**；连接过程仅在每一步内表达

### 7.4 与录音准备页联动

- Toggle OFF 或引导链未完成 → 录音准备页**禁用或引导**至设备控制台
- Toggle ON 且引导链完成 → 可与录音准备页联动为「可录」
- **禁止**：仅 Toggle ON 而无引导链完成就让用户误以为可录

|----------|-------------|---------------|---------|-----------|--------------|

## 9. 前置条件与门禁

| 用户动作 | 允许条件（AND） | 不允许时的拦截 |
|----------|----------------|---------------|
| 开启耳机录音模式（Toggle ON） | 已绑定卡片 + BLE 已连接 | Toggle 禁用 + 引导连接卡片 |
| 进入 Phase 1 引导（去系统蓝牙） | BLE 已连接 | 引导先连卡片 |
| 点击「开始扫描耳机」 | BLE 已建立 +（建议）Eureka 已连接 | **按钮禁用** + 文案说明 |
| 选择耳机并连接 | BLE 可用 + 扫描已发起且未超时 | 错误态 + 重试 |
| 从录音准备页开录（耳机模式） | 与设备控制台**同一状态源**判定通过 | 阻断或提示先恢复链路 |

### Phase 1 检测机制

App 检测 `Eureka-Audio` 连接状态：

- **iOS**：`AVAudioSession.currentRoute` 检查音频输出/输入设备；监听 `routeChangeNotification`
- **Android**：`BluetoothAdapter.getBondedDevices()` + `BluetoothProfile.getConnectedDevices(A2DP)`；注册 `ACL_CONNECTED/DISCONNECTED` 广播
- **兜底**：App 在 `onResume` / `viewDidAppear` 时主动拉取一次连接状态

## 10. 自动回连策略

### 10.1 触发时机

用户在设备控制台打开耳机模式 Toggle 后，按序触发：

1. **Toggle ON** → App 通过 BLE 指令卡片开启经典蓝牙能力
2. **检测 Eureka-Audio** → 已连接则跳过 Phase 1；未连接则弹 Bottom Sheet 引导
3. **Eureka 就绪后** → 检查历史耳机记录 → 有则下发回连指令（进入 W4）→ 无则进入 W5 手动扫描
4. **回连超时（10–15s）** → 降级到 W5 手动扫描

### 10.2 手机↔卡片 BLE 自动回连

- **前置**：卡片已绑定；卡片已唤醒进入广播；手机蓝牙开启；App 具备蓝牙权限
- **兜底**：短时间未回连 → 展示「请靠近卡片/点亮卡片」+ 重试

### 10.3 卡片↔耳机自动回连

- **前置**（需硬件确认）：卡片侧可持久化保存最近耳机标识；耳机出仓可连接态且未被手机抢占
- **兜底**：回连超时 → 提示用户让耳机进入配对态并点击「开始扫描」

### 10.4 耳机被手机抢占

| 阶段 | 策略 |
|------|------|
| **首次配对** | 用户必须先将耳机与手机断开，再由卡片扫描配对 |
| **非首次** | 卡片回连 → 超时失败 → App 提示「耳机可能已连接手机」→ 提供「前往系统蓝牙设置」Deep Link |

标准蓝牙协议无法"抢夺"已连接设备，此场景需文案引导用户手动操作。

## 11. 第三态：BLE 断 + 经典蓝牙仍在

### 11.1 定义与触发

当一张卡片的 BLE 断开，但其经典蓝牙链路（Eureka-Audio 和/或耳机桥接）仍然活跃时，App 进入第三态。

### 11.2 页面表现

设备控制台自动退回扫描页，扫描页顶部展示醒目横幅。

### 11.3 硬阻断规则

| 元素 | 状态 |
|------|------|
| 当前经典蓝牙活跃卡片的「重新连接」 | **可用**（突出展示，唯一允许操作） |
| 其他已绑定卡片的「连接」按钮 | **禁用灰显** |
| Available Devices 的「Bind」按钮 | **禁用灰显** |
| 「添加新的 BizCard」入口 | **禁用** |

### 11.4 退出途径

| 途径 | 结果 |
|------|------|
| 点击「重新连接 BLE」成功 | 跳转到该卡片的设备控制台 |
| 经典蓝牙自行断开（卡片 5 分钟超时关机、耳机没电） | App 清除横幅，回到纯净扫描态 |
| 用户靠近卡片，BLE 自动重连 | 同第一条 |

### 11.5 状态来源

App 维护 `classicBtActiveCard` 状态：

- **初始判断**：BLE 断前缓存「耳机模式 Toggle ON + 经典链路已连接」
- **实时校验**：系统蓝牙 API（`BluetoothAdapter` / `AVAudioSession`）检测 Eureka-Audio 连接状态
- **桥接状态**：仅靠缓存推断（App 不直接参与该链路）

### 11.6 设计意图

BLE 一次只能连一张卡，在经典蓝牙仍活跃时连接其他卡片会导致旧卡片处于「孤儿态」——录音可能仍在进行但无法停止、耳机模式无法关闭。硬阻断避免这种不可逆混乱。

## 12. 异常处理

| 场景 | 策略 | 打扰度 |
|------|------|--------|
| `Eureka-Audio` 未连或中途断开 | **耳机录音模式下立即终止录音**；提示重连后可重新开始；文件标记「链路异常中断」 | 不强推即时弹窗 |
| 卡片↔耳机桥接中途断开 | **耳机模式下立即终止录音**；提示检查耳机电量并重新桥接 | 不强推即时弹窗 |
| BLE 断开但双经典仍连 | **录音可继续**（音频不依赖 BLE）；失去远程控制；退回扫描页 + 第三态横幅 | 用户感知中 |
| 耳机扫描超时/无结果 | 允许重试；说明耳机需可发现 | 页内提示 |
| 耳机连接失败 | 错误码映射文案；重试/换设备 | 页内提示 |
| 耳机被手机抢占 | 提示「耳机可能已连接手机」→ Deep Link 去系统蓝牙 | 页内提示 |
| 未开耳机模式时耳机链路变化 | **不影响当前录音**；仅提示耳机不可用 | 弱提示 |
| App 退后台再回前台 | **主动拉取**硬件上报；必要时强制刷新 BLE | 无感 |

**断连策略核心口径**：耳机录音模式下，任一经典蓝牙链路断开 → **终止录音**。「无缝降级继续录音」**不作为产品承诺**。

**打扰度总原则**：

- BLE 断开：默认不弹 Toast/通知
- 经典链路断开导致停录：不强推 App 内即时弹窗（用户在耳机侧已即时感知）
- 回到 App 后：在设备控制台 Toggle 展开区内看到异常与修复入口

## 13. 录音中的设备控制台规则

以下动作会破坏音频链路或终止录音，在**录音中**须满足其一：

| 策略 | 说明 |
|------|------|
| **默认**（推荐） | 「更换耳机」/「断开耳机」/「关闭耳机模式」**禁用**（灰显 + 「录音中不可用」） |
| **允许点击** | 必须**二次确认**，文案明确「将停止当前录音」 |

设备控制台**不提供开始录音按钮**。

## 14. 反馈分工（App vs 卡片）

| 状态/事件 | 以谁为主 | 说明 |
|-----------|----------|------|
| BLE 连接、扫描耳机、连接失败重试 | **App** | 信息量大，需文案与列表 |
| 电量、极简连接态 | **卡片屏** | 弱打扰、抬手可见 |
| 耳机桥接成功 | 两端 | App 一次成功反馈 + 卡片常驻小图标；避免重复 Toast |
| 录音中链路异常 | 录音页/全局轻提示 | 避免在硬件页抢焦点 |

**状态所有权**：以设备控制台为**全链路展示主页面**；「耳机录音模式」开关与录音准备页共用**同一状态源**。硬件主动上报优先更新总控状态，录音准备页订阅同一源。

## 15. BLE 长连接与 5 分钟超时

| 阶段 | 行为 |
|------|------|
| 休眠下单击主键 | 亮屏「待连接」→ 进入 BLE Advertising → **启动 5 分钟倒计时** |
| 5 分钟内 App 连接成功 | **立即取消倒计时** → 进入 BLE Standby（低功耗待机） |
| 5 分钟内无连接且无本地操作 | **关机**（非深度休眠） |
| 本地录音进行中 | 录音优先，不因超时关机 |
| 本地任务结束后 | 重新启动 5 分钟广播 |

## 16. 录音标记（角键打点）

| 触发条件 | 动作 | 行为 |
|----------|------|------|
| 待机 | 角键**长按 0.5s** | 触发闪念 |
| 会议/通话录音中 | 角键**单击/短按** | 打时间戳锚点 |

**打点反馈**：

- **震动**：两次极短震（与开录长震、闪念单短震区分）
- **屏幕**：第 3 行波纹区临时显示 `MARKED` / `重点`，约 1.5s 后恢复波纹
- **闪念麦克风**：耳机已连时**必须**使用耳机麦克风（强接管不因闪念例外）
