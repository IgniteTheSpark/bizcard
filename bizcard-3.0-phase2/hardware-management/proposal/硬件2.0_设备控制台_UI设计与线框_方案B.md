# 硬件 2.0：设备控制台 UI 设计与线框（方案 B）

> **适用范围**：硬件管理路径（方案 B）下的 **设备控制台** 页 UI 设计与交互线框  
> **来源**：基于 `硬件2.0_硬件管理页改造方案.md` 已拍板口径  
> **关键口径**：
> 1. `BLE` 连接状态 = 当前 App 与卡片的控制连接状态  
> 2. `BLE` 断开时，页面应回到 **搜寻/连接卡片** 的主逻辑，而不是停留在只读态  
> 3. 耳机录音模式下任一经典蓝牙断开（`Eureka-Audio` 或桥接）→ **录音停止**  
> **更新时间**：2026-04-15

---

## 1. 目标与非目标

### 1.1 目标

- 让用户先完成 `App ↔ 卡片 BLE` 连接，再进行耳机模式配置。
- 让用户在打开 `耳机模式 Toggle` 后，清楚看到两条经典链路的真实状态：
  - `卡片 ↔ 手机（Eureka-Audio）`
  - `卡片 ↔ 耳机`
- 让异常场景具备一致的页面表现与修复路径。

### 1.2 非目标

- 不在本文件内定义固件协议字段与错误码枚举。
- 不展开录音详情页完整 UI。
- **整个硬件管理页不承载开始录音 CTA，也不承载录音页逻辑。**
- 录音列表对“异常中断文件”的呈现沿用现有方案，本文件不重复定义。

---

## 2. 页面信息架构（方案 B）

### 2.1 页面分工

- **Search Devices**：仅负责“找卡片 / 绑卡片”，绑定成功后进入设备控制台。
- **设备控制台**：先处理卡片连接，再处理耳机模式。

### 2.2 设备控制台结构：单页纵向（不用 Tab）

> **为何不用 Tab**：卡片连接与耳机模式不是平级关系。真实顺序是：先连卡片，再决定是否打开耳机模式。

1. **设备头部（Device Header）**
- 设备名、SN、电量
- `卡片连接状态`（即 App ↔ 卡片 BLE）
- 次要操作：帮助、解绑、刷新

2. **卡片连接区（主路径）**
- 主 CTA：`连接卡片`
- 当 BLE 未连接时，页面焦点和主操作都在这里
- 表达方式应尽量复用现有 `Search Devices` 的搜寻卡片心智

3. **耳机模式区（次级）**
- `耳机模式 Toggle`
- Toggle 同时承担 **卡片侧经典蓝牙能力开关**
- Toggle 默认 `OFF`
- Toggle 打开后，展开耳机模式连接项

4. **辅助信息 / 设备操作区**
- 帮助
- 固件信息
- 解绑等

### 2.3 Toggle 门禁与 BLE 断开后的页面逻辑

| 场景 | 页面行为 |
|------|----------|
| **BLE 未连接** | 页面主任务 = `连接卡片`；耳机模式区禁用 |
| **BLE 已连接** | Toggle 可操作；打开后展开经典链路配置 |
| **BLE 中途断开** | 页面重新回到 `连接卡片 / 搜寻卡片` 主逻辑；耳机模式区禁用；若能拿到旧链路状态，可弱展示，但不允许操作 |

> 口径统一：**BLE 是否连接 = 当前 App 是否与卡片建立控制连接**。  
> 因此只要 BLE 断开，当前页面的主交互就应该重新变成 **搜寻并连接卡片**。

### 2.4 耳机模式结果态

- **Toggle OFF**：耳机模式未启用，卡片侧经典蓝牙能力关闭。
- **Toggle ON**：耳机模式已启用，卡片侧经典蓝牙能力打开。
- **耳机模式 = ready**：仅当以下两条都为 `connected`
  - `卡片 ↔ 手机（Eureka-Audio）`
  - `卡片 ↔ 耳机`
- **耳机模式 = not ready**：除上述情况外全部属于 `not ready`

> **不定义“准备中”作为耳机模式聚合状态。**  
> 连接过程只在每一条连接项里表达，例如：`disconnected / connecting / connected`。

### 2.5 Toggle 展开后：连接项

当 Toggle 打开且 BLE 已连接时，耳机模式区展开以下连接项：

1. **手机系统蓝牙**
- 辅助前置项
- 若未开启，引导去系统设置打开

2. **卡片 ↔ 手机（Eureka-Audio）**
- 状态：`disconnected / connecting / connected`
- 操作：`去连接 Eureka`

3. **卡片 ↔ 耳机**
- 状态：`disconnected / connecting / connected`
- 操作：自动回连最近耳机；失败后手动扫描
- `connected` 时必须展示耳机名称
- 若系统无法读取名称，则降级展示最近设备标识

---

## 3. 状态与门禁（UI 必须显式表达）

### 3.1 门禁总原则

- **打开耳机模式 Toggle** 的前置条件：BLE 已连接。
- **扫描/连接耳机** 的前置条件：BLE 已连接 + `Eureka-Audio` 已连接。
- **BLE 断开**：页面回到 `连接卡片 / 搜寻卡片` 主流程；耳机模式区不可操作。
- **耳机录音模式下任一经典蓝牙断开**：录音停止；页面展示异常状态与修复入口。

### 3.2 展开区连接项展示规则

- 区块标题建议：`耳机连接`
- 连接项状态统一使用：`disconnected / connecting / connected`
- 每个连接项最多一个主动作
- 不使用“准备中”作为总状态徽标

---

## 4. 文案与视觉规范

- 避免使用“权限/授权”，统一使用“连接设备/连接链路”
- `Eureka-Audio` 的文案建议：`连接 Eureka 音频设备（系统蓝牙）`
- 异常提示必须同时包含：
  - 结果：如 `录音已停止` / `当前未连接卡片`
  - 下一步：如 `连接卡片` / `去连接 Eureka` / `重新连接耳机`

### 4.1 打扰度策略

- **BLE 断开**：默认不弹 Toast/通知；用户回到设备控制台时，直接看到 `连接卡片` 主逻辑。
- **经典链路断开导致停录**：默认不强推 App 内即时提醒；用户回到 App 后，在设备控制台看到异常项与修复入口。
- 整个硬件管理页不做录音启动相关 CTA。

## 5. 线框（Toggle 版）

### 5.1 W1：BLE 未连接

```text
[NavBar]  < 返回      设备控制台                 帮助

[Device Header]
BizCard Recording Card   SNxxxx
卡片连接：未连接   电量：--%

[卡片连接区]
[连接卡片]
辅助：请点亮卡片并靠近手机

[耳机模式]
耳机模式（经典蓝牙）  [OFF]   （禁用）
说明：请先连接卡片后再开启
```

### 5.2 W2：BLE 已连接，Toggle 默认 OFF

```text
[Device Header]
卡片连接：已连接   电量：85%

[卡片连接区]
已连接 · SNxxxx

[耳机模式]
耳机模式（经典蓝牙）  [OFF]
说明：开启后将连接 Eureka 音频设备并配置耳机
```

### 5.3 W3：Toggle ON，Eureka 未连接

```text
[耳机模式]  [ON]

[耳机连接]
手机系统蓝牙         connected
卡片 ↔ 手机          disconnected   [去连接 Eureka]
卡片 ↔ 耳机          disconnected

┌──────── Bottom Sheet ────────┐
| 连接 Eureka 音频设备          |
| [前往系统蓝牙设置]            |
| [我已连接，返回检测]          |
└─────────────────────────────┘
```

### 5.4 W4：Toggle ON，自动回连最近耳机

```text
[耳机模式]  [ON]

[耳机连接]
卡片 ↔ 手机          connected
卡片 ↔ 耳机          connecting
回连中… [取消] [重新扫描其他耳机]
```

### 5.5 W5：Toggle ON，手动扫描耳机

```text
[耳机模式]  [ON]

[耳机连接]
卡片 ↔ 手机          connected
卡片 ↔ 耳机          disconnected
[开始扫描]
设备列表…
```

### 5.6 W6：Toggle ON，全链路就绪

```text
[耳机模式]  [ON]

[耳机连接]
卡片 ↔ 手机          connected
卡片 ↔ 耳机          connected   AirPods Pro

耳机模式：ready
[更换耳机]
```

### 5.7 W7：BLE 断开后，回到搜寻/连接卡片逻辑

```text
[Device Header]
卡片连接：未连接
副文案：请重新连接卡片后继续管理硬件

[卡片连接区]
[连接卡片 / 重新连接卡片]
辅助：可沿用 Search Devices 的搜寻卡片逻辑与状态表达

[耳机模式]
耳机模式（经典蓝牙）  [OFF]   （禁用）
说明：请先重新连接卡片后再管理耳机模式
```

> 若底层还能拿到旧耳机链路状态，可在耳机模式区做弱提示，但不作为当前主操作区。

---

## 6. 运行时异常

### 6.1 BLE 断开（经典链路可能仍在）

- 结果：录音是否继续由经典链路决定；但从 App 视角看，当前已经失去与卡片的控制连接
- 设备控制台表现：
  - Header：`卡片连接：未连接`
  - 主区域回到 `连接卡片 / 搜寻卡片` 逻辑
  - 耳机模式区禁用，等待 BLE 恢复后再继续配置

### 6.2 耳机模式下任一经典链路断开

- 结果：录音停止
- 页面表现：
  - Toggle 保持 `ON`
  - `卡片 ↔ 手机` 或 `卡片 ↔ 耳机` 变为 `disconnected`
  - 提供修复入口：`去连接 Eureka` / `重新连接耳机`

### 6.3 录音中访问设备控制台

- 会破坏经典链路的操作默认禁用；或必须二次确认并明确提示：`将停止当前录音`
- 本页不提供开始录音按钮

---

## 7. 页面跳转与来源

- `Search Devices`（绑定成功）→ `设备控制台`
- `BLE` 断开后，设备控制台内部回到 `连接卡片` 主逻辑
- 运行时异常：默认不强制弹窗；用户进入 `设备控制台` 后看到对应异常项

---

# 硬件 2.0：设备控制台 UI 设计与线框（方案 B）

> **适用范围**：硬件管理路径（方案 B）下的 **设备控制台** 页 UI 设计与交互线框（命名统一为「设备控制台」，不混用「设备详情」）  
> **来源**：基于 `硬件2.0_硬件管理页改造方案.md`（同目录）已拍板口径  
> **关键口径**：BLE 断开不影响录音；耳机录音模式下任一经典蓝牙断开（`Eureka-Audio` 或桥接）→ **录音停止**  
> **更新时间**：2026-04-15

---

## 1. 目标与非目标

### 1.1 目标

- 让用户在 **设备控制台** 完成三链路的连接编排：**BLE 控制链路 → `Eureka-Audio` 音频链路 → 耳机桥接**。
- 让流程满足“**当前在哪一步 / 下一步做什么 / 做不了为什么**”的可视化闭环。
- 让运行时异常（断链/控制断开）具备**可测试**的 UI 表现与恢复路径。

### 1.2 非目标

- 不在本文件内定义固件协议字段与错误码枚举（仅在 UI 文案层做占位）。
- 不展开录音详情页完整 UI（但会定义其与设备控制台的跳转与提示口径）。
- **录音列表**在断链、未完成等情况下的展示与兜底逻辑 **沿用当前版本**，本文件不重复定义。

---

## 2. 页面信息架构（方案 B）

### 2.1 页面分工

- **Search Devices**：仅负责“找卡片/绑卡片”，绑定成功后进入设备控制台。
- **设备控制台**：全链路入口与状态总控页（本文件主设计对象）。

### 2.2 设备控制台结构：单页纵向（不用 Tab；卡片优先）

> **为何不用 Tab**：并列 Tab 易暗示「卡片 / 耳机同级」，但产品顺序固定为 **先 BLE 连上卡片**，再谈耳机模式；故改为 **单页 + 耳机模式 Toggle**，层级更清晰。

1) **设备头部（Device Header）**  
- 设备名、SN、电量、**App↔卡片 BLE（控制链路）**状态  
- 次要操作：刷新、解绑、帮助  

2) **卡片连接区（优先）**  
- 主路径：**连接卡片**（建立 BLE）；与 1.0 一致，用户进入后**先**完成本节再往下看。  
- BLE 未连接时，页面焦点与主 CTA 落在此区。

3) **耳机模式区（次级，由 Toggle 控制展开）**  
- **Toggle「耳机模式」** 同时承担 **卡片侧经典蓝牙能力开关**（产品语义：打开后才进入 Eureka / 耳机桥接等编排；具体与固件对齐）。  
- **Toggle OFF**：折叠为一行说明 + 开关，不展示耳机引导链。  
- **Toggle ON**：**展开**完整连接逻辑（见 §2.5）。  

4) **辅助信息 / 设备操作区（Support & Actions）**（页底）  
- 帮助、解绑、固件信息等  

### 2.3 Toggle 门禁与「BLE 断开 = 硬件管理页只读」（产品拍板）

| 场景 | Toggle 行为 |
|------|----------------|
| **BLE 未连接** | Toggle **禁用（不可点击）**；文案说明需先连接卡片。 |
| **BLE 已连接** | Toggle **可点击**；用户打开后展开耳机模式引导链。 |
| **BLE 中途断开**（无论经典链路是否仍在线） | **整个硬件管理页进入只读态**：Toggle、扫描、重连、更换耳机等所有依赖卡片控制的操作均禁用；仅保留 **重新连接卡片** 等恢复入口。 |

> 口径统一：**BLE 是否连接 = 当前 App 是否与卡片建立控制连接**。因此一旦 BLE 断开，页面只可展示状态，不应再承诺任何依赖卡片指令的操作可执行。

### 2.3.1 就绪与录音准备页联动（避免假激活）

- **Toggle OFF** 或 **引导链未完成**：耳机录音不可作为就绪路径；录音准备页应 **禁用或引导** 至设备控制台完成引导。  
- **Toggle ON 且引导链完成（耳机桥接就绪）**：可与录音准备页联动为「可录」。  
- **禁止**：仅 Toggle ON 而无展开区内步骤完成提示，让用户误以为已可录。

### 2.4 与 1.0 的差异：BLE 断开时的「连接态」心智

- **无耳机模式（1.0）**：BLE 断开时，用户进入 App 后看到的与「卡片未与 App 连接」一致，录音不受影响。
- **有耳机模式后**：可能出现 **BLE 已断，但手机↔卡片经典链路仍保持（耳机模式相关链路仍在运行）** 的差异化场景——此时 **不能** 与「完全未连接卡片」混用同一套空态。
- **UI 要求**：在设备控制台 **Header** 区分展示，例如：
  - `卡片连接：未连接` + 副文案：`当前 App 无法控制卡片，硬件管理功能暂不可用`
  - 若需补充耳机模式背景，可追加说明：`耳机模式链路可能仍在运行，但本页当前不可操作`
  - 或拆成两行状态：`App 控制链路` / `卡片音频链路（耳机模式）` 由固件/状态源可上报时再细化

### 2.5 Toggle 展开后：完整引导链（顺序与产品一致）

**仅当 Toggle ON 且 BLE 已连接时**展示**纵向步骤条 / 向导**（未完成步骤可折叠，当前步高亮）：

1. **手机系统蓝牙已开启**（若未开 → 引导去系统设置打开蓝牙）  
2. **App 与卡片 BLE 控制链路已连接**（本页上半部应已满足；若中途断开则回退 §2.3）  
3. **手机与卡片经典蓝牙音频 `Eureka-Audio` 已连接**（若未连 → Bottom Sheet / 前往系统蓝牙）  
4. **卡片与蓝牙耳机已桥接**（优先自动回连最近耳机 → 超时/失败则扫描列表）  

> 原 Phase 门禁不变；引导链**收纳在 Toggle 展开区内**，避免与「先连卡片」主路径并列同级。

## 3. 状态与门禁（UI 必须显式表达）

### 3.1 门禁总原则

- **Phase 2（扫描/连接耳机）必须满足**：BLE 已连接 + `Eureka-Audio` 已连接。
- **BLE 断开**：录音是否受影响取决于经典链路，但**硬件管理页一律进入只读态**；所有需要下发指令的控制动作（Toggle、扫描/重连/切换等）均禁用。
- **耳机录音模式下任一经典蓝牙断开**：录音停止；页面进入异常态并引导恢复链路。

### 3.2 引导链步骤的展示规则（仅在 Toggle 展开区内）

- 区块标题建议：**「耳机连接引导」**或**「耳机录音准备」**，与上方「连接卡片」区分层级。
- 每条子步骤仅用 4 种状态：`未就绪 / 进行中 / 已就绪 / 异常`
- 每个子步骤最多一个主动作（如“去开启蓝牙”“去连接”“去扫描”“去修复”），避免多 CTA 分散注意力。

## 4. 文案与视觉规范（用于统一设计稿）

- **避免“权限/授权”**：统一使用“连接音频设备/链路”
- **Step 2 标题**：建议写“连接 Eureka 音频设备（系统蓝牙）”
- **异常提示**：必须包含“结果”（例如：录音已停止 / 控制不可用）+ “下一步”（例如：去系统蓝牙 / 重新扫描）

### 4.1 打扰度策略（通知/Toast/横幅）

结合最新评审结论，本模式优先“**不主动打扰**”，在用户回到 App 或进入设备控制台时再呈现结果：

- **BLE 断开**：默认不弹 Toast/通知；仅在用户进入设备控制台时，通过 Header 状态可见，并将整个硬件管理页置为只读，仅保留“重新连接卡片”等恢复入口。
- **经典链路断开导致停录（耳机模式）**：不强推 App 内即时提醒；用户会在耳机侧立即感知（收不到对方音频/听感中断）。用户回到 App 后，可在 **设备控制台** Toggle 展开区内看到引导链异常与修复入口。  
  **录音文件在列表中的展示方式**由当前版本已有兜底逻辑承接，**本文件不定义**。

### 4.2 Toggle 与状态提示（无 Tab 圆点）

- **不**使用顶部 Tab；状态以 **Header（BLE）** + **Toggle（耳机模式开/关）** + **展开区内步骤** 表达。  
- Toggle 旁可用**小灰/绿点**（可选）表示「耳机链路是否就绪」，避免长文案堆在开关旁。

## 6. 运行时异常（录音进行中时的 UI 结果）

### 6.1 BLE 断开（经典链路仍在）

- **结果**：录音继续；控制不可用
- **设备控制台表现**：
  - Header：`卡片连接：未连接`；耳机模式区见 **§5.7**（整个硬件管理页只读）
  - 展开区内引导链 **② BLE** = 异常/断开；Toggle、扫描/重连/更换耳机等 **全部禁用**
  - 默认不主动打扰；用户进入页或点击操作时就地提示

### 6.2 耳机录音模式下任一经典链路断开（Eureka 或桥接）

- **结果**：录音停止
- **App 侧提示策略**：
  - 不强推即时弹窗/通知（用户在耳机侧已即时感知）
  - 用户回到 App 后，在设备控制台 **Toggle 展开区**内可见异常与修复入口
- **设备控制台表现**：
  - Toggle 旁可选 **灰点**（链路不再就绪）；引导链 **③ Eureka** 或 **④ 耳机桥接** = 异常
  - 主动作：`去连接 Eureka` / `重新连接耳机`

### 6.3 录音进行中访问设备控制台（防误操作，review 要求）

以下动作会**破坏当前音频链路**或**终止当前录音**，在 **当前正在录音** 时须满足其一：

- **默认**：`更换耳机` / `断开耳机` / `去系统蓝牙` 等 **禁用**（灰显 + 说明「录音中不可用」）；或
- **允许点击**：必须 **二次确认**，文案明确 **「将停止当前录音」**。

若产品允许「仅断开 App 控制但不影响录音」，则仅对「会断经典链路」的按钮做上述限制。

## 8. 待 UI 设计落稿时需补齐的清单（不影响线框评审）

- 图标体系：Step 状态（未就绪/进行中/已就绪/异常）与颜色语义
- Bottom Sheet 插画/引导图：连接 `Eureka-Audio`、耳机入仓配对示意
- 超时策略：自动回连倒计时（建议 10–15s）与扫描超时（建议 20–30s）
- 错误文案占位：连接失败/超时/被手机抢占等错误码到用户文案的映射表
