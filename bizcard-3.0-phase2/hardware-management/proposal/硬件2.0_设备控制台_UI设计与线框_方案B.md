# 硬件 2.0：设备控制台 UI 设计与线框（方案 B）

> **适用范围**：硬件管理路径（方案 B）下的 **设备控制台** 页 UI 设计与交互线框（命名统一为「设备控制台」）  
> **来源**：基于 `硬件2.0_硬件管理页改造方案.md` 已拍板口径  
> **关键口径**：
> 1. `BLE` 连接状态 = 当前 App 与卡片的控制连接状态  
> 2. `BLE` 断开时，页面应回到 **搜寻/连接卡片** 的主逻辑，而不是停留在只读态  
> 3. 耳机录音模式下任一经典蓝牙断开（`Eureka-Audio` 或桥接）→ **录音停止**  
> **更新时间**：2026-04-16

---

## 1. 目标与非目标

### 1.1 目标

- 让用户在 **设备控制台** 完成三链路的连接编排：**BLE 控制链路 → `Eureka-Audio` 音频链路 → 耳机桥接**。
- 让流程满足"**当前在哪一步 / 下一步做什么 / 做不了为什么**"的可视化闭环。
- 让运行时异常（断链/控制断开）具备**可测试**的 UI 表现与恢复路径。

### 1.2 非目标

- 不在本文件内定义固件协议字段与错误码枚举（仅在 UI 文案层做占位）。
- 不展开录音详情页完整 UI（但会定义其与设备控制台的跳转与提示口径）。
- **整个硬件管理页不承载开始录音 CTA，也不承载录音页逻辑。**
- **录音列表**在断链、未完成等情况下的展示与兜底逻辑 **沿用当前版本**，本文件不重复定义。

## 2. 页面信息架构（方案 B）

### 2.1 页面分工

- **扫描页（Search Devices）**：负责发现设备、展示已绑定卡片列表、绑定新卡片。BLE 连接成功后自动跳转到设备控制台。
- **设备控制台**：单张卡片的全链路管理（BLE 状态、耳机模式 Toggle、引导链）。BLE 断连后自动回到扫描页。

### 2.1.1 绑定模型

| 关系 | 基数 | 说明 |
|------|------|------|
| 账号 → 卡片 | 1 : N | 一个账号可绑定多张卡片 |
| 卡片 → 账号 | 1 : 1 | 一张卡片只能绑定一个账号，解绑后方可绑定其他账号 |

### 2.1.2 页面路由判断（入口决策）

用户进入设备页时，仅做一个判断：**当前是否有 BLE 连接中的卡片**。

| 条件 | 落地页面 |
|------|----------|
| 有 BLE 连接中的卡片 | 直接进入该卡片的 **设备控制台** |
| 无 BLE 连接 + 无经典蓝牙活跃 | 进入 **扫描页（纯净态）** |
| 无 BLE 连接 + 有卡片经典蓝牙仍在 | 进入 **扫描页 + 顶部第三态横幅** |

> 路由判断发生在每次进入设备页时；管理页内 BLE 断连也触发自动回到扫描页。

### 2.2 设备控制台结构：单页纵向（不用 Tab；卡片优先）

> **为何不用 Tab**：并列 Tab 易暗示「卡片 / 耳机同级」，但产品顺序固定为 **先 BLE 连上卡片**，再谈耳机模式；故改为 **单页 + 耳机模式 Toggle**，层级更清晰。

1) **设备头部（Device Header）**  
- 设备名、SN、电量、**App↔卡片 BLE（控制链路）**状态  
- 次要操作：刷新、解绑、帮助  

2) **卡片连接区（优先）**  
- 主路径：**连接卡片**（建立 BLE）；与 1.0 一致，用户进入后**先**完成本节再往下看。  
- BLE 未连接时，页面焦点与主 CTA 落在此区。

3) **耳机模式区（次级，由 Toggle 控制展开）**  
- **Toggle「耳机模式」** 同时承担 **卡片侧经典蓝牙能力开关**（产品语义：打开后才进入 Eureka / 耳机桥接等编排；具体与固件对齐）。  
- **Toggle OFF**：折叠为一行说明 + 开关，不展示耳机引导链。  
- **Toggle ON**：**展开**完整连接逻辑（见 §2.6）。  

4) **辅助信息 / 设备操作区（Support & Actions）**（页底）  
- 帮助、解绑、固件信息等  

### 2.3 Toggle 门禁与 BLE 断开后的页面逻辑

| 场景 | 页面行为 |
|------|----------|
| **BLE 未连接** | 页面主任务 = `连接卡片`；耳机模式区禁用（Toggle 不可点击） |
| **BLE 已连接** | Toggle 可操作；打开后展开经典链路配置 |
| **BLE 中途断开** | 页面重新回到 `连接卡片 / 搜寻卡片` 主逻辑；耳机模式区禁用；若能拿到旧链路状态，可弱展示，但不允许操作 |

> 口径统一：**BLE 是否连接 = 当前 App 是否与卡片建立控制连接**。  
> 因此只要 BLE 断开，当前页面的主交互就应该重新变成 **搜寻并连接卡片**。

### 2.3.1 就绪与录音准备页联动（避免假激活）

- **Toggle OFF** 或 **引导链未完成**：耳机录音不可作为就绪路径；录音准备页应 **禁用或引导** 至设备控制台完成引导。  
- **Toggle ON 且引导链完成（耳机桥接就绪）**：可与录音准备页联动为「可录」。  
- **禁止**：仅 Toggle ON 而无展开区内步骤完成提示，让用户误以为已可录。

### 2.4 与 1.0 的差异：BLE 断开时的「连接态」心智

- **无耳机模式（1.0）**：BLE 断开时，用户进入 App 后看到的与「卡片未与 App 连接」一致，心智简单。
- **有耳机模式后**：可能出现 **BLE 已断，但手机↔卡片经典链路仍保持（耳机模式相关链路仍在运行）** 的差异化场景——此时 **不能** 与「完全未连接卡片」混用同一套空态。
- **UI 要求**：在设备控制台 **Header** 区分展示，例如：
  - `卡片连接：未连接` + 副文案：`请重新连接卡片后继续管理硬件`
  - 若需补充耳机模式背景：`耳机模式链路可能仍在运行，但当前无法通过 App 控制`（控制在一句话以内）

### 2.5 耳机模式结果态

- **Toggle OFF**：耳机模式未启用，卡片侧经典蓝牙能力关闭。
- **Toggle ON**：耳机模式已启用，卡片侧经典蓝牙能力打开。
- **耳机模式 = ready**：仅当以下两条都为 `connected`
  - `卡片 ↔ 手机（Eureka-Audio）`
  - `卡片 ↔ 耳机`
- **耳机模式 = not ready**：除上述情况外全部属于 `not ready`

> **不定义"准备中"作为耳机模式聚合状态。**  
> 连接过程只在每一条连接项里表达，例如：`disconnected / connecting / connected`。

### 2.6 Toggle 展开后：完整引导链

**仅当 Toggle ON 且 BLE 已连接时**展示引导链（未完成步骤可折叠，当前步高亮）：

1. **手机系统蓝牙已开启**（若未开 → 引导去系统设置打开蓝牙）  
2. **App 与卡片 BLE 控制链路已连接**（本页上半部应已满足；若中途断开则回退 §2.3）  
3. **手机与卡片经典蓝牙音频 `Eureka-Audio` 已连接**（若未连 → Bottom Sheet / 前往系统蓝牙）  
4. **卡片与蓝牙耳机已桥接**（优先自动回连最近耳机 → 超时/失败则扫描列表）  

> 原 Phase 门禁不变；引导链**收纳在 Toggle 展开区内**，避免与「先连卡片」主路径并列同级。

### 2.7 扫描页结构（Search Devices）

扫描页在以下两种子状态间切换：**纯净态** 和 **横幅态（第三态）**。

#### 2.7.1 纯净态 · 线框

```
[NavBar] Search Devices                    [刷新]

[Scan Hero]
 📡 Scanning Nearby Devices
 Automatically searching...

[My Devices]  ── 已绑定卡片列表（1:N）
 Card A  · SN-xxx  · 已连接 · 电量 85%  [管理]
 Card B  · SN-xxx  · 未连接             [连接]

[+ 添加新的 BizCard]  ── 进入全局扫描

[Available Devices]  ── 附近可绑定卡片（扫描结果）
 Card C  · SN-xxx  · 可用    [Bind]

[Before you start]  ── 操作提示
```

**交互规则**：

- 点击 My Devices 中任一卡片的「连接」→ 发起 BLE 连接 → 成功后自动跳转到该卡片的设备控制台
- 点击 Available Devices 中的「Bind」→ 绑定流程 → 绑定成功弹窗 → 确认后进入设备控制台
- 刷新按钮触发重新扫描附近设备

#### 2.7.2 第三态横幅设计（BLE 断 + 经典蓝牙仍在）

**触发条件**：App 层面维护状态 `classicBtActiveCard`——标识哪张卡片的经典蓝牙链路（Eureka-Audio / 耳机桥接）仍然活跃。当 BLE 断开但该状态不为空时，扫描页顶部展示横幅。

```
[NavBar] Search Devices                    [刷新]

┌──────────────────────────────────────────────────┐
│  ⚠️  SN-xxx 的耳机模式仍在运行                   │
│  音频链路活跃中，请先恢复连接或等待链路自行断开    │
│                                                  │
│  [重新连接 BLE]                                  │
│                                                  │
│  其他设备暂不可操作，需先处理当前设备连接         │
└──────────────────────────────────────────────────┘

[My Devices]
 Card A  · SN-xxx  · ⚠️ 耳机模式运行中  [重新连接]   ← 唯一可操作项
 Card B  · SN-xxx  · 未连接             [连接]       ← 禁用灰显

[+ 添加新的 BizCard]  ← 禁用

[Available Devices]
 Card C  · SN-xxx  · 可用    [Bind]   ← 禁用灰显
```

**硬阻断规则**：

| 元素 | 纯净态 | 横幅态（第三态） |
|------|--------|-----------------|
| 当前经典蓝牙活跃的卡片 | 正常「连接」按钮 | 突出「重新连接」按钮（**唯一允许操作**） |
| 其他已绑定卡片的「连接」按钮 | 可用 | **禁用灰显** + 说明文案 |
| Available Devices 的「Bind」按钮 | 可用 | **禁用灰显** |
| 「添加新的 BizCard」 | 可用 | **禁用** |

**横幅交互**：

- **「重新连接 BLE」**：App 尝试重新建立该卡片的 BLE → 成功则自动跳转到该卡片的设备控制台
- **横幅消失条件**：
  - BLE 重连成功（跳转管理页）
  - 经典蓝牙也断开了（App 收到状态变化通知，或固件 5 分钟超时关机）
  - 用户在管理页中明确关闭了耳机模式后断开

**设计意图**：BLE 一次只能连一张卡，在经典蓝牙仍活跃时连接其他卡片会导致旧卡片处于完全不可控的「孤儿态」——录音可能仍在进行但无法停止、耳机模式无法关闭。硬阻断避免了这种不可逆的混乱状态。

#### 2.7.3 退出第三态的途径

| 途径 | 结果 |
|------|------|
| 点击横幅「重新连接 BLE」成功 | 跳转到该卡片管理页，用户可正常操作（含关闭耳机模式） |
| 经典蓝牙自行断开（卡片 5 分钟超时关机、耳机没电） | App 检测到后自动清除横幅，回到纯净扫描态 |
| 用户靠近卡片，BLE 自动重连 | 同第一条 |

#### 2.7.4 状态来源：App 如何知道经典蓝牙还活着？

- **方案 A（缓存推断）**：BLE 断开前，App 已缓存「耳机模式 Toggle ON + 经典链路已连接」的状态 → BLE 断后，视为「经典蓝牙可能仍在」，采取保守策略展示横幅
- **方案 B（系统检测）**：即使 BLE 断开，手机系统蓝牙层仍可通过 `BluetoothAdapter` / `AVAudioSession` 检测到 `Eureka-Audio` 的连接状态 → 可实时确认
- **建议**：两者结合。用 BLE 断前的缓存作为初始判断，同时用系统蓝牙 API 做实时校验（至少能确认 Eureka-Audio 是否仍在）。卡片↔耳机的桥接状态只能靠缓存推断。

## 3. 状态与门禁（UI 必须显式表达）

### 3.1 门禁总原则

- **打开耳机模式 Toggle** 的前置条件：BLE 已连接。
- **Phase 2（扫描/连接耳机）必须满足**：BLE 已连接 + `Eureka-Audio` 已连接。
- **BLE 断开**：页面回到 `连接卡片 / 搜寻卡片` 主流程；耳机模式区不可操作。
- **耳机录音模式下任一经典蓝牙断开**：录音停止；页面展示异常状态与修复入口。

### 3.2 引导链步骤的展示规则（仅在 Toggle 展开区内）

- 区块标题建议：**「耳机连接」**
- 每条子步骤使用 4 种状态：`未就绪 / 进行中 / 已就绪 / 异常`
- 每个子步骤最多一个主动作（如"去开启蓝牙""去连接""去扫描""去修复"），避免多 CTA 分散注意力
- 不使用"准备中"作为总状态徽标

## 4. 文案与视觉规范

- **避免"权限/授权"**：统一使用"连接音频设备/链路"
- **Step 3 标题**：建议写"连接 Eureka 音频设备（系统蓝牙）"
- **异常提示**：必须包含"结果"（例如：录音已停止 / 控制不可用）+ "下一步"（例如：去系统蓝牙 / 重新扫描）

### 4.1 打扰度策略

- **BLE 断开**：默认不弹 Toast/通知；用户回到设备控制台时，直接看到 `连接卡片` 主逻辑。
- **经典链路断开导致停录（耳机模式）**：不强推 App 内即时弹窗；用户在耳机侧可即时感知（听感中断）。回到 App 后，在设备控制台 Toggle 展开区内看到引导链异常与修复入口。
- 整个硬件管理页不做录音启动相关 CTA。

### 4.2 Toggle 与状态提示

- 状态以 **Header（BLE）** + **Toggle（耳机模式开/关）** + **展开区内步骤** 表达。
- Toggle 旁可用 **小灰/绿点**（可选）表示「耳机链路是否就绪」，避免长文案堆在开关旁。

## 6. 运行时异常（录音进行中时的 UI 结果）

### 6.1 BLE 断开（经典链路可能仍在）

- **结果**：录音是否继续由经典链路决定；但从 App 视角看，已失去与卡片的控制连接
- **设备控制台表现**：
  - Header：`卡片连接：未连接`
  - 主区域回到 `连接卡片 / 搜寻卡片` 逻辑
  - 耳机模式区禁用，等待 BLE 恢复后再继续配置
  - 默认不主动打扰；用户进入页面时就地呈现

### 6.2 耳机录音模式下任一经典链路断开（Eureka 或桥接）

- **结果**：录音停止
- **App 侧提示策略**：
  - 不强推即时弹窗/通知（用户在耳机侧已即时感知）
  - 用户回到 App 后，在设备控制台 Toggle 展开区内可见异常与修复入口
- **设备控制台表现**：
  - Toggle 旁可选 **灰点**（链路不再就绪）；引导链 **③ Eureka** 或 **④ 耳机桥接** = 异常
  - 修复入口：`去连接 Eureka` / `重新连接耳机`

### 6.3 录音进行中访问设备控制台（防误操作）

以下动作会**破坏当前音频链路**或**终止当前录音**，在 **当前正在录音** 时须满足其一：

- **默认**：`更换耳机` / `断开耳机` / `关闭耳机模式` 等 **禁用**（灰显 + 说明「录音中不可用」）；或
- **允许点击**：必须 **二次确认**，文案明确 **「将停止当前录音」**。

本页不提供开始录音按钮。

## 7. 页面跳转与来源（含路由）

### 7.1 页面路由

| 来源 | 条件 | 目标 |
|------|------|------|
| 任意入口进入设备页 | 有 BLE 连接中的卡片 | 设备控制台（该卡片） |
| 任意入口进入设备页 | 无 BLE + 无经典蓝牙活跃 | 扫描页（纯净态） |
| 任意入口进入设备页 | 无 BLE + 经典蓝牙仍在 | 扫描页 + 第三态横幅 |

### 7.2 页面间跳转

- **扫描页** → 点击「连接」/「Bind」成功 → **设备控制台**
- **设备控制台** → BLE 断开（无经典蓝牙活跃）→ **扫描页（纯净态）**
- **设备控制台** → BLE 断开（经典蓝牙仍在）→ **扫描页 + 第三态横幅**
- **设备控制台** → 用户点击「返回」→ **扫描页**（BLE 连接可保持，再次进入仍进管理页）
- **设备控制台**（W6，全链路就绪）→ **录音准备页**
- 运行时异常：默认不强制弹窗；用户进入 `设备控制台` 后落在具体异常 Step
