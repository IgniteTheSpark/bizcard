# 硬件 2.0《硬件管理页改造方案》Review

> review 角色：硬件工程 + 交互设计
> review 对象：`proposal/硬件2.0_硬件管理页改造方案.md`
> review 日期：2026-04-15
> review 结论：**当前 proposal 已具备讨论基础，但还不适合直接进入实现评审。建议先补齐状态机口径、链路终点定义、异常门禁与交互阈值，再进入 UI 定稿。**

---

## 1. 总体评价

这份 proposal 的优点是：

- 已经把「卡片 / 手机 / 蓝牙耳机」三方关系讲清了大半。
- 能看出作者在尝试把现有 `Search Devices` 页面从“找设备”升级为“连接编排”。
- 对 `Phase 1` / `Phase 2` 的主流程拆分是合理的，适合作为 UI 方案的基础。

但从硬件实现闭环和交互防呆角度看，文档目前仍有几个关键问题：

- 同一能力在不同文档中存在**互相冲突的定义**。
- Proposal 对“**前置条件**”“**异常降级**”“**状态来源**”写得不够硬，容易在联调时才暴露问题。
- 一些命名会误导评审，把“系统蓝牙连接”理解成“权限授权”。

---

## 2. 阻塞级问题（建议评审会先拍板）

### 2.1 5 分钟超时后的设备状态定义冲突

**问题**

相关文档里同时出现了“5 分钟未连接后进入**深度休眠**”和“5 分钟未连接后进入**关机**”两种说法。

**为什么这是阻塞项**

- 这会直接影响唤醒方式、重连耗时、功耗模型和用户心智。
- App 侧也无法定义“等待设备回来”的轮询策略和失败文案。
- 固件、交互、测试三方会基于不同假设工作，最后一定返工。

**建议**

- 必须统一成一个唯一口径，并写成权威定义。
- 如果最终是“深度休眠”，要补充“如何被唤醒、是否保持部分蓝牙态、用户看到什么”。
- 如果最终是“关机”，要补充“用户如何重新进入广播态、是否需要重新建立完整链路”。

**建议结论**

先统一“超时终态”，再讨论页面文案和状态机。

### 2.2 角键“闪念”触发阈值冲突

**问题**

现有文档中同时存在：

- 闪念长按 `0.5s`
- 闪念长按 `0.1-0.2s`

**为什么这是阻塞项**

- 这个阈值决定误触率、学习成本和与“单击打点”的区分度。
- `0.1-0.2s` 更接近“稍重一点的点击”，很容易和打点冲突。
- 如果工程实现先写死一个值，后续交互再改会波及固件判定、震动反馈、测试用例。

**建议**

- 统一一个最终阈值，并给出工程容差。
- 结合“会议中单击打点”的存在，我更倾向于保留更清晰的长按阈值，不建议过短。
- 所有引用角键行为的文档都要同步刷新，不能保留 PM 建议版和实现版并存。

### 2.3 第三条蓝牙链路的音频终点定义不一致

**问题**

Proposal 的叙述更接近“耳机麦克风的声音进入卡片并参与卡片录音”，但另一个流程文档里又写成“耳机麦克风声音传回手机”。

**为什么这是阻塞项**

- 这会让评审无法判断：录音主设备到底是**卡片**还是**手机**。
- 一旦主叙事不清，App 状态命名、失败处理和后续录音文件归属都会混乱。
- 对硬件链路设计来说，这是架构级歧义，不是文案小问题。

**建议**

- 明确写一句权威定义：**录音编码与文件落点在哪一侧**。
- 如果手机只是会议音频来源，而录音始终在卡片侧完成，就不要再出现“传回手机”这种表述。
- 如果确实有一部分上行需要回手机，也要拆成“通话链路”和“录音链路”分别说明。

### 2.4 “耳机强接管”规则和“闪念录音”规则未闭环

**问题**

一处文档说：只要耳机已连接，任何录音操作都应该强制走耳机麦克风。另一处又保留了“闪念可调用环境麦 / 耳机麦”的表述。

**为什么这是阻塞项**

- 这会导致状态机优先级无法落地。
- 测试同学无法写确定性的 case。
- 用户也会在“为什么我明明连了耳机，闪念还走环境麦”这类场景里产生困惑。

**建议**

- 补一张明确的优先级矩阵：
  - 是否已连耳机
  - 当前是否录音中
  - 用户操作是主键还是角键
  - 滑块位置是什么
  - 最终启用哪个麦克风
- 如果“闪念”是例外能力，需要明确写成“例外策略”，不能只靠上下文猜。

### 2.5 Proposal 没有把 BLE 控制链路当成 Phase 2 的硬前置条件

**问题**

文档写了 App 通过 BLE 指令让卡片扫描耳机，但 Proposal 的主流程里没有把“BLE 是否已握手”写成 Phase 2 的门禁条件。

**为什么这是阻塞项**

- 用户可能已经在系统里连上了 `Eureka-Audio`，但 BLE 控制链路并没有准备好。
- 这时页面如果允许“开始扫描耳机”，就会出现 UI 看似可操作、实际指令发不出去的假闭环。
- 这是非常典型的联调事故源头。

**建议**

- 在 Proposal 中明确区分至少三种状态：
  - 已发现卡片 / 已绑定
  - 已建立 BLE 控制链路
  - 已建立系统音频路由 `Eureka-Audio`
  - 已连接耳机桥接
- Phase 2 必须依赖“BLE 控制链路可用”，否则只允许展示引导，不允许进入扫描态。

## 3. 中等级问题

### 3.1 “系统授权”命名不准确，容易误导

当前 `Phase 1` 标题带有“系统授权”色彩，但实际动作更像“去系统蓝牙里连接音频设备”，不是常见意义上的权限弹窗。

**建议**

- 改成更准确的表达，例如“连接手机音频链路”或“连接 Eureka 音频设备”。
- 避免使用“权限”一词，否则用户会误以为是在申请麦克风权限或蓝牙权限。

### 3.2 Proposal 对异常链路着墨不足

当前文档重点放在主成功路径，但以下异常没有形成完整产品闭环：

- `Eureka-Audio` 未连接或中途断开
- BLE 已断但经典蓝牙还在
- 耳机扫描超时
- 耳机连接失败
- 耳机录音中突然断电或断连
- 返回 App 后状态未及时刷新

**建议**

- 至少补齐“用户看到什么 + 系统怎么降级 + 是否允许重试”的定义。
- 不然 UI 线框完成了，真实体验还是会断。

### 3.3 信息架构 A / B 还停留在页面层，缺少“状态所有权”视角

Proposal 已提出：

- A：同页扩展
- B：两级结构

但真正需要先拍板的，不只是页面在哪，而是：

- 谁是“硬件状态总控页”
- 录音准备页和设备控制台是否共用同一状态源
- 当状态变化来自硬件上报而不是用户点击时，哪一页负责兜底呈现

**建议**

在 A / B 之外，再补一个“状态所有权”小节，不然页面方案容易看起来合理，但最终状态不一致。

### 3.4 缺少“历史耳机 / 最近耳机”的交互策略

文档已把这个问题放进待评审清单，但从体验上看，它其实不是可有可无的小优化，而是决定耳机录音模式是否能高频复用的关键点。

**建议**

- 至少定义是否支持“一键重连最近一次耳机”。
- 如果不支持，也要说明原因，例如系统限制、硬件限制或首版范围控制。

### 3.5 屏幕反馈与 App 反馈之间还缺一层协同原则

目前文档提到卡片小图标、Toast、扫描页等，但还没有一个总原则说明：

- 哪些状态以卡片为主反馈
- 哪些状态以 App 为主反馈
- 哪些状态两端都要反馈，但避免重复打扰

**建议**

补一段“反馈分工原则”，否则设计时会出现两端都提示或者两端都不提示的问题。

## 4. 我建议 Proposal 必补的内容

如果希望这份 Proposal 能真正支持后续实现和评审，我建议至少补齐以下 5 块：

### 4.1 统一状态机总表

把以下对象放进一张表：

- BLE 控制链路
- `Eureka-Audio` 系统音频链路
- 卡片与耳机桥接链路
- 当前录音状态
- 当前麦克风来源
- 当前 UI 应展示的主状态

这张表会极大提升跨团队对齐效率。

### 4.2 前置条件与门禁条件

对每一个主动作，明确写清：

- 用户是否可点击
- 如果不可点击，拦截在哪里
- 拦截文案是什么

尤其是：

- 什么时候允许“开始扫描耳机”
- 什么时候允许进入“耳机录音模式”
- 什么时候只能提示先连卡片 / 先连音频设备

### 4.3 异常与降级策略

至少覆盖：

- 耳机扫描不到
- 耳机连接失败
- 连接中超时
- 录音中耳机断开
- App 退后台后回来状态不同步

并且要写明：

- 是否自动重试
- 是否自动降级
- 是否中断当前录音
- 用户如何感知

### 4.4 命名与术语统一

建议统一以下术语，不要混用：

- “系统授权” vs “系统蓝牙连接”
- “耳机录音模式” vs “耳机连接模式”
- “手机音频权限” vs “连接 Eureka-Audio”
- “休眠” vs “关机”

### 4.5 决策项与非决策项分离

Proposal 目前把“已知方案”和“待拍板问题”混在一起。建议单独列一个 `Decision Needed` 区块，把评审会真正要拍的内容限定在 5 项以内。

我建议优先拍这 5 项：

1. 5 分钟超时后的唯一终态是什么。
2. 角键闪念的最终触发阈值是多少。
3. 录音文件的主落点是在卡片还是手机。
4. 耳机连接时，闪念是否也必须强制走耳机麦。
5. 信息架构选 A 还是 B，以及状态源归谁。

## 5. 对当前方案方向的判断

如果只看当前文档，我**倾向支持方案 B：两级结构**，原因如下：

- 现有 `Search Devices` 的用户心智就是“找卡片 / 绑卡片”，不适合继续塞入三条蓝牙链路与耳机桥接状态。
- 耳机录音模式本质上已经不是“搜设备”问题，而是“连接编排 + 状态管理”问题。
- 后续不管是耳机状态、麦克风来源、异常降级还是录音前检查，都更适合落在“设备控制台 / 设备详情页”。

但即使选 B，也不能跳过前面的状态机统一工作。否则只是把问题从一个页面搬到另一个页面。

---

## 6. 建议的下一步

建议按下面顺序推进：

1. 先用半小时把所有冲突口径拍平，输出一版“唯一事实源”。
2. 再补一张跨链路状态机表。
3. 然后再定信息架构 A / B。
4. 最后才进入页面线框和视觉细节评审。

如果直接先做 UI，会很快陷入“页面看起来完整，但真实状态全是灰区”的问题。

---

## 7. 一句话结论

**这份 proposal 的方向是对的，但目前更像“可讨论的交互草案”，还不是“可进入实现的方案文档”。在硬件链路、状态优先级、异常门禁没有统一前，不建议直接开做。**

## 8. 第二轮 Review（针对 2026-04-15 定稿版）

### 8.1 先说结论

新版主方案相比上一版有明显进步，以下关键问题已经被正面收敛：

- `5 分钟超时终态` 已统一为 **关机**。
- `闪念阈值` 已统一为 **长按 0.5s**。
- `录音文件主落点` 已明确为 **卡片侧**。
- `耳机强接管` 已明确覆盖到 **闪念**。
- `Phase 2` 已补上 **BLE 控制链路门禁**。
- `信息架构` 已明确倾向并拍板为 **方案 B**。

从架构方向上看，这版已经可以进入“实现评审准备”阶段；但从文档治理和测试可执行性角度看，仍有几个问题需要继续收口。

### 8.2 当前仍需处理的 review findings

#### Finding 1：文档自称“定稿/可实现评审”，但正文里仍保留未收口的旧表述

**问题**

文档头部已经写成“定稿 / 可实现评审”，并且 §16 标记为“已拍板决策”，但正文其他位置仍保留了旧状态的描述：

- §6.2 仍写“评审时选定 A 或 B”
- §7 仍写“P0 选定信息架构 A/B”
- §9 仍写“状态机总表 v0”
- §11 仍写“方向性策略（待与固件对齐）”
- §15 仍写“首版至少定义其一：支持一键重连最近耳机，或明确不支持”

**为什么这仍是问题**

- 这会让评审人不知道哪些内容已经拍板，哪些仍是待定项。
- 开发、测试、产品分别读取不同段落时，可能得出不同结论。
- 如果这份文档要承担“唯一事实源”角色，就不能同时存在“已定稿”和“待评审”两套语气。

**建议**

- 把所有已被 §16 覆盖的“待定”表述改为最终口径。
- 如果某些内容确实还没定，就不要再把文档标题写成“定稿 / 可实现评审”。

#### Finding 2：Proposal 声称“相关文档已清理”，但关联文档里仍保留旧冲突口径

**问题**

主方案 §16 已写明：

- `闪念长按阈值 = 0.5s`
- “相关文档中的含混表述已清理”

但当前关联文档 `硬件2.0 录音标记功能.md` 仍然保留：

- `长按 0.1-0.2s`
- “PM 建议：闪念是长按 0.1-0.2s”

也就是说，主方案已经定稿，但下游文档并没有真的同步。

**为什么这仍是问题**

- 测试用例、交互验收、固件联调时，很容易继续引用旧文档。
- 这类“主方案是新的、子文档还是旧的”问题，在联调阶段会非常伤效率。
- Proposal 既然强调“唯一事实源”，就应该把明显冲突的下游文档一并清干净。

**建议**

- 立即同步修订 `硬件2.0 录音标记功能.md`。
- 同时顺手统一其中的屏幕尺寸表述（当前仍有 `0.96` / `0.97` 混用）。

#### Finding 3：`Eureka-Audio` 中途断开时的产品结果仍不够清楚，当前“降级为机身麦”不够闭环

**问题**

在 §11 中，`Eureka-Audio` 未连或中途断开的策略被写为：

- 阻断耳机模式，或
- 降级为机身麦

这个表述在“本地录音”场景下还能成立，但在“会议 / 通话录音”场景下不够完整，因为：

- `Eureka-Audio` 承担的是 **手机下行音频来源**
- 一旦它断开，丢失的不只是“麦克风输入方式”，更可能是“对方声音”整条来源

**为什么这仍是问题**

- 如果只是写“降级为机身麦”，容易让人误以为会议录音还能完整继续。
- 实际上此时最多只能保证“我的声音”继续采集，对“对方声音”是否还能持续录入，需要单独定义。
- 这是测试预期和用户预期都很敏感的地方。

**建议**

- 单独补一条：`会议/通话录音中，若 Eureka-Audio 断开，录音结果如何定义`
- 至少明确下面 4 点：
  - 当前录音是否继续
  - 是否只保留本地麦克风音轨
  - 是否提示“对方声音可能中断”
  - 最终文件/转写是否要带异常标记

#### Finding 4：BLE 在录音中断开时，文档只定义了“不能扫描”，但没有定义“现有录音态怎么表现”

**问题**

当前文档对 BLE 断开主要描述为：

- 禁止发扫描指令
- 提示控制链路断开

但没有明确说明：如果用户已经处于耳机录音或会议录音中，BLE 中途断开时：

- 正在录的内容是否受影响
- UI 主状态是否仍显示“录音中（耳机麦）”
- 什么时候才需要把用户拉回“先连接卡片”

**为什么这仍是问题**

- BLE 是控制链路，不等于音频链路。
- 如果只是因为 BLE 断了，就把 UI 回退成“先连接卡片”，会误导用户觉得录音已经失效。
- 这会直接影响状态栏、录音页提示和故障恢复体验。

**建议**

- 在状态机里补一个运行时场景：
  - `BLE 断开，但 Eureka-Audio 与耳机桥接仍在`
- 明确此时产品语义是：
  - “当前录音继续，但失去远程控制能力”
  - 还是“立即进入异常态并提醒用户”

### 8.3 第二轮结论

这版主方案已经从“草案”推进到了“可讨论的实现方案”，方向上我基本认可；但如果你们希望它真的成为研发和测试可直接执行的定稿，我建议至少再完成两件事：

1. 把所有关联文档里的旧口径清理干净。
2. 把 `Eureka-Audio` 断开、BLE 断开这两类运行时异常写成更可测试的明确结果。

做完这两步，这份方案会稳很多。

## 9. 第三轮 Review（针对最新修订版）

### 9.1 本轮先说好的部分

这次更新里，有两点是明确变好的：

- `硬件2.0 录音标记功能.md` 里关于 `闪念 0.1-0.2s` 的旧口径已经基本清掉，至少主动作表已经统一为 `0.5s`。

但与此同时，主方案里又出现了一个新的高优先级冲突。

### 9.2 Findings

#### Finding 1：主方案内部又出现“断连后终止录音” vs “断连后无缝降级”的硬冲突

**严重级别：高**

**问题**

主方案前面仍然写着：

- 耳机已连接时强接管；
- **断连时在约定时间内降级到物理滑块对应麦克风**。

但在新的 §11、§9 里，又把运行时结果写成：

- `Eureka-Audio` 中途断开：**立即终止录音**
- 卡片 ↔ 耳机桥接中途断开：**立即终止录音**
- 状态机里也写成 `录音中断（手机音频链路断开）`、`录音中断（耳机桥接断开）`

这和你们此前的需求文档、以及主方案第 2 节自己的描述是冲突的。

**为什么这是问题**

- 这是比“文案没统一”更严重的实现冲突，因为它直接改变了产品行为。
- 如果测试按 §11 验证，会测出“断连立即停录”；如果固件按需求文档实现，会做成“无缝降级继续录”。
- 最终会出现最糟糕的一种情况：产品评审时以为已经拍板，联调时才发现 App 和固件对断连策略理解完全不同。

**建议**

- 必须明确区分两种断连：
  - `耳机桥接断开`
  - `Eureka-Audio` 下行音频断开`
- 然后逐条定义：
  - 当前录音是否继续
  - 哪些音轨还能保留
  - 是否允许降级到机身麦
  - 是否必须中断并生成异常文件

如果你们最终决定从“无缝降级”改为“立即终止录音”，那就必须同步改掉：

- 主方案 §2 的“断连时降级”描述
- `_硬件2.0 需求内容.md` 里的无缝降级定义

否则现在是明显的双口径。

#### Finding 2：文档已经写“已拍板方案 B”，但正文仍保留“A/B 待评审”语气

**严重级别：中**

**问题**

这一轮里，`§16` 仍明确写了：

- 已拍板：`方案 B（两级结构）`

但正文 `§6.2` 和 `§7` 还保留：

- “评审时选定 A 或 B”
- “P0 选定信息架构 A/B”

**为什么这是问题**

- 这会导致开发、设计、测试看到不同段落时拿到不同结论。
- 文档既然想承担“唯一事实源”，这类旧口吻就不该继续保留。

**建议**

- `§6.2` 保留 A/B 对比可以，但标题和结尾要明确改成“**已评估选型**”或“**备选方案回顾，最终选 B**”。
- `§7` 里的落地阶段也要改成“按 B 推进”，不要再写“选定 A/B”。

#### Finding 3：`历史耳机 / 最近耳机` 仍然没有结论，不适合挂在“已拍板关键决策”文档里悬空

**严重级别：中**

**问题**

`§15` 还在写：

- “首版至少定义其一——支持一键重连最近一次耳机，或明确不支持”

也就是说，这一块其实仍未定。

**为什么这是问题**

- 这个点虽然不是架构阻塞项，但它已经进入体验闭环范围了。
- 如果文档状态是“进入实现评审准备”，这种仍未决的体验点最好显式标成 `Open Question`，而不是混在主方案正文里。

**建议**

- 二选一：
  - 直接拍板首版是否支持最近耳机一键重连；
  - 或者把这一段挪到单独的“未决事项 / Out of Scope”里。

### 9.3 第三轮结论

这轮更新说明你们已经开始把方案往“可测试、可实现”方向推进了，尤其是 `BLE 断开` 的处理比上一版更像工程化定义。

但当前最大的风险，已经从“信息不够”变成了“**运行时异常策略前后打架**”。  
我认为现在最优先要拍平的，不再是页面结构，而是这一句：

**耳机模式录音过程中，任一经典蓝牙链路断开，到底是继续录、降级录，还是立即停录？**

这句话如果不统一，后面的 UI、提示文案、测试 case 都会漂。

### 10.1 结论

这一轮相比上一轮继续向前推进了：

但我这轮 review 仍然看到 3 个需要继续清理的问题，其中 1 个我认为是高优先级一致性问题。

### 10.2 Findings

#### Finding 1：主方案与需求文档仍残留“断连降级继续录”的旧句子，和当前“终止录音”口径冲突

**严重级别：高**

**问题**

虽然你们已经把异常策略主体改成了“经典蓝牙链路断开即终止录音”，但几个旧句子还在：

- 主方案 `§2` 仍写“断连时在约定时间内 **降级** 到物理滑块对应的麦克风”
- 主方案 `§16.4` 仍写“仅在耳机未连接或**断连降级**时，才使用侧边滑块对应麦克风”
- `_硬件2.0 需求内容.md` 的 `\[闪念\]` 条目里也仍写“耳机断连降级时，才使用侧边物理滑块对应麦克风”

但你们这轮已经明确：

- 会议/通话录音（耳机模式）任一经典蓝牙链路断开 → **终止录音**
- “无缝降级继续录音”不再作为产品承诺

**为什么这是问题**

- 这会让实现同学继续怀疑：到底还有没有某种“断连后切机身麦继续录”的例外。
- 测试在写 case 时也会卡住：闪念是不是还保留降级，还是所有耳机态都应该停录。
- 这类残留句子会削弱你们“唯一事实源”的可信度。

**建议**

- 把所有“断连降级”措辞改成更精确的条件表达：
  - “耳机未连接时，使用侧边滑块对应麦克风”
  - 不再写“耳机断连降级时”
- 如果你们仍然想保留某个非会议场景的降级例外，就必须单独点名该场景，而不是继续用泛化表述。

#### Finding 2：文档虽然已拍板方案 B，但 `§7` 和 `§8` 仍残留“未定/倾向”语气

**严重级别：中**

**问题**

这轮 `§6.2` 已经收好了，但还有两个地方没完全同步：

- `§7` 仍写“P0：选定信息架构 A/B”
- `§8` 仍写“Review 方倾向方案 B”

而 `§16` 已经明确写成：

- 已拍板：采用 `方案 B`

**为什么这是问题**

- 这会让文档呈现出“正文已定、摘要还没定”的错位感。
- 对第一次阅读的人来说，会降低结论感。

**建议**

- `§7` 改成“按方案 B 推进设备控制台路径落地”。
- `§8` 改成“Review 已收敛并支持方案 B”，不要再用“倾向”。

#### Finding 3：`§11` 的“录音中耳机断电或断连（非耳机模式）”表述语义不清

**严重级别：中**

**问题**

当前这句：

- `录音中耳机断电或断连（非耳机模式）`

从字面上会让人困惑：

- 既然是“非耳机模式”，为什么“耳机断电或断连”会成为这个分支标题？
- 这里到底描述的是“耳机曾连接但当前录音没走耳机模式”，还是“本机录音场景下耳机状态变化无需关心”？

而且后面的策略又写成：

- 统一按产品策略终止或提示用户重新开始（具体以录音类型定义）

这其实还是偏模糊。

**为什么这是问题**

- 这个分支标题不利于研发落状态机，也不利于测试命名 case。
- 当文档其他地方都在往“可测试口径”收敛时，这一条会显得格外虚。

**建议**

- 把这个场景改成更明确的标题，例如：
  - “本机录音场景下的外部耳机状态变化”
  - 或 “未启用耳机录音模式时，耳机链路变化的处理”
- 并明确一句最终语义：
  - “若当前录音未依赖耳机链路，则耳机断开不影响当前录音”
  - 或者相反，明确说仍需终止

### 10.3 第四轮结论

这版已经接近可以收尾了，说明你们的 debate 已经开始收敛到真正的实现口径，而不是还停留在概念层。

- **大的架构和核心决策基本可以过**
- **剩下主要是“最后一公里的文档一致性清理”**

如果把上面 3 个点再抹平，这份方案就会从“基本靠谱”升级成“研发和测试拿去就能干”的状态。

## 11. 第五轮 Review（并判断是否可进入 UI 设计）

### 11.1 结论先说

我的判断是：

- **可以开始进入 UI 设计逻辑了**
- 但更准确地说，是可以进入 **信息架构、主流程线框、关键状态页与异常态占位** 的设计阶段
- **不建议立刻冻结全链路高保真**，因为还有少数几处逻辑口径没彻底擦平

也就是说，当前方案已经不再卡在“能不能设计”，而是进入了“设计时哪些点必须标注为待确认”的阶段。

### 11.2 仍会阻塞 UI 的问题

#### Finding 1：主方案正文里仍残留“断连后降级”的旧口径

**严重级别：高**

当前主方案里仍能看到这类表述：

- `§2`：耳机断连时在约定时间内降级到物理滑块对应麦克风
- `§16.4`：耳机未连接或**断连降级**时，才使用侧边滑块对应麦克风

而与此同时，你们已经在：

- `§11`
- `_硬件2.0 需求内容.md`

里明确改成：

- **耳机模式下任一经典蓝牙链路断开 → 终止录音**
- “无缝降级继续录音”不再作为产品承诺

**为什么它会阻塞 UI**

- 这会直接影响异常态设计：到底是“录音中断页/横幅/结果标记”，还是“录音继续但切到机身麦”的过渡态。
- 如果这个口径不清，设计师会多画一整组根本不存在的降级状态。

**建议**

- 在进入 UI 设计前，把主方案里所有“断连降级”旧句全部删干净或改写成最终口径。

#### Finding 2：`BLE 断开后是否会级联拉断经典蓝牙链路` 仍取决于固件策略

**严重级别：中高**

当前 `§11` 已定义：

- 如果 BLE 断开但两条经典蓝牙还在，则录音继续，只失去控制能力
- 但如果固件策略是 BLE 断开会主动断开桥接，则按“经典链路断开 → 停录”处理

**为什么它会阻塞 UI**

- 这会影响录音中的异常优先级、横幅文案、是否跳出中断提示。
- 如果现在直接做高保真，后续可能因为固件联动策略变化而重画录音中异常表现。

**建议**

- 如果当前只是要开线框，这个点可以先用 `TODO/待固件确认` 标注。
- 如果要直接做高保真和交互细节，就建议先跟固件把这个分支拍死。

### 11.3 不阻塞 UI、可以边设计边补的问题

下面这些我认为已经不再阻塞 UI 结构设计：

- 方案 B 已拍板，设备控制台/设备详情作为总控页的方向已足够支撑页面结构。
- `最近耳机自动回连` 已经从“未定”变成“支持的方向”，足以先画等待回连态。
- `Phase 1 / Phase 2` 的门禁和主流程已经够清晰，可以支撑主成功路径。
- `BLE 断开但经典链路仍在` 的场景已经可以画出“录音继续 + 控制链路断开横幅”。
- 返回系统设置后的轮询、错误码映射、法务文案、具体 Toast 样式都可以后置。

### 11.4 进入 UI 设计前最小必锁清单

如果你们现在要正式进入 UI 设计，我建议至少锁定下面这些输入：

1. **页面结构**：采用方案 B，`Search Devices` 只负责发现/绑定，`设备控制台/设备详情页` 承担全链路状态与耳机模式。
2. **默认落地**：绑定成功后进入设备控制台，而不是回纯设备列表。
3. **主流程**：耳机模式 = `Phase 1 连接 Eureka-Audio` → `Phase 2 扫描/连接耳机`。
4. **门禁**：BLE 未就绪时，禁止出现可点击的“开始扫描耳机”。
5. **主状态集**：至少按 `先连接卡片 / 需连接 Eureka-Audio / 可配耳机 / 耳机已就绪 / 录音中 / 控制链路断开 / 录音中断` 这些状态去建线框。
6. **异常主策略**：耳机模式下经典蓝牙链路断开 → 停录；BLE 断开且经典链路还在 → 继续录，但失去控制能力。
7. **状态源**：设备控制台和录音准备页使用同一状态源。
8. **自动回连范围**：本期 UI 是否包含“等待回连最近耳机”这一态，如果包含，要在信息架构里占位。

### 11.5 最终判断

如果你问的是：

- **“现在能不能开始做页面结构和交互逻辑？”**

我的回答是：

- **能，可以开始。**

如果你问的是：

- **“现在能不能把所有高保真和异常细节一次性定死？”**

我的回答是：

- **还不建议。**

最合理的推进方式是：

1. 先进入 UI 逻辑设计，产出 IA、主流程线框、关键状态页。
2. 并行把“断连降级”的残余旧口径清干净。
3. 在固件确认 `BLE 断开是否级联拉断经典链路` 后，再冻结高保真异常态。

## 12. UI 线框 Review（针对 `硬件2.0_设备控制台_UI设计与线框_方案B.md`）

### 12.1 先说总体判断

这份 UI 线框稿整体方向是对的，而且已经明显超过“概念草图”阶段了。

我认可的点有：

- 方案 B 的信息架构落得比较稳，`Search Devices` 和 `设备控制台` 的职责分开了。
- `3-step` 只放在耳机模式内部展示，这个处理是对的，避免误导普通用户。
- `BLE 断开但经典链路仍在` 被设计成“录音继续 + 控制不可用”，符合已拍板逻辑。
- `自动回连最近耳机` 和 `扫描兜底路径` 被拆成两条流程，用户心智是清楚的。

所以结论不是“不能做”，而是：

- **这版线框可以继续往下走**
- 但在进高保真前，还需要补几条关键交互定义，否则后面会反复

### 12.2 Findings

#### Finding 1：`耳机录音模式 [ON]` 与“链路已就绪可用”之间的语义没有被拆开，容易造成假激活

**严重级别：高**

在 `W3 / W4 / W5` 中，页面已经显示：

- `耳机录音模式 [ON]`

但此时实际上可能只是：

- 用户表达了“我要进入耳机模式”的意图
- 还没有完成 `Eureka-Audio` 连接
- 也还没有完成耳机桥接

也就是说，当前 UI 里：

- `开关 ON`
- `耳机模式已完全可用`

这两个语义被混在一起了。

**为什么这是问题**

- 如果录音准备页和设备控制台共用同一状态源，另一个页面可能会把这个 `ON` 理解为“模式已生效”。
- 用户也会误以为：我都看到 `ON` 了，为什么还不能直接录。
- 这是典型的“视觉上已激活，系统上未就绪”的假激活问题。

**建议**

- 把“模式意图”和“模式就绪”拆成两层状态：
  - `意图已开启 / 准备中`
  - `已就绪 / 可用于耳机录音`
- UI 上可以有两种做法，任选一种：
  - 开关维持 `OFF`，进入准备流程时显示“正在开启耳机模式”
  - 开关显示 `ON`，但旁边必须强提示 `准备中` / `未完成连接`

如果不拆这两个层级，后续联动页一定会出问题。

#### Finding 2：缺少“录音进行中”的正常态控制规则，当前 `更换耳机 / 断开耳机 / 关闭耳机录音模式` 存在误操作风险

**严重级别：高**

`W6` 里给了这些次要动作：

- `更换耳机`
- `断开耳机`
- `关闭耳机录音模式`

但文档没有说明：

- 当用户已经在录音中时，这些动作是否仍可点
- 点击后是否弹二次确认
- 是否会直接终止当前录音

**为什么这是问题**

- 这些动作本质上都会破坏当前音频链路。
- 如果录音中还能直接点，用户很容易自己把录音打断。
- 这不是边角问题，而是一个典型的高风险交互口。

**建议**

- 增加一条明确规则：
  - `录音中，设备控制台内所有会破坏链路的动作默认禁用`
  - 或者 `允许点击，但必须二次确认，并明确提示“这会停止当前录音”`
- 最好补一个“录音进行中访问控制台”的简版状态，不一定要单独成页，但至少要定义动作权限。

#### Finding 3：`设备控制台结构（从上到下固定三块）` 但正文只写了两块，结构定义不完整

**严重级别：中**

在 `§2.2` 中写的是：

- 从上到下固定三块

但实际只列出了：

1. `设备头部`
2. `耳机录音模式编排区`

第三块没有写出来。

**为什么这是问题**

- 这会让设计师和产品对页面底部区域的职责理解不一致。
- 也不利于后面继续补充诸如“最近耳机信息”“设备操作区”“帮助说明区”的落位。

**建议**

- 直接补齐第三块，并命名清楚。
- 如果第三块就是“辅助信息 / 帮助 / 设备操作区”，建议现在就写死，不要留隐形结构。

#### Finding 4：`W4 自动回连` 缺少“取消后去哪里”的状态闭环

**严重级别：中**

在 `W4` 中有：

- 回连中（15s） `[取消]`
- `[重新扫描其他耳机]`

但没有定义：

- 点 `取消` 之后页面回到哪里
- 是回到 `耳机模式 ON 但未就绪`
- 还是直接退回 `OFF`
- 还是进入 `W5` 的扫描入口

**为什么这是问题**

- 用户一旦取消自动回连，系统必须立刻给出新的稳定态。
- 否则线框虽然可看，但真实交互会断在中间。

**建议**

- 明确取消后的唯一落点。
- 我更倾向于：`取消自动回连 → 落到 W5 的手动扫描入口`，这样最直观。

#### Finding 5：`设备控制台 / 设备详情` 的命名仍在混用，UI 落稿前需要统一

**严重级别：中**

当前 UI 线框文档和主方案里，仍在交替使用：

- `设备控制台`
- `设备详情`

**为什么这是问题**

- 这会影响导航标题、埋点、设计稿 Frame 命名和研发页面命名。
- 对一个新页面来说，名字本身就是用户心智的一部分。

**建议**

- 在 UI 落稿前统一成一个正式名称。
- 如果这个页面强调“状态总控 + 操作”，我更倾向继续用 `设备控制台`。

### 12.3 不阻塞线框推进的问题

下面这些我认为不阻塞继续设计：

- 图标、颜色、插画、阴影等视觉规范
- 扫描/回连的具体秒数
- 错误码到文案的完整映射
- 耳机名称取不到时的降级显示细节

这些都可以在高保真或研发联调阶段再补。

### 12.4 UI 设计是否可以继续推进

我的结论是：

- **可以继续推进 UI 设计**
- 但建议先补齐上面 5 个点里的前 2 个，再进入高保真

如果只做下一轮线框细化，现在已经够了。  
如果马上做最终稿，我会担心“假激活”和“录音中误操作”这两个问题会在评审时被打回来。
