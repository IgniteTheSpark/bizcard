# 硬件 2.0《设备控制台 UI 设计与线框（方案 B）》Review

> review 角色：交互设计 + 硬件产品协同
> review 对象：`proposal/硬件2.0_设备控制台_UI设计与线框_方案B.md`
> review 日期：2026-04-15
> review 结论：**这版 UI 线框方向正确，可以继续推进；但在进入高保真前，建议先补齐“录音中权限规则”“模式 ON 与模式就绪的区分”“异常提示策略一致性”这 3 类问题。**

---

## 1. 总体评价

这版 UI 方案整体质量是好的，已经不只是概念草图，而是具备继续落线框和高保真输入价值的方案稿。

我认可的点：

- `方案 B` 的页面分工已经比较稳，`Search Devices` 与 `设备控制台` 的职责切分清楚。
- `3-step` 只在耳机模式内部展示，这一点非常关键，避免误导普通用户。
- `自动回连最近耳机` 和 `手动扫描兜底` 的双路径设计合理。
- `BLE 断开但经典链路仍在` 被处理为“录音继续，控制不可用”，与已拍板口径一致。
- `设备头部 / 编排区 / 辅助信息区` 的三段式结构比前一版更完整。

所以这份 UI 文档已经可以支撑下一轮设计工作。

## 2. 高优先级问题

### 2.1 缺少“录音进行中”的正常态权限规则

当前文档定义了：

- `W6` 全链路就绪态
- `§6` 录音进行中的异常态

但没有单独定义：

- **用户正在录音中时，设备控制台是怎么表现的**
- 此时哪些动作可点，哪些动作必须禁用

这会导致 `W6` 里的以下操作存在高风险歧义：

- `更换耳机`
- `断开耳机`
- `关闭耳机录音模式`

**风险**

- 这些动作都会直接影响音频链路。
- 如果录音中仍允许直接操作，用户可能会亲手把录音打断。
- 研发和测试无法确定是否需要二次确认、是否要停录、是否要弹危险提示。

**建议**

- 补一个“录音中访问设备控制台”的规则块，不一定要单独成完整页面，但必须明确：
  - 录音中哪些操作禁用
  - 哪些操作允许，但必须二次确认
  - 提示文案是否明确写出“这会停止当前录音”

### 2.2 `耳机录音模式 ON` 与“已准备完成”仍然容易混淆

在 `W3 / W4 / W5` 中，页面已经显示：

- `耳机录音模式 [ON]`

但这时实际上仍可能处于：

- `Eureka-Audio` 未连接
- 耳机桥接未完成
- 自动回连进行中

也就是说，当前 UI 仍然存在：

- 用户意图：我想进入耳机模式
- 系统状态：耳机模式已经完全可用

这两个语义被视觉上合并的问题。

**风险**

- 用户会误解为“我都开了，为什么还不能直接录”。
- 如果录音准备页与设备控制台共用同一状态源，另一个页面也可能误把这个 `ON` 当作“已就绪”。

**建议**

- 显式拆成两层状态：
  - `耳机模式已开启（准备中）`
  - `耳机模式已就绪`
- UI 上至少给出一个稳定标识，例如：
  - 开关右侧增加 `准备中`
  - 或在 Orchestrator 头部常驻 `尚未完成连接`

### 2.3 异常提示策略前后不完全一致

`§4.1` 里写的是：

- 经典链路断开导致停录时，不强推 App 内即时提醒
- 用户回到 App 后再在设备控制台和录音列表看到结果

但 `§7` 又写：

- `运行时异常弹窗/通知 → 设备控制台`

这两处口径不完全一致。

**风险**

- 设计师会不知道是否需要画“即时弹窗/通知”。
- 研发也无法确认是否要做主动跳转来源。

**建议**

- 二选一并写死：
  - 要么完全采用“非主动打扰”，删除 `运行时异常弹窗/通知` 的跳转描述
  - 要么明确哪些异常允许主动通知，哪些只在回到 App 后展示

## 3. 中优先级问题

### 3.1 `设备控制台 / 设备详情` 命名还在混用

文档里仍交替使用：

- `设备控制台`
- `设备详情`

**风险**

- 导航标题、Frame 命名、埋点名、研发页面名都会分叉。

**建议**

- 在 UI 落稿前统一成一个正式名称。
- 以当前语义看，我更倾向保留 `设备控制台`。

### 3.2 `W1` 的开关交互还没有收成唯一方案

当前写法是：

- 开关禁用
- 或可点但回弹拦截

这其实是两种完全不同的交互哲学。

**风险**

- 禁用态更稳定，但解释性弱。
- 可点回弹更易教学，但打扰更强。

**建议**

- 不要两种都留给后面。
- 建议直接选一种并写死。
- 如果你们更强调学习路径，我倾向“可点后回弹 + 轻提示”；如果更强调页面简洁，我倾向“禁用 + 明确说明文案”。

### 3.3 `W6` 的录音入口位置仍是双口径

当前写法是：

- `[开始录音]`（若此页承载录音入口）
- 或提示去 `录音准备页`

**风险**

- 这会影响 W6 的主 CTA 设计、页面层级、导航走向。

**建议**

- 在线框阶段就先拍一个版本：
  - 要么设备控制台只负责编排，不直接开录
  - 要么设备控制台允许直接开录

如果还没最终定，也建议在文档里明确标为 `Open Decision`，不要写成并列可选态。

## 4. 不阻塞推进的问题

下面这些我认为不阻塞继续做 UI：

- 图标体系、颜色语义、插画风格
- 自动回连和扫描超时的精确秒数
- 错误码到用户文案的完整映射
- 取不到耳机名称时的降级展示

这些都可以在高保真或研发联调阶段再补。

---

## 5. 是否可继续推进

我的结论是：

- **可以继续推进 UI 设计**
- **可以继续细化线框和页面逻辑**
- **暂时不建议直接冻结最终高保真**

更准确地说，当前最适合做的是：

1. 固化页面结构和主流程
2. 补齐录音中权限规则
3. 统一异常提示策略
4. 再进入视觉定稿

---

## 6. 建议的下一步

建议按下面顺序继续：

1. 在 UI 文档中新增一个“录音中控制规则”小节。
2. 明确 `耳机模式 ON` 与 `耳机模式已就绪` 的视觉区别。
3. 统一“是否主动提醒异常”的策略。
4. 拍板 `设备控制台` 是否直接承载录音入口。

如果这 4 步补完，这份 UI 方案就会非常稳。

## 7. 第二轮 UI Review（针对最新修订版）

### 7.1 先说结论

这轮更新是明显进步的，我认为这份 UI 文档已经从“可讨论线框”进入到了“可继续细化并准备高保真”的阶段。

相比上一版，以下问题已经明显收住：

- `设备控制台 / 设备详情` 的命名已经统一到 `设备控制台`。
- `耳机模式 ON = 已可录` 的假激活问题，已经通过 `Tab + 徽标 + 引导链` 的结构被拆开了。
- `录音进行中访问设备控制台` 的控制规则已经补进文档。
- `异常提示策略` 已基本统一为“不主动打扰，回到 App 后再在设备控制台呈现”。
- `W4 取消后去哪里` 也已经明确落到 `W5`。

所以总体判断是：

- **这版可以继续推进**
- **已经可以进入更细的 UI 设计和交互定稿**
- 但仍有少量“最后一公里”问题，最好在高保真前收掉

### 7.2 当前仍建议处理的 Findings

#### Finding 1：`W2` 的 `耳机模式 · 准备中` 容易把“未配置”和“进行中”混成一类

**严重级别：中**

文档前面 `§2.2` 已经给了比较好的聚合状态定义：

- 从未进入过该 Tab：可无徽标或 `未配置`
- 引导链未完成：`准备中`
- 全链路完成：`已就绪`

但在 `W2` 这个“BLE 已连，设备默认态”的线框里，Tab 被直接写成：

- `耳机模式 · 准备中`

这会让用户在还没真正进入耳机模式之前，就感觉系统已经开始一项未完成任务。

**为什么这是问题**

- 会把“用户从未配置过耳机模式”和“用户已经开始但没完成”这两种状态混在一起。
- 容易让页面产生不必要的催促感，像有个永远没完成的待办。

**建议**

- `W2` 默认态优先使用：
  - 无徽标
  - 或 `未配置`
- 只有当用户真的进入过耳机模式流程、且链路未走完时，再使用 `准备中`

#### Finding 2：`W6` 的主 CTA 仍然是双口径，没有完全拍板

**严重级别：中**

当前 `W6` 里仍是：

- `[去录音准备页]`
- 或 `[开始录音]（按导航定）`

这说明主 CTA 还没完全收敛。

**为什么这是问题**

- 主 CTA 决定页面重心。
- 如果设备控制台自己就能开录，那它是“编排 + 执行页”。
- 如果只能跳录音准备页，那它是“配置页”。
- 这是两种不同的页面心智。

**建议**

- 在高保真前最好拍板一个版本。
- 如果短期还不能定，建议明确标注为 `Open Decision`，不要继续放成并列表达。

#### Finding 3：`W7` 与 `§6.1` 对 BLE 断开时 Tab 徽标语义还不完全一致

**严重级别：中**

`§6.1` 中写的是：

- BLE 断开但经典链路仍在时，Tab 徽标可显示 `准备中 / 需修复`

但 `W7` 里写的是：

- `耳机模式 · 准备中 / 已就绪`

这里会有一点语义打架：

- 如果 BLE 已断，虽然录音可能还在继续，但从“可操作性”看，这已经不是一个纯粹的 `已就绪` 态了。

**为什么这是问题**

- 会影响状态颜色、角标含义和用户对当前系统健康度的理解。
- 设计师可能会把“仍可录”理解成“完全正常”，从而弱化异常提示。

**建议**

- 明确一条聚合原则：
  - **只要影响控制能力，就不再展示为纯 `已就绪`**
- 我更建议 BLE 断开但经典链路仍在时，统一显示为：
  - `需修复`
  - 或 `控制受限`

这样比 `已就绪` 更准确。

### 7.3 不再阻塞推进的问题

我认为下面这些点已经不再需要作为主要 review 阻塞项：

- 是否继续使用 `设备详情` 这个名字
- 是否区分“模式意图”和“模式已可录”
- 是否需要补录音中权限规则
- 是否主动弹异常通知

这些在当前版本里都已经基本收敛。

### 7.4 第二轮结论

如果只从 UI 文档质量判断，我现在的结论比上一轮更积极：

- **这版文档已经足够支持继续做高保真前的交互定稿**
- **剩余问题主要是状态文案与聚合语义的精修，不再是结构级问题**

也就是说，当前风险已经从“架构会不会返工”降到了“状态词和 CTA 是否够准”。

## 8. 第三轮 UI Review（基于“BLE 断开 = 硬件管理页只读”新口径）

### 8.1 结论

这一轮口径更新后，UI 逻辑反而更干净了。

核心新前提是：

- **BLE 连接状态 = 当前 App 与卡片的控制连接状态**
- 因此一旦 **BLE 断开**，**整个硬件管理页都应进入只读态**
- 不再保留“BLE 断开后 Toggle 还能关闭”等特殊交互

基于这个前提，我认为这版 UI 方案又稳定了一步，原因是它消除了之前一个最容易引发争议的灰区：  
**控制链路都断了，App 到底还能不能继续控制卡片。**

现在答案已经统一成：

- **不能**

这会让设计、研发、测试三方都更容易对齐。

### 8.2 这轮我认可的改动

- `方案B` 已经从“Tab”切换为“单页 + Toggle + 展开引导链”的结构，符合真实依赖关系。
- `BLE 断开` 不再被处理成“局部不可操作”，而是升级成“硬件管理页只读”，这比“有些按钮能点、有些不能点”更一致。
- `W7` 的语义现在更像一个真正的异常只读页，而不是一个半可操作的受限态。
- Header 从单纯写 `BLE：未连接`，转向更接近用户理解的 `卡片连接：未连接`，这个表达更好。

### 8.3 当前剩余的 review 点

#### Finding 1：`W2` 默认态是否显示“未配置 / 无状态”仍建议再拍一下

**严重级别：中**

你们已经不再使用 Tab，也不再强调“准备中”这个聚合态了，这是好事。  
但新的单页结构下，仍建议在默认态明确一点：

- 用户第一次进入页面
- BLE 已连
- 耳机模式 Toggle 还没打开

此时耳机模式区到底是：

- 一个纯 `OFF` 开关 + 说明
- 还是带一个 `未配置` 小提示

这虽然不是结构问题，但会影响首次进入的气质。

**建议**

- 如果想要页面更轻，就只保留 `OFF + 说明`
- 如果想强调这是一个可配置能力，可以加 `未开启` 或 `未配置` 的弱标签

不要再回到“准备中”。

#### Finding 2：`ready / not ready` 的结果态可以成立，但过程反馈仍需保留

**严重级别：中**

如果你们当前已经决定：

- 耳机模式只分 `ready / not ready`

我认为这是可接受的。  
但仍要提醒一点：

- **结果态可以简化**
- **过程态不能消失**

也就是在以下场景中，仍要保留过程反馈：

- 正在连接 `Eureka-Audio`
- 正在自动回连耳机
- 正在扫描耳机

否则用户只能看到 `not ready`，却不知道系统是否在工作。

**建议**

- 文档里可以继续不用“准备中”做模式总状态
- 但在线框和交互里必须保留：
  - loading
  - connecting
  - scanning
  - retrying

这些过程反馈

#### Finding 3：`W6` 主 CTA 仍值得尽快拍板

**严重级别：中**

这个点和上一轮一样，仍建议尽快定：

- `设备控制台` 是不是直接承载 `[开始录音]`
- 还是只跳去 `录音准备页`

这个决定会影响：

- W6 主按钮位置
- 页面主重心
- 用户是否把这个页面理解成“配置页”还是“执行页”

我仍建议在高保真前拍板，不要长期保持双口径。

### 8.4 第三轮结论

基于目前这套新口径，我的判断是：

- **这版 UI 方案已经具备继续向高保真推进的条件**
- 剩余问题已经主要是：
  - 默认态文案
  - 过程反馈细节
  - 主 CTA 定位

它们都属于交互精修，不再属于产品逻辑不稳。

## 9. 第四轮 UI Review（基于“Toggle 默认 OFF / 不做准备中聚合态 / 无录音 CTA”）

### 9.1 结论

这一轮改完之后，UI 方案已经更接近最终可执行版本了。

我认为你们这次新收敛的 3 个决定是对的：

1. **Toggle 默认就是 OFF**
2. **耳机模式不再定义“准备中”聚合态**
3. **整个硬件管理页不承载开始录音 CTA**

这 3 个决定把页面心智拉得更干净：

- 页面职责 = 连接与配置
- 不是录音执行页
- 耳机模式状态 = `ready / not ready`
- 连接过程只在具体连接项里体现

这是比前几版更稳定的交互模型。

### 9.2 这轮我认可的点

- `方案B` 已经重写成单页纵向结构，主路径明确。
- `BLE` 断开后的逻辑现在非常统一：硬件管理页只读。
- `卡片 ↔ 手机` / `卡片 ↔ 耳机` 两条连接项已经成为核心 UI 单元，这个抽象是对的。
- 已明确要求耳机连接成功后展示耳机名称，这对用户信心很重要。
- 已删除硬件管理页的录音 CTA，页面职责更纯了。

### 9.3 当前剩余建议

#### Finding 1：连接项状态建议中英统一，不要长期保留裸英文

**严重级别：低**

当前文档里连接项直接写了：

- `disconnected`
- `connecting`
- `connected`

这个作为设计内部稿没问题，但如果后面要继续给更广泛的同学看，建议尽快统一成：

- 中文
- 或中英对照

例如：

- `未连接（disconnected）`
- `连接中（connecting）`
- `已连接（connected）`

这样对产品、研发、测试都更友好。

#### Finding 2：`ready / not ready` 的落点文案最好再精简

**严重级别：低**

现在逻辑上已经很清楚了，但在最终视觉稿里，我不建议大面积直接展示英文 `ready / not ready`。

更建议收敛成更自然的用户文案，例如：

- `耳机模式已就绪`
- `耳机模式未就绪`

或者甚至不单独写模式态，只通过两条连接项结果和耳机名称来表达。

#### Finding 3：`W6` 的主交互现在只剩“更换耳机”，这是合理的，但建议再补一个“关闭耳机模式”的显式位置说明

**严重级别：低**

当前逻辑里 Toggle 本身仍然承担总开关角色，所以理论上“关闭耳机模式”已经成立。  
但在线框层最好再补一句：

- W6 下关闭 Toggle 后，页面会如何回到 OFF 态

这不是大问题，但补上会更完整。

### 9.4 第四轮结论

如果只看 UI 文档本身，我现在的结论已经可以进一步收敛为：

- **这版可以继续进入高保真前的最终交互定稿**
- **剩余问题已经基本都是文案与细节表达**
- **不再存在结构级阻塞项**

也就是说，这份 UI 方案现在已经从“需要大量 debate”进入到“可以收边、准备落视觉”的阶段了。

## 10. 第五轮 UI Review（纠偏：BLE 断开不应是只读态，而应回到搜寻卡片逻辑）

### 10.1 纠偏结论

这一轮需要明确纠正我前几轮里的一个判断：

- **`BLE 断开 != 硬件管理页只读态`**

更准确的产品口径应该是：

- `BLE` 连接状态，本质上就是 **当前 App 与卡片是否连接**
- 因此一旦 `BLE` 断开，当前页面的主任务就重新变成：
  - **搜寻卡片**
  - **连接卡片**

也就是说，页面不应该表现成“停在原地但所有按钮变灰”，而应该表现成：

- 回到 `连接卡片` 主逻辑
- 耳机模式区降级为不可操作
- 如果需要，可以弱提示耳机相关链路可能仍在运行，但这不应成为当前主交互

这个纠偏非常重要，因为它更符合你们当前 1.0 的用户心智，也更符合你给的 `hardware-page.png` 参考逻辑。

### 10.2 这轮我认可的新方向

- `设备控制台` 仍然成立，不需要退回到单独的 `Search Devices` 页面。
- 但在 `BLE` 断开时，控制台内部的主视图应回到“连接卡片”逻辑，而不是只读异常页。
- `耳机模式 Toggle` 继续默认 `OFF`，且只有 `BLE` 已连接时才能操作，这个门禁很清楚。
- `ready / not ready` 继续成立，且不需要“准备中”聚合态。

### 10.3 这次更新后，UI 方案更合理的状态心智

我建议你们内部统一按下面这组心智理解：

1. **BLE 未连接**
- 当前 App 未连接卡片
- 页面主任务：连接卡片
- 耳机模式区：禁用

2. **BLE 已连接，Toggle OFF**
- 当前 App 已连接卡片
- 页面主任务：决定是否开启耳机模式
- 耳机模式区：可操作但未展开

3. **BLE 已连接，Toggle ON**
- 当前 App 已连接卡片
- 页面主任务：完成耳机模式两条经典链路
- 耳机模式结果：`ready / not ready`

4. **BLE 中途断开**
- 页面主任务重新切回：连接卡片
- 耳机模式区退回禁用
- 若能拿到旧耳机状态，只做弱提示，不抢主流程

### 10.4 这轮 review 后剩余建议

#### Finding 1：`W7` 最好再明确“是否复用 Search Devices 视觉组件”

**严重级别：低**

现在逻辑上已经对了，但还可以再明确一点：

- `BLE` 断开后，设备控制台内部是
  - 完全复用 `Search Devices` 的扫描卡片样式
  - 还是仅复用其交互逻辑，但保留设备控制台视觉框架

这属于视觉层问题，不是逻辑阻塞，但提前说清会减少设计反复。

#### Finding 2：`耳机模式区` 在 BLE 断开时的“弱提示”文案可以再压缩

**严重级别：低**

因为此时页面主任务已经回到“连接卡片”，耳机模式区最好不要写太多背景说明。

我建议控制在一句话以内，例如：

- `请先连接卡片后再管理耳机模式`

比长篇解释“耳机链路可能仍在运行”更不容易分散注意力。

### 10.5 第五轮结论

基于这次纠偏，我的最新判断是：

- **这版 UI 方案比“整页只读”的版本更符合现有产品心智**
- **BLE 断开时回到搜寻卡片逻辑，是正确方向**
- **当前方案仍然可以继续向高保真推进**

现在剩余问题已经完全不是结构风险，而只是：

- 视觉是否复用现有搜索卡片样式
- 文案是否再压缩一点

这说明方案已经相当稳了。
