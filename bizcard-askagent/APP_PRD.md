## 一、概述与目标

### 1.1 产品背景

在 BizCard App 内提供 **Ask Agent** 能力：用户通过自然语言向 agent 提问，agent 基于用户的 contacts、meetings、reminders 及全局 memory 作答，并在对话中呈现**结构化内容**（文本 + 可跳转的资产卡片），形成 AI 原生的问询与呈现体验。

### 1.2 目标用户

- 使用 BizCard 管理联系人、会议、提醒的用户。
- 需要快速查询（如「Kevin 的联系方式」「明天待办」）、深度探讨（如「这场会按话题拆一下」）、或对已有数据进行增删改的用户。

### 1.3 核心目标

- **双入口**：General（全局）与 Context-based（某 meeting/contact/reminder 详情内）；**Chat history 按入口分范围**：从 context 进入只展示该 context 的 threads，从 global 进入展示当次 app 进程内的 threads（见下）。
- **清晰呈现**：agent 回复按「查 / 深入沟通 / 增改」三类场景，用 **text（markdown）+ 资产卡片**呈现；卡片可点击跳转 App 内详情。
- **会话与记忆**：每条对话为一个 **thread**，各有独立上下文；所有 thread 共享**全局 memory**。

```

+----------------------------------------------+
|  [BizCard]  [Calendar]  ( Ask Agent )  [Cont] [Me]  |

+----------------------------------------------+
```

- **Ask Agent**：中间位置，视觉上突出（如更大图标、不同形状），点击后**直接进入新对话**（见下），不先进入列表。

### 2.2 点击 Ask Agent 后：主输入框为页面核心 + 汉堡按钮侧边 Chat history

### 2.4 Chat history 的展示范围：Global 当次进程 vs Context 按实体

- **从 Global 入口进入**：侧边 **chat history 只展示当次 app 进程内**产生的 global threads。每次进入 Ask（进程仍在）都展示当次的这份列表；进程重启后是否持久化可按产品后续约定（v1 可仅当次进程）。
- **从 Context-based 入口进入**：侧边 **chat history 只展示该 context 下的 threads**。例如从某场会议详情进入，侧边只显示与这场会相关的对话；从另一会议进入则看到另一份列表。即：**context 入口 → 该 context 的 chat history**；**global 入口 → 当次进程的 chat history**。

## 三、功能需求

### 3.1 双入口与 Chat history 范围

| 入口 | 位置 | 进入后默认 | Chat history 侧边展示 |
|------|------|------------|------------------------|
| **General** | 底部中间突出的 Ask Agent | 直接**新对话**（主输入框 + general hint） | **全局共享**的历史会话列表 |
| **Context-based** | 某 meeting/contact/reminder 详情页内底部悬浮操作栏 | 新对话或该 context 的已有 thread（带 context hint） | **全局共享**的历史会话列表 |

### 3.2 Thread 与 Chat history

- **一个 thread** = 一条对话 = 后端的一个 session_key（如 `ask_agent:global:{user_id}:{thread_id}` 或 `ask_agent:meeting:{meeting_id}:{thread_id}`）。
- **Chat history 侧边**：通过顶部汉堡按钮 [≡] 打开；**所有入口统一展示全局共享的历史会话列表**。
- **每个 thread** 仅有本 thread 的消息上下文；**所有 thread 共享全局 memory**（nanobot 侧如 MEMORY.md）。

### 3.3 三类场景与呈现

| 场景 | 用户意图 | 呈现要点 | Block 组合 |
|------|----------|----------|------------|
| **查** | 基础信息查询 | 以文本（text）回答核心诉求 + 附带相关 contact/meeting/reminder 的紧凑卡片供跳转 | text（回答）+ contact_card / meeting_card / reminder_card（附带跳转入口） |
| **深入沟通** | 追问、深度分析 | **结合用户的全局 Memory（偏好/职业）与外部信息（Web Search），生成高度定制化的 Markdown 报告、表格或数据结构**；必要时附带提及的其他实体卡片 | 以 text（MD 报表/表格）为主，按需加卡片 |
| **增/删/改** | 对 profile 或已有数据修改 | 修改确认（text）+ 修改后的实体卡片 | text（确认）+ contact_card / reminder_card / meeting_card |

**原子能力边界声明**：目前底层 Agent (Nanobot) 支持的原子能力覆盖如下（前端展示及意图理解均以此为基准）：
- **Contacts**: 查、增、改、删
- **Reminders**: 查、增、改、删
- **Meetings**: 查、改、删（注：会议的“增”主要由 App 硬件录音/扫描流程触发，Agent 本身不提供凭空捏造会议的能力）

**上下文防冗余 (Context-Aware Card Suppression)**：这是一个重要的体验规范。如果用户是从具体的某场会议/联系人详情页进入的 Ask Agent，Agent 在回复时**不应再附带当前会议/联系人的卡片**（因为用户已经在该实体的背景下了，再出卡片纯属视觉冗余），而只应附带**提及的其他关联实体**的卡片。

## 3.4 响应契约：Text + Cards (Attachments) 结构

基于“先展示文本，底部附带卡片提供跳转”的呈现原则，前端与后端的契约应当明确区分内容（文本）与附件（卡片）。
**此 JSON 结构是由后端 BFF 层的 Reply Builder 组装好后吐给前端的**，而非直接来自于 Nanobot。

```json
{
  "text": "上次和 Kevin 的会议主要讨论了产品设计，Kevin 的联系方式是 138-xxxx-xxxx。\n\n另外提醒你明天有个跟进待办。",
  "cards": [
    {
      "type": "meeting_card",
      "payload": {
        "meeting_id": "m1",
        "title": "Product Design Sync",
        "meeting_at": "2026-01-22T11:30:00Z"
      }
    },
    {
      "type": "contact_card",
      "payload": {
        "id": "c1",
        "name": "Kevin Chen",
        "title": "PM",
        "company": "Acme Corp"
      }
    }
  ]
}
```

- **text**: 纯文本 / Markdown，包含 Agent 对于问题的**核心解答**。前端将其交给 Markdown 渲染器直接渲染。（Agent 返回的表格、Mermaid 图表等富文本形式也会直接包在这个字段中）
- **cards (或 attachments)**: 当需要在回答中提供相关实体的跳转/操作入口时提供。前端将其作为附件容器展示在文本下方（例如横向滑动列表 Carousel）。这样避免了在文本流中强行插入 UI 块，保持排版的干净。

### 3.5 结构化输出的生成机制 (Agent 侧的直接产出)

为了让前端能够按照上述 `Text + Cards` 的契约进行排版，底层的 Agent (Nanobot) 需要在回复时主动将资产卡片融入到结构化输出中。虽然本 PRD 不涉及具体 Skill 的代码实现，但在逻辑机制上建议采用 **Prompt / Skill 级约束**：

- **机制说明**：在面向 Ask Agent 渠道的相关资产操作 Skill 中，明确向大模型下发**输出约束**。Agent 在调用 Tool 拿到详情后，直接负责返回包含核心解答与相关实体卡片（`cards` 数组）的完整 JSON 结构，并交由前端直接解析渲染。
- **Agent 的核心行为转变**：
  1. **主动挂载附件 (按需出卡)**：Agent 在调用查询/修改原子能力后，如果判断该实体需要给用户提供跳转或操作入口，则必须在最终的 JSON 响应的 `cards` 数组中，组装出该实体的关键信息（如 `entity_type` 和 `entity_id` 等，按前端要求）。
  2. **文本表达更克制 (Text 提炼)**：受制于该输出规范，Agent 在生成 `text` 时会自动变得更加克制，**只提炼回答用户问题的核心要点**，而不会把实体的所有冗余字段罗列在文本中。
  3. **上下文防冗余 (Context-Aware Card Suppression)**：如果用户当前处于特定的详情页（如某个会议的上下文），对应的 Skill 会约束大模型，不要再次在卡片附件中输出该会议的实体对象，避免视图冗余。

这种机制将“需要挂载哪些卡片”和“哪些场景不应该挂载卡片”的复杂业务语义决策权交给了特定的 Skill（大模型），同时去除了后端的转换层，使得前后端链路更加轻量，前端只需做纯粹的排版渲染。

---

## 四、阶段拆分

需求按**两个阶段**落地：第一阶段完成 Ask Agent 页面与对话展示、多入口配置；第二阶段在能力上扩展（如发邮件），并依赖前置的邮箱等配置。

### 第一阶段：页面 + 对话展示 + 多入口配置

**目标**：用户能在各入口进入 Ask Agent，完成对话，并看到正确的展示形态。

| 范围 | 说明 |
|------|------|
| **Ask Agent 页面** | 实现 Ask Agent 主界面：主输入框为核心、汉堡按钮侧边 chat history、初始 hint（General / Context-based）、新对话与 thread 切换。 |
| **对话展示** | 默认以 **text**（markdown）渲染 agent 回复；第一阶段额外支持 **三种卡片**：**meeting_card**、**contact_card**、**reminder_card**，用于在对话中展示会议、联系人、提醒并支持「查看详情」跳转。其他 block 类型（如 quick_reply、reminder_list）可后续再加。 |
| **多入口配置** | 在三个入口接入并配置 Ask Agent：**Global 入口**（底部中间突出的 Ask Agent）、**Meeting 详情页**（悬浮栏）、**Contact 详情页**（悬浮栏）。Reminder 详情页入口可按资源排期放在一阶段或二阶段。 |

**交付**：用户从上述任一入口进入 → 可与 agent 多轮对话 → 回复中以 text + meeting/contact/reminder 卡片展示，卡片可跳转 App 内详情；侧边 chat history 范围统一为全局共享列表。

---

### 第二阶段：扩展能力（如发邮件）+ 前置配置

**目标**：在对话内支持「发邮件」等需要**对外通道与用户授权**的能力；在此之前需完成用户邮箱等配置的前置流程。

| 范围 | 说明 |
|------|------|
| **扩展 Ask Agent 能力** | 例如在对话中支持「给 XXX 发邮件」「给 Kevin 发一封跟进邮件」等（对应 outreach / 发信息 skill）。Agent 拟稿后需能真正从用户邮箱发出，而非仅落库或展示草稿。 |
| **前置流程：用户邮箱配置** | 发邮件依赖 **nanobot 接管用户邮箱**（见 AGENT_ARCHITECTURE 与 outreach skill）：需在 App 或设置流程中完成 **邮箱配置**（IMAP 收件 + SMTP 发件、账号密码或 OAuth）及用户**授权（consent）**，nanobot 侧 Email channel 启用后，agent 才能以用户身份代发/代收。第一阶段不要求必配邮箱；第二阶段若启用发邮件能力，则需在「使用发邮件前」引导用户完成邮箱配置与授权。 |
| **其他可选能力** | 如日历/约会（查 calendar、建 reminder、代复约）、更多 block 类型（quick_reply、reminder_list）等，按优先级排入二阶段或后续迭代。 |

**交付**：用户完成邮箱配置与授权后，可在 Ask Agent 对话中发起发邮件等操作；agent 拟稿并经由用户邮箱发出；未配置邮箱时，发邮件相关能力不展示或提示先完成配置。

---

## 五、非功能需求（主要针对第一阶段）

- **性能**：打开侧边 chat history、切换 thread、加载历史消息应在可接受时延内完成（具体指标按项目定）。
- **离线**：若需支持离线查看历史，需约定本地存储与同步策略（可选 v2）。
- **无障碍**：`text` 作为兜底/读屏时可依赖；卡片需具备可访问的「查看详情」操作。

## 六、验收标准

### 第一阶段

- [ ] 底部为 **4 个 Tab + 1 个突出的 Ask Agent**（中间）：BizCard、Calendar、Contacts、Me；Ask Agent 视觉突出，点击后直接进入新对话。
- [ ] Ask Agent 默认页以**主输入框为核心**，顶部有**汉堡按钮 [≡]**，点击后侧边滑出 **chat history**；列表范围：所有入口共享**统一的全局历史会话列表**。
- [ ] 进入 Ask Agent 时展示**初始 hint**：General 入口为 general 类 hint，Context 入口为基于该 context 的 hint（如会议：「按话题拆一下」「总结待办」等）。
- [ ] 「新对话」可新建 thread，主区展示空状态 + hint + 输入框；侧边选中某条 thread 后主区展示该 thread 的消息流 + 底部输入框。
- [ ] 对话展示：agent 回复默认按 **text**（markdown）渲染；支持 **meeting_card、contact_card、reminder_card** 三种卡片，带「查看详情」并可跳转 App 内对应详情。
- [ ] **多入口**：**Global 入口**、**Meeting 详情页**、**Contact 详情页** 均已配置 Ask Agent 入口；从 meeting/contact 详情发起「问 Agent 关于…」时，进入 Ask 且侧边仅展示该 context 的 chat history，新建 thread 出现在该侧边列表。
- [ ] 多轮对话在同一 thread 内连续，agent 能基于该 thread 上下文与全局 memory 作答。

### 第二阶段（规划）

- [ ] 对话内支持**发邮件**等扩展能力（如「给 Kevin 发一封跟进邮件」）；agent 拟稿后能经用户邮箱实际发出。
- [ ] **前置流程**：在 App 内提供**用户邮箱配置**（IMAP/SMTP 或 OAuth）及**授权（consent）**；未配置时发邮件能力不展示或引导先配置。Nanobot 侧 Email channel 与配置对接后，agent 可代发/代收。
- [ ] 其他能力（如约会、quick_reply 等）按优先级排期。
