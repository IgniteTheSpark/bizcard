# Ask Agent（问 BizCard）— Prompt 与能力设计说明

> **目标**：在 App 内嵌入「Ask Agent」能力，类似 Plaud 的 Ask Plaud、Ticnotes 的 Ask Shadow：用户向大模型提问，Agent 既能用模型自身能力回答，又能**查询用户自己的知识库**（会议纪要、闪念、联系人等）做检索增强。  
> **产品边界**：**保留 Plaud 式 Ask Agent 的全部能力**（通用问答 + 会议/录音/笔记知识库），在此之上**再叠加** BizCard 的名片、Contacts、Profile 及所有管理与关联能力；二者并存，不做二选一。  
> **结论**：**需要定制化 Prompt**，用于明确「何时查库、如何融合、如何输出」，否则模型无法稳定地同时用好「通用知识」与「用户私有知识」。

---

## 一、为什么需要定制 Prompt？

| 维度 | 说明 |
|------|------|
| **双源回答** | 若不写进 Prompt，模型可能只用自己的知识答题，不主动查用户知识库；或反过来过度依赖检索结果。需要明确：**先查用户知识库，有相关结果则优先基于检索内容回答并注明来源；无相关结果再用模型通用知识**。 |
| **检索时机与范围** | 需要约定：哪些问题必须/建议先检索（如「我最近的会议讲了什么」「Kevin 上次说了什么」），哪些可以直接用通用知识（如「埃隆·马斯克是谁」）。Prompt 里可写「若用户问题涉及【我的会议/闪念/联系人/日程】等，请基于下方提供的【检索结果】回答」。 |
| **输出格式与安全** | 可要求：回答结构、引用来源（如「根据您 1 月 22 日的会议纪要…」）、未检索到时的表述（「在您的记录中未找到相关信息，根据通用知识…」），避免幻觉。 |
| **与现有规范一致** | BizCard 已在用定制 Prompt 做 Reminder 提取（见 `AI_EXTRACT_REMINDERS_SPEC.md`），Ask Agent 的「RAG + 生成」同样适合用一套清晰的 Prompt 规范，便于迭代和换模型。 |

因此，**定制 Prompt 不是可选项，而是实现「模型能力 + 用户知识库」双源能力的必要手段**。

---

## 二、能力架构简述

```
用户提问
    ↓
意图/路由（可选）→ 若涉及「我的数据」则触发知识库检索
    ↓
检索用户知识库（会议摘要、闪念、纪要、联系人等）→ 得到 Top-K 片段
    ↓
组装 Prompt：System（角色+规则）+ 检索到的上下文 + 用户问题
    ↓
调用大模型 → 生成回答（可要求引用来源、无则用通用知识）
    ↓
返回用户（可选：模型选择器、@ 引用、语音输入等，见 Ticnotes 截图）
```

- **知识库**：用户私有数据（会议 summary/transcript、闪念、Reminders、联系人备注等），需做向量化与检索（RAG）。
- **Prompt**：在 System 或首条 Instruction 中写明「你有两类信息：1）下面提供的【用户知识库检索结果】；2）你的通用知识。请优先基于 1）回答与用户本人相关的问题；若 1）无相关内容再用 2）。」

## 三、Prompt 设计要点

### 3.1 角色与任务

- **角色**：用户的个人助理，既能回答通用问题，又能基于用户自己的会议、闪念、联系人等数据回答问题。
- **任务**：根据用户问题，若与「用户自己的记录」相关，则**优先使用提供的【知识库检索结果】**组织回答；否则使用模型自身知识。回答需简洁、有用，必要时注明来源（如「根据您 1 月 22 日的会议纪要」）。

### 3.2 何时用知识库 vs 通用知识

- **建议先查库再答**：用户问题中出现「我的会议」「最近的会议」「某联系人说了什么」「我的闪念」「我的待办/日程」等，或明确在问某个会议/某个人时，应在 Prompt 中注入**检索到的片段**，并指示模型「以下为从用户知识库检索到的内容，请基于此回答」。
- **可直接用通用知识**：纯常识、百科、当前事件等（如「埃隆·马斯克是谁」），无需查用户库；若已查库但无相关内容，Prompt 中可写「未检索到与用户记录相关的内容，请仅用你的通用知识回答」。

### 3.3 输出约束（可选但推荐）

- 若使用了检索结果，可在回答中简要注明来源（如会议日期、标题），避免用户误以为全是模型「编的」。
- 若检索结果为空且问题明显与「我的数据」相关，可要求模型回答「在您当前的记录中未找到相关内容」，再视情况补充通用建议。

### 3.4 示例 System Prompt（可直接或微调）

```text
你是用户的个人助理，内置于 BizCard App 中。你有两种信息来源：
1）【用户知识库检索结果】下面会提供从用户自己的会议纪要、闪念、联系人等中检索到的内容（可能为空）。
2）你的通用知识。

【规则】
- 当用户的问题涉及「我的会议」「我的记录」「某次会议」「某个联系人说了什么」「我的闪念/待办」等时，必须优先基于「用户知识库检索结果」回答；若检索结果为空，则明确说明「在您当前的记录中未找到相关内容」，再视情况用通用知识补充。
- 当用户的问题与用户本人数据无关（如常识、百科）时，可直接用你的通用知识回答。
- 若你使用了检索结果，请在回答中简要注明来源（如「根据您 1 月 22 日的会议纪要」）。
- 回答简洁、友好，符合对话助手语气。

【用户知识库检索结果】
{{retrieved_context}}

（若未做检索或检索为空，此处为「（无）」）

【用户问题】
{{user_question}}
```

将 `{{retrieved_context}}` 替换为 RAG 返回的 Top-K 片段拼接；`{{user_question}}` 替换为当前用户输入。若支持多轮，可把历史对话摘要或最近几轮拼进 Prompt（注意长度与成本）。

## 四、与现有 BizCard 能力的关系

| 现有能力 | 与 Ask Agent 的关系 |
|----------|----------------------|
| **会议管线**（转写、总结、Reminder 提取） | 会议 summary / summaryData 可作为知识库的重要数据源；Reminder 提取已有独立 Prompt，Ask Agent 只消费「检索结果」，不替代该管线。 |
| **闪念 / 语音备忘** | 闪念文本、清理后的 paraphrase 可进入用户知识库，供 Ask Agent 检索。 |
| **Timeline / Agent Call** | Timeline 上的会议、Agent Call 摘要等也可作为检索源；硬件「按住问：Kevin 上次说了什么」可视为 Ask Agent 的语音入口，后端同样「检索 + Prompt + 模型」生成回复后回传 E-ink。 |
| **My Agent（Voice & Knowledge）** | 当前 Agent 配置中的「Knowledge」可与 Ask Agent 共用同一套知识库与检索；Ask Agent 的 Prompt 负责「如何用」这些知识。 |

## 五、小结

- **是否需要定制 Prompt？** **需要。** 用于明确：优先使用用户知识库检索结果、何时用通用知识、如何注明来源、如何避免幻觉。
- **实现要点**：在调用大模型前，对「用户自己的数据」做检索（RAG），将检索结果与用户问题一起组装进 **System/User Prompt**，并约定输出规范。
- **与 Ticnotes Ask Shadow 的对应**：对方界面中的「知识库」入口、模型选择器（如通义千问）、以及能回答「我最近的会议讲了什么」并引用会议内容，其背后都是「检索 + 定制 Prompt + 单/多模型」的实现；BizCard 可按同一思路做 Ask Agent，并复用现有会议/闪念数据与 AI 规范风格。

## 六、Contacts 与 Profile 带来的改造点（相对 Plaud / Ticnotes）

Plaud、Ticnotes 的框架中**没有「名片」属性**：知识库主要是「我的录音/会议/笔记」，没有第一方的联系人实体（Contacts）与个人身份/名片（Profile）。BizCard 则把 **Contacts** 和 **Profile** 作为核心内容，因此 Ask Agent 的改造点集中在：**知识库范围、检索维度、问答场景、Prompt 话术** 四块。

### 6.1 知识库数据范围的改造

| 维度 | Plaud / Ticnotes 典型范围 | BizCard 改造点 |
|------|---------------------------|----------------|
| **会议/录音** | 会议转写、总结、笔记 | 同左，且增加 **与会议关联的 Contact**（参与者、手动关联的联系人），便于「按人」检索。 |
| **联系人** | 无第一方联系人实体 | **Contacts**：姓名、公司、职位、来源（交换/扫码/Agent Call）、**与该联系人关联的 Meeting/Reminder/Exchange**。 |
| **身份/名片** | 无 | **Profile**：用户自己的名片信息（姓名、Title、公司、Bio、链接等）；可被 Ask Agent 用于回答「我的名片上写的是什么」「帮我根据最近会议改一句简介」等。 |
| **交换与关系** | 无 | **Exchange / 碰一碰记录**：谁在何时保存了我的名片、我与谁换了名片、First Contact 等，仅 BizCard 有的数据。 |

即：知识库不仅要存「会议摘要、闪念」，还要把 **Contact 元数据 + 与 Contact 绑定的会议/Reminder/Exchange**、以及 **Profile** 纳入可检索内容。

### 6.2 检索维度的改造

- **按「人」检索（Contact-scoped）**  
  当用户问「Kevin 上次说了什么」「和 Alice 相关的待办有哪些」时，应先解析出实体（Kevin / Alice），再按 **Contact 维度** 检索：该联系人的会议摘要、会议中的要点、关联的 Reminders、与该人相关的闪念或 Exchange。Plaud/Ticnotes 没有 Contact 实体，只能做全文/语义检索，无法稳定地「锁定到某个人」。
- **按「我的身份/名片」检索**  
  当用户问「我名片上写的是什么」「我的简介」时，检索结果应包含 **Profile** 的当前字段；若问「根据最近几场会议帮我改一句 Slogan」，则需同时检索会议摘要 + Profile，由模型融合生成建议。
- **按「关系事件」检索**  
  如「上次和谁换了名片」「谁保存了我的名片」：依赖 **Exchange / 碰一碰** 等仅 BizCard 有的数据，检索后由 Agent 用自然语言回答。

### 6.3 问答场景的改造（差异化能力）

| 场景类型 | 典型问题示例 | 依赖的数据 | 改造点说明 |
|----------|--------------|------------|------------|
| **联系人 + 会议** | 「Kevin 上次说了什么」「我最近和 Kevin 的会议讲了什么」 | Contact + 关联 Meeting summary/transcript | 以 Contact 为主键做检索，再注入会议内容；回答可注明「根据您与 Kevin 在 1 月 22 日的会议」。 |
| **联系人 + 待办** | 「我欠 Kevin 什么」「和 Alice 相关的待办有哪些」 | Contact + 关联 Reminders | 检索该联系人关联的 Reminders，由 Agent 汇总并注明来源（会议/手动）。 |
| **联系人 + 关系** | 「我和 Kevin 是怎么认识的」「上次见 Kevin 是什么时候」 | Contact + Exchange / Meeting 时间线 | 用 Contact 的 First Contact、最近会议时间等结构化信息 + 时间线，生成简短回答。 |
| **我的名片/身份** | 「我的名片上写的是什么」「帮我根据最近会议更新一句我的简介」 | Profile + 可选会议摘要 | 检索 Profile 与（可选）近期会议，模型只读或生成建议，不替用户直接改 Profile（除非产品明确支持）。 |
| **交换与触达** | 「上次和谁换了名片」「谁保存了我的名片」 | Exchange / 碰一碰记录 | 仅 BizCard 有的数据与场景，需在知识库与检索中显式支持。 |

上述场景在 Plaud/Ticnotes 中要么无法实现（无 Contact/Profile），要么只能弱化为「全文里提到 Kevin 的段落」，无法做到「以人为中心的上下文」。

### 6.4 Prompt 与产品话术的改造

- **System Prompt 中显式加入「联系人」与「身份」**  
  在「用户知识库检索结果」的说明里，除会议/闪念外，应写明：可包含 **联系人（Contacts）及其关联的会议、待办、交换记录**，以及用户自己的 **名片/Profile**。并约定：当问题涉及**具体人名**时，优先使用「按该联系人检索到的内容」回答，并注明来源（如会议日期、联系人名）。
- **回答风格**  
  对「某人说了什么」「我欠谁什么」类问题，可要求 Agent 用「人」作为主语组织答案（如「Kevin 在 1 月 22 日的会议中提到……」），与 BizCard 的「身份 → 建联 → 当下 → 事后」叙事一致。
- **产品命名与引导**  
  若产品内将能力命名为「问 BizCard」或「Ask BizCard」，可在空状态或引导中强调：不仅能问会议和笔记，还能问 **「某联系人说了什么」「和谁相关的待办」「我的名片」** 等，突出与纯录音/笔记类产品的差异。

### 6.5 小结：改造点一览

| 改造点 | 说明 |
|--------|------|
| **知识库** | 纳入 **Contacts**（及关联的 Meeting/Reminder/Exchange）、**Profile**、Exchange/碰一碰记录，而不只是会议+闪念。 |
| **检索** | 支持 **Contact 维度**（按人检索）、**Profile 维度**（我的名片/身份）、**关系事件**（换名片、谁保存了我）。 |
| **问答场景** | 支持「某人说了什么」「我欠谁什么」「我和某人怎么认识的」「我的名片/简介」「谁换了我名片」等仅 BizCard 能答的问题。 |
| **Prompt** | System 中明确「联系人 + 身份」是知识库的一部分；涉及人名时优先用按人检索的结果并注明来源。 |

这样，Ask Agent 在「会议 + 闪念」的基础上，自然延伸出 **以人和关系为中心** 的差异化能力，与 Plaud/Ticnotes 形成清晰区隔。

## 七、追问与多轮逻辑（联系人澄清 + 按内容查会议）

在保留 Plaud 式全部能力并叠加名片/联系人后，需要约定两类**多轮或检索逻辑**：一是当用户用**非唯一称呼**指代联系人时，Agent 应**追问澄清**；二是当用户按**主题/内容**问「哪一次会议」时，应**查总结/纪要内容**再回答。

### 7.1 联系人指代不明时的追问（如「王总」→ 请用户确认大名）

**场景**：用户问「我和王总上次开会聊了什么」。  
「王总」可能是称呼而非全名，通讯录中可能有多位姓王或备注为「王总」的联系人（如王伟、王强、王磊），系统无法唯一确定是哪一个。

**逻辑**：

1. **解析与检索**：从用户问题中解析出联系人指代（如「王总」），在 Contacts 中检索可能匹配的联系人（按姓名、备注、职位等）。
2. **若匹配到唯一联系人**：按该 Contact 做会议/纪要检索，正常注入上下文并回答。
3. **若匹配到多位候选人**：**不猜测**，由 Agent **向用户追问**，待用户确认后再查库回答。
   - 追问方式可二选一或并存：
     - **请用户提供大名**：「您通讯录里有好几位王总，能告诉我您指的是哪一位吗？可以说全名，比如王伟、王强。」
     - **列出候选人让用户选**：「您指的是王伟、王强还是王磊？」（列表来自检索到的 Contact 候选。）

**实现要点**：

- 后端在「按人检索」时，若发现多个 Contact 匹配同一指代，可返回 `disambiguation_required: true` 及 `candidates: [{ id, name, ... }]`，不在此时注入会议内容。
- Prompt 中约定：当【检索结果】中注明「联系人指代对应多位候选人，请先向用户追问确认」时，**只输出追问句**，不基于「某一位」生成回答，避免张冠李戴。

**示例 System 补充（可并入主 Prompt）**：

```text
【联系人澄清】
当用户用非全名指代某人（如「王总」「李经理」）且检索结果显示该指代对应多位联系人时，你必须先向用户追问确认（请用户说全名，或列出候选人让用户选），在用户明确指定是哪一位之前，不要基于「猜测的某一人」回答会议内容。
```

### 7.2 按内容/主题查会议（如「上次聊的 RWA 是哪一次会议」）

**场景**：用户问「上次聊的 RWA 是哪一次会议」「我们哪次会讨论了合规问题」。  
这里**不是按人查**，而是按**讨论的主题/关键词**在会议总结或纪要内容里查，找出「哪一次（或哪几次）会议」提到了该主题。

**逻辑**：

1. **按内容检索**：用用户问题中的主题/关键词（如「RWA」「合规」）对**会议总结（summary）、纪要（summaryData）、可选 transcript 片段**做语义或关键词检索，得到包含该主题的会议列表及对应片段。
2. **组装上下文**：将检索到的「会议标题 + 日期 + 相关摘要或片段」注入 Prompt。
3. **回答**：Agent 基于检索结果回答「是您 x 月 x 日的《xxx》会议」或「在 1 月 22 日的产品会与 1 月 25 日的合规会都有提到 RWA……」并可选附一句摘要。

**与「按人查」的区别**：

| 用户问法 | 检索维度 | 数据来源 |
|----------|----------|----------|
| 「我和王总上次开会聊了什么」 | **按人**（先解析「王总」→ Contact，再查该人的会议） | Contact + 关联 Meeting summary/transcript |
| 「上次聊的 RWA 是哪一次会议」 | **按内容/主题**（查总结里含「RWA」的会议） | Meeting summary / summaryData / transcript 的语义或关键词检索 |

**实现要点**：

- 知识库需支持对「会议总结 + 纪要」的**内容检索**（向量或关键词），返回命中会议及其片段，而不只是按 Contact ID 查。
- Prompt 中约定：当用户问「哪一次会议」「哪次会讨论了 xxx」时，优先使用**按内容检索到的会议列表与片段**组织答案，并注明会议日期与标题。

### 7.3 小结

| 逻辑 | 触发条件 | 行为 |
|------|----------|------|
| **联系人追问** | 问题中有人物指代（如王总）且 Contact 检索得到**多位候选人** | Agent 只做追问（请用户说全名或从候选中选），不猜一人作答。 |
| **按内容查会议** | 用户问「哪一次会议聊了 xxx」「上次聊的 RWA 是哪次会」等 | 对会议**总结/纪要内容**做检索，返回对应会议（日期+标题），再组织回答。 |

二者可与「按人查会议」「通用知识」等能力同时存在，共同构成 Ask Agent 的完整行为。

## 八、通过 Agent 修改/更新用户 Profile

除「问我的名片上写的是什么」「根据会议帮我改一句简介建议」等**只读**能力外，若希望用户**通过对话直接修改/更新 Profile**（如「把职位改成产品经理」「更新我的简介为 xxx」），需要增加**写能力**：通过 **Tool** 让 Agent 发起修改，后端或客户端再决定是直接落库还是先确认再落库。

### 8.1 两种实现方式（产品二选一或组合）

| 方式 | 说明 | 适用场景 |
|------|------|----------|
| **A. 仅建议（无写 Tool）** | Agent 只输出「建议修改」的文本或结构化内容（如「建议将职位改为：产品经理」），由 App 展示确认 UI，用户点击确认后再调后端接口真正写入。 | 希望所有修改都经用户**显式确认**再落库，Agent 不直接动数据。 |
| **B. Tool 发起修改（推荐）** | 提供 **`update_my_profile`**（或类似）Tool；Agent 在识别到用户要求改 Profile 时调用该 Tool，传入要改的字段与新值。后端可**直接更新**，或返回「待确认」+ 变更摘要，由客户端弹确认后再调「确认写入」接口。 | 体验更顺滑；是否二次确认由产品/后端策略决定。 |

推荐 **B**：用 Tool 统一「意图→执行」路径，是否二次确认放在后端或客户端策略里（见 8.3）。

### 8.2 Tool 设计：`update_my_profile`

**职责**：根据用户自然语言表达的修改意图，更新当前用户的名片/Profile 的指定字段。

**建议的输入 schema（示例）**：

- `changes`：对象，仅包含**用户明确要求修改**的字段及其新值，例如：
  - `name`、`title`、`company`、`bio`、`avatar_url`（若支持）、`links`（或拆成 `linkedin`、`email` 等，依你现有 Profile 模型而定）
- 不传的字段表示「不修改」；Agent 不应臆造用户没说的内容。

**建议的返回（后端 → Agent）**：

- `success: true`：已写入；可选带 `updated_fields: ["title", "bio"]`。
- `success: false, reason: "..."`：校验失败或权限问题。
- 若采用「先确认再写入」：可返回 `pending_confirmation: true, summary: "将职位改为「产品经理」、简介改为「…」"`，由客户端展示确认 UI，用户确认后再调后端「确认 Profile 更新」接口；Agent 可根据 `pending_confirmation` 回复用户「已按你的要求准备好修改，请在 App 中确认后生效」。

**安全与校验**：

- 仅允许更新**当前登录用户**的 Profile；后端用 session/ token 识别用户，不信任 Tool 入参里的「用户 ID」。
- 后端对**可写字段白名单**、长度、格式（如 URL、邮箱）做校验，避免注入或非法数据。

### 8.3 System Prompt 补充（Profile 修改规则）

在 System 中增加一段，例如：

```text
【Profile 修改】
- 当用户明确要求修改自己的名片/Profile（如「把职位改成产品经理」「更新我的简介为……」「公司名改成 xxx」）时，你必须调用 update_my_profile 工具，且仅传入用户**明确说出**的字段与新值；不要编造或推断用户未提及的字段。
- 若用户只说「帮我改一下简介」但未给出新内容，应追问「您希望简介改成什么内容？」而不是自己编一段。
- 调用后根据工具返回结果回复用户：已更新则简要确认；若返回待确认，则提示用户在 App 中确认后生效。
```

这样 Agent 只在「用户明确表达修改意图且给出新值」时调用 Tool，避免误改或幻觉。

### 8.4 可写字段与现有 Profile 的对应

与现有 BizCard Profile 能力对齐（见 BIZCARD_POSITIONING、各 PRD 中的 Profile/名片编辑）：通常包括姓名、职位、公司、简介（Bio）、头像 URL、社交/业务链接等。Tool 的 `changes` 的 key 与后端 Profile 模型的字段一致即可；若后端只开放部分字段给 Agent 修改，则在 Tool 描述中写明，并在后端做白名单校验。

### 8.5 小结

| 要点 | 说明 |
|------|------|
| **实现方式** | 推荐用 **Tool `update_my_profile`** 让 Agent 发起修改；是否「先确认再写入」由后端或客户端策略决定。 |
| **Prompt** | System 中约定：仅当用户**明确说出**要改的字段与新值时才调用；否则追问，不臆造。 |
| **安全** | 后端只允许改当前用户、白名单字段，并做长度/格式校验。 |
| **与只读并存** | 「我的名片上写的是什么」「根据会议给一句简介建议」等只读能力不变；修改能力作为新增 Tool + 一段 System 规则即可。 |

## 九、迭代节奏：会议总结 + 联系人关联先行，再扩展

整体能力可以**分阶段实现**，但 **BizCard 的差异化在名片/联系人**，因此 **Phase 1 必须包含联系人关联**（按人查会议、联系人澄清），不能只做「纯会议总结」查询。首版即体现「会议 + 人」的闭环，再逐步扩展待办、Timeline、Profile 等。

### 9.1 Phase 1（当前优先）：Ask Agent — 会议总结 + 联系人关联

**范围**：

- 上线 **Ask Agent** 入口（一个输入框 + 对话/结果展示）。
- **数据与能力**：**会议总结/纪要** + **联系人及与会议的关联**。
  - **会议相关**：如「我最近的会议讲了什么」「某次会议的要点」「上次聊的 RWA 是哪一次会议」（按内容/主题查总结）。
  - **联系人关联（必须）**：如「我和王总上次开会聊了什么」「Kevin 上次说了什么」——即**按人查会议**；当用户用非唯一称呼（如「王总」）时，**联系人澄清**（追问大名或列出候选人），再基于唯一联系人查其关联会议并回答。
- **实现要点**：
  - **知识库/检索**：接入**会议 summary、summaryData**（及可选 transcript 片段），以及 **Contacts** 及**会议与联系人的关联**（Meeting ↔ Contact）；支持按 Contact 维度检索其关联会议。
  - **Prompt**：System 中约定「基于会议总结/纪要回答」+ **联系人澄清规则**（指代对应多人时只追问、不猜）+ **按人查会议**时优先用该联系人关联的会议内容回答并注明来源。
  - **返回形式**：Phase 1 可先采用**纯文本/对话气泡**回答，不必先做多 `response_type` 的结构化返回与多组件渲染。

**不纳入 Phase 1（后续再做）**：

- 待办/Reminders、Timeline、通话记录、Profile 的查询与修改。
- 结构化返回（如 `todo_list` / `timeline`）与前端按类型渲染。

### 9.2 后续阶段（示意，按需排期）

| 阶段 | 可扩展方向 |
|------|------------|
| **Phase 2** | 按**内容/主题**查会议的完善；或接入**闪念**检索，支持「我的闪念/笔记」类问题。 |
| **Phase 3** | 待办、Timeline、通话记录等查询；可选结构化返回（response_type）与前端多形态渲染。 |
| **Phase 4** | Profile 只读与修改（Tool `update_my_profile`）、名片/身份类问答。 |

以上顺序可按产品优先级调整；核心是 **Phase 1 = 会议总结 + 联系人关联**，首版就体现 BizCard 的名片/联系人差异化。

## 十、用户问题类型梳理

Ask Agent 面对的典型问题可按「检索主键」与「返回目标」分为以下几类，便于设计检索、Tools 与 Prompt。

| 类型 | 描述 | 检索主键 → 返回目标 | 典型问法示例 |
|------|------|----------------------|--------------|
| **1. 按话题/内容 → 查会议** | 用户记得聊过某话题，想找**是哪次会**。 | 主题/关键词（如 web3、RWA）→ 命中该内容的**会议**（标题、日期、摘要）。 | 「上次聊到 web3 和 RWA 的应用是哪次会议来着？」 |
| **2. 按话题/内容 → 查人** | 用户记得聊过某话题，想找**是和谁聊的**。 | 主题/关键词 → 命中该内容的**会议**，再取会议**关联的联系人**（参会人）。 | 「上次和谁聊到 web3 和 RWA 来着？」 |
| **3. 按人（明确）→ 查会议** | 用户指定**具体的人**（全名或唯一匹配），查与之相关的会议及内容。 | 联系人（如张春丽）→ 该联系人**关联的会议**及总结/要点。 | 「和张春丽上次开会聊什么来着？」 |
| **4. 按人（模糊昵称）→ 查会议** | 用户用**昵称/称呼**（王总、李总），可能对应多人，需先**澄清**再按人查会议。 | 先按称呼解析联系人 → 若多人则追问（大名或候选）；唯一确定后 → 该联系人**关联的会议**及重点。 | 「我们和王总还有李总上次会议的重点是什么？」 |
| **5. 按会议 → 查人** | 用户指定**某次会议**（如日期、标题），查**参会人/关联联系人**。 | 会议（日期或标题等）→ 该会议**关联的联系人**列表。 | 「10 月 1 号那个会议参会人有谁？」 |

### 实现上的对应关系

| 类型 | 主要依赖 | 说明 |
|------|----------|------|
| 1 | **会议总结/纪要的内容检索**（语义或关键词） | 对 summary/summaryData/transcript 做「含某主题」的检索，返回命中会议。 |
| 2 | **内容检索 + 会议↔联系人关联** | 先同 1 得到会议，再从 Meeting 取关联的 Contact(s)。 |
| 3 | **联系人解析（唯一）+ 会议↔联系人关联** | 按姓名/唯一标识解析出 Contact，再查该 Contact 关联的 Meeting 及总结。 |
| 4 | **联系人解析（可能多候选）+ 澄清 + 会议↔联系人关联** | 按昵称解析 Contact，若多候选则追问（第七章）；确定后同 3。 |
| 5 | **会议检索（按日期/标题等）+ 会议↔联系人关联** | 先定位到某 Meeting，再返回该 Meeting 关联的 Contact(s)（参会人）。 |

Phase 1 覆盖 1～5 时，需具备：**会议总结的内容检索**、**联系人及会议↔联系人关联**、**联系人解析与澄清**（含模糊昵称）、**按会议查参会人**。与第九章「会议总结 + 联系人关联」一致。

## 十一、知识库检索 Tools（输入/输出）

知识库内容均需**经后端检索**，不预注入全文。因此每种检索都应对应为 Agent 可调用的 **Tool**，每个 Tool 约定清晰的**输入**与**输出**，由后端实现、Agent 按需调用。

### 11.1 约定

- **输入**：Tool 的入参（如关键词、id、日期等），由 Agent 从用户问题或上一轮 Tool 结果中解析后传入。
- **输出**：可为 **仅 id 列表**（如 `meeting_ids`），或 **带关键字段的列表**（如 `[{ meeting_id, title, date, summary_snippet }]`），由后端契约与前端/Agent 约定；带字段可减少多轮 Tool 调用，推荐在返回会议/待办时带必要展示字段。

### 11.2 Phase 1 必备 Tools（会议 + 联系人关联）

| Tool 名称 | 输入 | 输出 | 对应问题类型 |
|-----------|------|------|--------------|
| **search_meetings_by_content** | 关键词/查询文本（string） | 会议列表：`meeting_id` 或 `[{ meeting_id, title, date, summary_snippet }]` | 1（按内容→会议）、2（按内容→人时先得会议再查人） |
| **resolve_contact** | 姓名或昵称（string） | 若唯一：`contact_id`（或带 name）；若多人：`candidates: [{ contact_id, name }]`，供 Agent 追问澄清 | 3、4（按人查会议前先解析/澄清人） |
| **get_meetings_by_contact** | 联系人 id（contact_id） | 会议列表：`meeting_id[]` 或 `[{ meeting_id, title, date, summary }]` | 3、4（按人→会议） |
| **get_contacts_by_meeting** | 会议 id（meeting_id） | 联系人列表：`contact_id[]` 或 `[{ contact_id, name }]` | 5（按会议→参会人） |

类型 2（按内容→人）：Agent 先调 **search_meetings_by_content** 得到会议（或 meeting_id），再对命中会议调 **get_contacts_by_meeting** 得到「和谁聊的」。

### 11.3 后续可扩展的 Tools（示例）

| Tool 名称 | 输入 | 输出 | 说明 |
|-----------|------|------|------|
| **get_reminders_by_contact** | 联系人 id（contact_id） | 待办 id 列表或 `[{ reminder_id, title, due_date, ... }]` | 按人查待办，如「我欠 Kevin 什么」 |
| **get_meeting_by_id** | 会议 id（meeting_id） | 单条会议详情（title, date, summary, participants 等） | 当仅持有 id 需补全详情时使用 |
| **search_meetings_by_date** | 日期或日期范围 | 会议列表（同上） | 按日期查会议，如「10 月 1 号的会议」 |

### 11.4 小结

- 知识库访问**全部通过后端 Tools**，每个 Tool 对应一种**输入→输出**的检索契约。
- Phase 1 至少实现：**search_meetings_by_content**、**resolve_contact**、**get_meetings_by_contact**、**get_contacts_by_meeting**，即可覆盖第十章五类问题。
- 输出格式（仅 id vs 带字段列表）由后端与 Agent/前端约定，推荐会议与待办返回时带必要展示字段以减少轮次。
