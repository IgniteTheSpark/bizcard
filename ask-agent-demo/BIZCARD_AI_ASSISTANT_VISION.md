# BizCard 作为 All-in-One AI Assistant — 产品愿景与架构方向

> **核心想法**：BizCard 可以做成一个 **all-in-one 的 AI 助手**：**Context 完全基于用户自己的内容**（会议、闪念、待办、Timeline、联系人、通话记录、Profile 等）；用户通过**一个 Ask Agent 的输入框**用自然语言要任何东西，Agent 查询后返回；**前端只负责把 Agent 返回的内容按类型渲染成对应的样子**。导航极简：**日历、Contacts、我的** 几个汇总入口 + **一个 Agent Ask 入口**即可。  
> **迭代**：当前优先 **Phase 1**——上线 **会议总结 + 联系人关联** 的 Ask Agent（按人查会议、联系人澄清等），体现 BizCard 名片差异化；本愿景为后续逐步扩展的方向。详见 [ASK_AGENT_PROMPT_SPEC.md 第九章](./ASK_AGENT_PROMPT_SPEC.md)。

---

## 一、愿景一句话

**BizCard = 以「我的数据」为唯一 context 的 All-in-One AI 助手**：一个问框，要什么查什么、返回什么就按什么形式展示；再加少量汇总入口（日历、联系人、我）做快速跳转与浏览。

---

## 二、三个支柱

| 支柱 | 说明 |
|------|------|
| **Context 完全基于用户** | Agent 的「知识」与可操作范围 = 用户自己的数据：会议/转写/总结、闪念、待办/Reminders、Timeline（会议+Agent Call+交换等）、联系人、通话记录、Profile/名片。不掺入「别人的数据」或开放互联网，除非产品明确增加该能力。 |
| **一个 Ask 入口，返回任意类型** | 用户输入一句（如「今天的待办」「最近的 timeline」「最近的通话记录」「我和王总上次聊了什么」）→ Agent 理解意图、查库/调 Tool、组装结果 → 返回**结构化结果**；前端根据**返回类型**选择对应渲染组件（待办列表、Timeline 卡片、通话列表、联系人卡片、纯文本等）。 |
| **极简导航：汇总入口 + Agent** | 不做「每个功能一个 Tab」。只保留少数**汇总型入口**：**日历**（Agenda / Reminders 视图）、**Contacts**（联系人列表）、**我的**（Profile / 设置 / 硬件）；以及 **Agent Ask 输入** 作为主入口（可放在首页顶部、底部或全局悬浮），用户「想问什么就问什么、想查什么就查什么」，结果在同一个对话流或结果区按类型渲染。 |

## 三、交互模型：Ask → 结构化返回 → 按类型渲染

### 3.1 流程

```
用户输入（自然语言）
    「今天的待办」「最近的 timeline」「最近的通话记录」「我和 Kevin 上次开会聊了什么」
    ↓
Agent（意图理解 + 查库/Tools）
    → 查 Reminders / Timeline / Agent Call 记录 / 会议摘要 / Contacts 等
    ↓
Agent 返回「结构化结果 + 展示类型」
    → 例如：{ type: "todo_list", data: [...] } / { type: "timeline", data: [...] } / { type: "call_list", data: [...] } / { type: "text", data: "…" }
    ↓
前端根据 type 选择组件渲染
    → 待办 → 列表/勾选；Timeline → 时间线卡片；通话 → 通话列表；纯回答 → 气泡/富文本
```

### 3.2 为何要「结构化 + 展示类型」？

若 Agent 只返回一段自然语言，前端很难自动判断「该用待办 UI 还是 Timeline UI 还是通话列表 UI」。因此需要约定：

- **Agent 侧**：在返回中带一个 **`response_type`**（或等价字段），以及该类型对应的 **payload**（列表、卡片数组、文本等）。
- **前端**：维护一张 **`response_type → 组件/页面`** 的映射，根据 type 渲染成「对应的样子」。

这样既保留「一个输入框要任何东西」的体验，又保证「待办像待办、Timeline 像 Timeline、通话像通话」的呈现一致性。

### 3.3 建议的 response_type 示例（可与后端/Agent 契约对齐）

| response_type | 含义 | 前端渲染建议 | 典型用户问法 |
|---------------|------|--------------|--------------|
| `todo_list` / `reminders` | 待办/Reminders 列表 | 带日期、联系人、勾选状态的列表 | 「今天的待办」「我欠 Kevin 什么」 |
| `timeline` | 时间线（会议+通话+交换等） | 时间线卡片流 | 「最近的 timeline」「这周的动态」 |
| `call_list` | 通话记录（Agent Call 等） | 通话列表（时间、对方、摘要） | 「最近的通话记录」「昨天的电话」 |
| `meeting_list` | 会议列表 | 会议卡片（标题、日期、摘要） | 「最近的会议」「和 Kevin 的会议」 |
| `contact_list` / `contact_card` | 联系人列表或单张名片 | 联系人列表/名片卡片 | 「我的联系人」「Kevin 的名片」 |
| `profile` | 用户自己的 Profile/名片 | 名片预览或编辑入口 | 「我的名片」「看一下我的简介」 |
| `text` / `answer` | 纯文本回答（摘要、解释、建议） | 对话气泡/富文本 | 「我和王总上次聊了什么」「上次聊 RWA 是哪次会」 |
| `disambiguation` | 需用户澄清（如多选一联系人） | 展示选项 + 追问 UI | 「王总」对应多人时的追问 |

实际类型集合可按产品需要增删；关键是与 Agent 输出契约、前端映射表一致。

## 四、导航与入口结构（极简版）

| 入口 | 说明 |
|------|------|
| **Agent Ask** | 主入口。输入框可置于首页顶部、底部或全局可唤出；用户输入 → 得到结构化结果 → 按 type 渲染。可保留多轮对话（历史 + 当前结果）。 |
| **日历** | 汇总入口：Agenda / 按日期的 Reminders 与会议，便于「看日程」而不必用自然语言。 |
| **Contacts** | 汇总入口：联系人列表/搜索，便于浏览与选择人。 |
| **我的** | 汇总入口：Profile 编辑、设置、硬件连接、Agent 设置等。 |

其他能力（会议详情、Timeline 详情、单次通话详情等）可通过 **Agent 返回结果中的点击跳转** 或 **深度链接** 进入，不必再在主导航占位。

## 六、实现要点小结

| 维度 | 要点 |
|------|------|
| **Context** | 仅用户自己的数据：会议、闪念、待办、Timeline、联系人、通话、Profile；Agent 与 Tools 只在该范围内查询与操作。 |
| **入口** | 一个 Ask 输入 + 日历、Contacts、我的 三个汇总入口；Ask 为主，其余为快捷浏览。 |
| **Agent 返回** | 除自然语言外，约定 **response_type + payload**（如 todo_list / timeline / call_list / text / …），与前端契约一致。 |
| **前端** | 根据 response_type 映射到对应组件（待办列表、Timeline、通话列表、名片、纯文本等）；一套对话/结果容器，多种展示形态。 |

这样 BizCard 在体验上就趋近「一个问框 + 我的数据 = 要什么查什么、返回什么就按什么样子展示」的 All-in-One AI 助手，同时保留日历/联系人/我的三个汇总入口，结构清晰、便于迭代。
