## Executive Summary

BizCard 是一个"软硬结合"的商务社交产品，通过录音硬件捕获线下会面场景，结合AI能力帮助用户管理和维护商务关系。

**核心定位**：
> "你负责遇到对的人，BizCard负责帮你不搞砸这段关系。"

**当前Slogan**：
> "Turn conversations into contact notes"

**战略选择**：
- **方向A（Relationship Intelligence）** 作为产品核心基础设施
- **方向B（活动场景）** 作为增强型使用场景
- 硬件作为高效数据采集入口，软件作为价值沉淀和变现主体
- **短期聚焦硬件叙事**（Kickstarter），**长期强化软件独立价值**

**目标用户**：
- 核心：商务人士（销售、BD、创业者、投资人）
- 扩展：职业社交场景（行业活动、社群、专业会议）

## 1. 品牌定位与Slogan

### 1.1 当前Slogan分析

**当前**："Turn conversations into contact notes"

| 维度 | 分析 |
|------|------|
| 优点 | 简洁清晰，强调AI处理能力 |
| 局限 | 聚焦"notes"（记录），未体现"action"（行动）和"relationship"（关系） |
| 覆盖层 | Layer 1-2（采集+处理），未体现Layer 3-4（展示+驱动） |

**核心问题**：Notes是中间产物，不是最终价值。用户真正要的是：
- ✅ 不遗忘重要的人
- ✅ 知道下一步该做什么
- ✅ 在关键时刻有context可用

### 1.2 Slogan迭代方向（建议）

| 方向 | 示例 | 侧重点 |
|------|------|--------|
| 强调行动 | "Turn conversations into actions" | 从记录到执行 |
| 强调关系 | "Turn encounters into relationships" | 从一次见面到持续关系 |
| 强调不遗忘 | "Never forget who you met" | 解决社交失忆 |
| 强调专业感 | "Your AI-powered relationship memory" | 第二大脑 |

**建议**：当前slogan可保留用于Kickstarter阶段（聚焦硬件能力），Phase 2后根据产品演进考虑迭代。

### 1.3 产品叙事

**一句话定位**：
> "你负责遇到对的人，BizCard负责帮你不搞砸这段关系。"

**电梯演讲**：
> "BizCard是你商务世界的'行车记录仪'与'智能秘书'。你只管在现实中握手（Tap），BizCard负责帮你记住一切（Capture）、整理一切（Process）并完成跟进（Close）。"

## 2. 目标用户定位

### 2.1 用户群分层

| 层级 | 用户画像 | 使用场景 | 优先级 |
|------|----------|----------|--------|
| **核心用户** | 销售、BD、创业者、投资人 | 商务会议、客户拜访、销售跟进 | P0 |
| **扩展用户** | 职业社交活跃者 | 行业活动、社群聚会、专业会议 | P1 |
| **观望用户** | 普通职场人 | 偶尔的商务会面 | P2 |

### 2.2 为什么聚焦商务人士

| 维度 | 商务场景 | 泛社交场景 |
|------|----------|------------|
| 录音接受度 | 高（常见行为） | 低（可能让人不适） |
| 跟进需求 | 强（签单/合作） | 弱（无明确目标） |
| 付费意愿 | 高（效率=钱） | 低 |
| 硬件形态匹配 | 高（名片形态自然） | 低（派对拿名片很奇怪） |

### 2.3 不建议短期扩展的场景

| 场景 | 能用吗？ | 问题 |
|------|----------|------|
| Casual meetup | ⚠️ 勉强 | 录音可能让人不适 |
| 派对/社交活动 | ❌ 不适合 | 硬件形态不match |
| 约会 | ❌ 不适合 | 录音约会对象=creepy |

### 2.4 可扩展的中间地带

```
"职业社交"场景（介于纯商务和纯社交之间）

• 行业社群活动（Web3 meetup、设计师聚会）
• 投资人/创业者社交
• 校友会、商会活动
• 专业会议的社交环节

特点：
• 有"跟进"需求（不只是交朋友）
• 录音可接受度较高
• 用户愿意付费
```

## 3. 市场痛点分析（我们的战场）

### 3.1 商务社交的两大核心痛点

| 痛点 | 描述 | 现有解决方案 | BizCard能做吗？ |
|------|------|--------------|-----------------|
| **帮我找到合适的人** | 在茫茫人海中发现潜在客户/合作伙伴 | LinkedIn、Apollo、Hunter | ❌ 不能 |
| **帮我维护好找到的人** | 建立关系后不遗忘、持续跟进 | Salesforce、HubSpot、Covve | ⚠️ 可以做轻量版 |

### 3.2 BizCard的定位：中间过渡层

```
找人（高价值，难做）         ←  LinkedIn/Apollo 的领域
       ↓
   ┌─────────────────────────────────────────────────────┐
   │  BizCard 的战场                                     │
   │  "遇到对的人之后的一切"                              │
   │                                                     │
   │  • 快速建联（硬件Tap）                               │
   │  • 自动记录（录音+AI）                               │
   │  • 行动驱动（Actions）                               │
   │  • 关系维护（Reminders）                             │
   └─────────────────────────────────────────────────────┘
       ↓
深度管理（高价值，难做）      ←  Salesforce/HubSpot 的领域
```

### 3.3 诚实的自我定位

| 我们能做的 | 我们不能/不应该做的 |
|-----------|---------------------|
| 硬件实现高效建联 | 帮用户发现潜在客户 |
| AI总结会议内容 | 深度销售漏斗管理 |
| 提取Action Items并推动执行 | 复杂的Pipeline追踪 |
| 关系维护提醒 | 团队协作和权限管理 |
| 轻度联系人管理 | 企业级CRM功能 |

## 5. 战略方向选择

### 5.1 两个可选方向

#### 方向A：Relationship Intelligence（关系智能）

**核心理念**：日常的商务关系管理和维护工具

**特点**：
- 以"人"和"关系"为核心
- 日常使用场景
- 需要培养用户习惯
- 订阅变现

**对标产品**：Covve + AI能力增强

---

#### 方向B：Event-based 活动场景

**核心理念**：围绕展会/活动的社交效率工具

**特点**：
- 以"活动"为核心
- 低频但高强度使用
- 硬件价值最大化的场景
- 可做B端销售

**挑战**：
- 活动前"推荐对的人"数据不足，难以实现
- 运营和商务难度大
- 一年只有几次大展，日常留存低

---

## 7. 阶段性规划

### 7.1 Phase 1: Reminder Loop（规划中）

> **目标**：打通"事件-提醒-人"的闭环，以轻量日历模式驱动用户行动

**设计理念变化**：
- 从"Action Hub（待办清单）"调整为"Reminders Hub（轻量日历）"
- 降低push感，以日程安排的方式呈现待办事项
- 新增Calendar页面作为Reminders的详细管理入口

| 功能 | 描述 | 价值 |
|------|------|------|
| 首页Reminders Hub | 轻量日历组件，按Today/Next 7 Days分组 | 解决"不知道做什么"，降低焦虑感 |
| Calendar页面 | Day/Week/Month视图，支持Snooze/Delete | 完整的日程管理 |
| Meeting Reminders | 从会议中提取可执行提醒 | 录音→行动的转化 |
| Contact Activity上下文 | Activity不只是标题 | 完整的人物画像 |
| Profile精简 | 首页聚焦日程和关系 | 减少干扰 |

**成功标准**：
- [ ] 用户进入首页能明确知道"今天要做什么"
- [ ] Reminder提取准确率 > 70%
- [ ] Reminder完成率 > 30%

---

### 7.2 Phase 2: Engagement Loop + AI搜索

> **目标**：让用户有理由每天/每周回来，并能从社交资产中获取价值

| 功能 | 描述 | 价值 |
|------|------|------|
| **个人AI搜索** | 从社交资产（人物、会议、待办）中查询/提取内容 | 数据可用性 |
| **Action可执行** | Email真正发送、Calendly booking | 闭环完成 |
| **主动Reminder** | 关系check-in提醒 | 被动→主动 |
| **关系健康度** | 联系人分层：活跃/需关注/休眠 | 可视化价值 |
| **Pre-meeting情报** | 明天见的人，上次聊了什么 | 会前准备 |

**成功标准**：
- [ ] DAU/MAU > 25%
- [ ] 7日留存 > 40%
- [ ] Action完成率 > 50%
- [ ] AI搜索使用率 > 20%

---

### 7.3 Phase 3: 深度关系管理

> **目标**：强化关系维护能力，向Relationship Intelligence深处走

| 功能 | 描述 | 价值 |
|------|------|------|
| 双向网络效应 | 两个Bizcard用户见面时的双向同步 | 网络价值 |

### 7.4 Phase 4: Event Mode（可选增强）

> **目标**：活动场景的专项增强

## 8. 硬件战略定位

### 8.1 硬件的价值重新评估

| 硬件能力 | 软件替代方案 | 硬件优势 |
|----------|--------------|----------|
| Tap开始录音 | App点击录音 | 3秒 vs 10秒 |
| NFC传名片 | 二维码/AirDrop | 更酷，更快 |
| E-ink展示 | 手机展示 | 专业感，省电 |
| 被动录音 | 需要手动操作 | Always-on |

**结论**：硬件提供的是"更优雅的体验"，而非"不可替代的能力"。

### 8.2 硬件的战略角色

```
当前认知：硬件是核心，软件是配套
长期调整：软件是核心，硬件是增强

硬件的角色：
1. 高效数据采集入口（尤其在活动场景）
2. 差异化卖点和品牌形象
3. Premium tier的付费点
4. 双边网络效应的触发器（两个硬件用户碰一碰）
```

### 8.3 软件录音功能策略

**背景**：当前App未开放独立录音功能，仅支持硬件触发录音。

**原因**：
1. Kickstarter众筹阶段需要突出硬件价值
2. 同时支持软件和硬件录音会增加App逻辑复杂度

**阶段性策略**：

| 阶段 | 软件录音 | 理由 |
|------|----------|------|
| **Kickstarter阶段** | ❌ 不开放 | 聚焦硬件叙事，强化"为什么要买硬件" |
| **硬件发货后3-6个月** | ⚠️ 评估 | 根据硬件用户反馈和留存数据决定 |
| **长期** | ✅ 开放（可能有限制） | 扩大用户漏斗，验证软件独立价值 |

**长期开放软件录音的好处**：
- 软件可以免费拉新（试用 → 转化买硬件）
- 覆盖硬件不适用的场景（视频会议、电话）
- 验证"软件独立存在是否有价值"的假设
- 硬件成为"升级体验"而非"入门门槛"

**软件录音 vs 硬件录音的定位**：

| 维度 | 软件录音 | 硬件录音 |
|------|----------|----------|
| 触发方式 | 打开App → 点击按钮 | Tap一下 |
| 速度 | 10秒 | 3秒 |
| 适用场景 | 视频会议、电话、有准备的会面 | 线下偶遇、展会、快速场景 |
| 仪式感 | 低（掏手机） | 高（专业设备） |
| 用户群 | 所有人（免费入门） | 高频用户/付费用户 |

### 8.4 软硬件关系

| 用户类型 | 使用方式 | 付费模式 |
|----------|----------|----------|
| **软件用户** | App-only，手动录音/扫码 | 基础订阅 |
| **硬件用户** | 硬件+App，高效采集 | 硬件购买 + 高级订阅 |

**关键验证点**：
> 软件单独存在时，用户会用吗？愿意付多少钱？

如果答案是肯定的，硬件可以作为premium增强。
如果答案是否定的，需要重新评估软件的独立价值。

```
│  • 基础AI Summary                                       │
│  • 有限Contacts（如50人）                                │
│  • 目的：试用 → 转化                                     │
└─────────────────────────────────────────────────────────┘
```

### 9.2 变现节奏

| 阶段 | 变现重点 | 策略 |
|------|----------|------|
| Kickstarter | 硬件销售 | 早鸟价众筹 |
| 发货后6个月 | 硬件 + 订阅 | 硬件用户转订阅 |
| 长期 | 订阅为主 | 开放Free tier拉新，转化Pro |

### 9.3 关键假设验证

| 假设 | 验证方式 | 时间点 |
|------|----------|--------|
| 用户愿意为Pro订阅付费 | 付费转化率 > 5% | Phase 2 |
| 软件单独有价值 | 无硬件用户的留存和付费 | 软件录音开放后 |
| 硬件提升体验明显 | 硬件用户 vs 软件用户的留存差异 | 硬件发货后3个月 |

## 10. 成功指标体系

### 10.1 北极星指标

> **"每周完成的Reminders数量"**

理由：
- 反映了用户的真实使用价值
- 涵盖了采集→处理→行动的完整链路
- 可以驱动留存和付费

### 10.2 关键指标矩阵

| 类别 | 指标 | Phase 1 目标 | Phase 2 目标 |
|------|------|--------------|--------------|
| **留存** | DAU/MAU | 15% | 25% |
| **留存** | 7日留存 | 25% | 40% |
| **采集** | 周均新增Contacts | 2 | 3 |
| **处理** | Reminder提取准确率 | 70% | 80% |
| **行动** | Reminder完成率 | 30% | 50% |
| **变现** | 付费转化率 | — | 5% |

### 10.3 用户旅程健康度

```
用户旅程：
注册 → 首次录音 → 首个Reminder → 首次完成Reminder → 周活跃 → 付费

关键转化点：
1. 注册→首次录音：设备引导是否顺畅
2. 首次录音→首个Reminder：AI提取是否有效
3. 首个Reminder→完成：执行路径是否顺滑
4. 完成→周活跃：是否有理由回来（Calendar成为入口）
5. 周活跃→付费：价值感知是否明确
```
