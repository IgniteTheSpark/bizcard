# BizCard 当下关键点（共识版）

> 用于对齐：用户来源、叙事、资产模型与功能锚点的当前共识。  
> 与 [USER_ANCHORING_STRATEGY_REFLECTION.md](./USER_ANCHORING_STRATEGY_REFLECTION.md) 中「以硬件为主」方向一致。

---

## 1. 用户来源：以硬件为主

- **当前**：用户主要来自**硬件**（购买/众筹/开箱激活）。
- **含义**：获客与激活以硬件为圆心；App 是硬件用户的「回来后的主场」，而非独立拉新的主入口。

---

## 2. 叙事：个人商务效率工具 + 个人 CRM

- **对外主叙事**：个人商务效率工具 + 个人 CRM（个人的商务助手、关系与待办管理）。
- **不强调**：纯「电子名片」或纯「录音笔」；名片与录音都是该叙事下的能力，不是独立品类。

---

## 3. App 内资产类型（三类）

用户在 App 内拥有的核心资产只有三种：

| 资产类型 | 说明 |
|----------|------|
| **Notes** | 记录类（会议/闪念/手动笔记等），来自硬件录音或 App 内创建。 |
| **Reminders** | 待办/提醒，可从 Notes 提取或手动创建。 |
| **Contacts** | 联系人；与 Notes、Reminders 可关联。 |

产品设计与 Ask Agent 的能力边界都围绕这三类资产展开。

---

## 4. Ask Agent：基于用户 Context 的行动能力

- Ask Agent 可基于用户**当前上下文**（某条 Note、某个 Contact、某条 Reminder 或全局）进行查询与行动。
- 能力范围：理解与操作对象 = Notes + Reminders + Contacts（及由此衍生的日程、摘要等）。

---

## 5. 以硬件为主 = 以录音为主 = Notes 为关键锚点

- **链条**：硬件 → 录音 → **Notes** 为关键锚点。
- **关系**：
  - **Notes 关联人物**（Contacts）：一条 Note 可绑定到具体联系人。
  - **Notes 关联 Reminders**：从 Note 提取或创建 Reminder，形成「记录 → 待办」闭环。
- **设计含义**：首页与主流程应突出「Notes 为中心」的视图与动线（今日会议/闪念、Note→人、Note→Reminder），而不是泛泛的「联系人列表」或「日历」优先。

---

## 6. Contacts → 名片添加与交换 → Profile

- 因为用户有 **Contacts** 需求，所以需要**添加别人名片**；添加过程中会**交换自己的名片**，因此需要**名片 / Profile**。
- **因果**：Contacts（管人）→ 加人、换名片 → 需要「我的名片」可被对方保存 → **Profile** 存在。
- **定位**：Profile 是「为联系人场景服务的我的身份卡片」，支撑交换与留存，而非独立于 Contacts 的另一个主叙事。

---

## 小结（一句话）

> **用户来自硬件；叙事是个人商务效率 + 个人 CRM；App 内资产是 Notes、Reminders、Contacts；以硬件/录音为入口故以 Notes 为锚点，Notes 关联人与 Reminder；因有 Contacts 才有加名片与换名片，故需要 Profile。**

---

## 从要点到「核心卖点」与进 App 后的感知

上述 6 点更多是**内部结构共识**（我们有什么、因果是什么），本身还不是用户一眼能懂的**卖点**。要让用户进 App 后有感知，需要把结构翻译成**价值话术**，并在**首屏、空状态、首次闭环**里体现。

### 要点 → 可对外说的核心卖点（建议）

| 内部要点 | 用户侧卖点（一句话/感知） |
|----------|---------------------------|
| 硬件为主 + 录音 → Notes 为锚点 | **「录下来的，都会变成可查、可跟进的笔记」**——不丢事、不丢人。 |
| Notes 关联人 + 关联 Reminders | **「每条记录都连到对的人、和该做的下一步」**——从记录到待办到人，一条链。 |
| 个人商务效率 + 个人 CRM | **「你的第二大脑：谁、说了什么、要做什么，都在这」**——个人向 CRM，不靠记脑子。 |
| Ask Agent 基于 context 行动 | **「用说的就能查、就能做」**——问「上次和 Kevin 说了啥」「明天要跟进的」都能答、能行动。 |
| Contacts + 交换名片 → Profile | **「管好人的同时，换名片时对方拿到的也是你」**——加人、换片、被存，一体。 |

**汇总成一句进 App 后希望用户形成的认知：**  
> **「我录的/记的都会变成笔记，笔记会自动连到人和待办；我要查、要跟，用说就行。」**

---

### 进 App 后如何让用户「有所感知」（落点建议）

目标：用户**第一次打开 App**（尤其是硬件用户刚激活后）能在 1～2 屏内感受到上述价值，而不是只看到空列表或抽象菜单。

| 落点 | 做法建议 | 对应卖点 |
|------|----------|----------|
| **首屏信息层级** | 首屏以 **Notes（今日/最近）** 为视觉重心，其次 **Reminders**；Contacts 与 Profile 不抢第一眼。有数据时：优先展示「带人物的 Note」「带 Reminder 的 Note」。 | 「录下来的变成笔记」「记录连到人与待办」 |
| **空状态文案** | 无数据时，不用泛泛的「暂无内容」。改为**预期设定**：如「用设备录下的会议与闪念会出现在这里，并自动关联联系人与待办」「说完就记，记完就跟」。 | 建立「录音 → 笔记 → 人/待办」的预期 |
| **首次有数据后的反馈** | 硬件首次同步出 Notes 后，**一次性轻量提示**：如「已为你生成 3 条笔记、2 个待办，并关联到 2 位联系人」或「你的第一条会议笔记已就绪，要试试让助手帮你总结吗？」 | 强化「记录→人→待办」闭环、引出 Ask Agent |
| **Ask Agent 入口与首句** | 进入 Ask Agent 时，**默认 hint** 直接体现能力边界：如「可以问我：今天见了谁、和 Kevin 聊了什么、明天要跟进什么」。 | 「用说的就能查、就能做」 |
| **一处「人+记录+待办」同屏** | 至少在**一个核心页面**（如某条 Note 详情，或 Contact 详情）同时露出：这条记录、关联的人、从记录里出的 Reminders。让用户一眼看到「笔记—人—待办」三条线在一起。 | 「每条记录都连到对的人和下一步」 |

---

### 自检问题（做首屏/空状态/引导时可用）

- 用户打开 App 第一眼，能否在 **10 秒内** 用一句话说出「这个 App 是干什么的」？
- 若用户**还没有任何录音/笔记**，空状态是否仍然传达「将来会有什么、为什么值得留在这里」？
- 用户**第一次**看到一条带 Reminder、带联系人的 Note 时，是否有明显「原来是这样连在一起的」的感知？

若以上三点都能答「是」，当前要点才真正转化为用户可感知的核心卖点。
