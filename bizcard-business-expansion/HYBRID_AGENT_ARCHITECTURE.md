# 混合 Agent 架构设计：MCP 极速路由与 OpenClaw 沙箱的协同

## 一、 为什么需要混合架构 (Hybrid Architecture)？

在 Ask Agent 升级为“全能执行助理”的愿景下，我们面临两类截然不同的任务：

1. **结构化/高频/秒级任务**：
   - 需求：“找出上周关于 RWA 的会议结论”、“查一下飞书下周二下午两点有没有空”。
   - 特征：依赖内部数据库或标准 API（如钉钉/飞书 Open API），要求**毫秒到秒级的极速响应**。
2. **非结构化/跨系统/长流任务**：
   - 需求：“去网上找一份关于 RWA 的最新研报，并用我的邮箱发给 Kevin”。
   - 特征：系统未提供 API、需要模拟人类打开浏览器搜索、点击、下载文件、跨应用粘贴。此类任务通常需要**分钟级的执行时间**。

如果全部用传统 API，覆盖不了场景 2；如果全部用 OpenClaw (基于 Vision/Computer Use 的沙箱)，成本极高且响应奇慢，会毁掉场景 1 的体验。

**解决方案：Nanobot 作为大脑进行意图路由，结合 `MCP Tools (快车道)` 与 `OpenClaw 沙箱 (慢车道)` 的混合架构。**

## 三、 核心协作流推演

### 3.1 任务拆解与降级机制 (Nanobot 的核心逻辑)

Nanobot 在收到复杂请求时，会执行一套严格的评估机制：
- **优先使用 MCP**：只要能通过现有的 MCP Tools 完成的，绝对不唤醒 OpenClaw（省钱、省时）。
- **任务切片**：例如任务“根据会议记录给 Kevin 发一封附带研报的邮件”。
  - 切片 A：获取会议记录和 Kevin 邮箱（路由给 `MCP (bizcard-db)`，极速完成）。
  - 切片 B：搜索研报并发送邮件（路由给 `OpenClaw`，因为没有外部研报库 API）。

### 3.2 异步交互与用户感知 (UX 设计)

因为 OpenClaw 的执行是慢速的（通常需要 1-3 分钟），UI 层不能死等，必须有一套完美的异步交互机制：

1. **接单确认**：Nanobot 拆解任务后，立刻在 Chatbox 回复：
   > “✅ 已拿到 Kevin 的联系方式和会议记录。\n ⏳ 正在唤醒您的专属云端助理为您检索研报并起草邮件，这可能需要 1-2 分钟，您可以先去忙别的，完成后我会通知您。”
2. **实时进度流 (Stream Status)**：在聊天气泡下方，展示类似 CI/CD 流水线的进度条：
   - `[正在打开浏览器检索...]`
   - `[成功下载 RWA_Report_2026.pdf...]`
   - `[正在登录邮箱起草正文...]`
3. **完成验收与截图证明**：OpenClaw 执行完毕后，向 Nanobot 发送回调。Nanobot 推送最终结果给用户：
   > “🎉 邮件已成功发送给 Kevin！附带了刚刚为您找到的 30 页研报。”

## 四、 商业与安全价值

1. **物理级数据隔离**：对于商务人士，安全是红线。为用户部署动态拉起、用完即毁的独立 OpenClaw 沙箱 VM，是极具说服力的安全卖点。
2. **降维打击的体验**：目前市面上大多数 Agent 产品只能做文字游戏或有限的 API 调用。BizCard 融合 OpenClaw 后，真正具备了“代替人类敲键盘点鼠标”的能力，这是对传统工具类 SaaS 的降维打击。
