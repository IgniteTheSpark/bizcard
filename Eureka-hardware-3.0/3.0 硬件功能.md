# 3.0 硬件功能

# 08 Eureka-3.0 硬件功能预期与场景说明

> 本文面向 Eureka-3.0 硬件、嵌入式、App、算法与业务团队对齐预期，描述我们希望硬件具备的能力、用户使用场景和软硬件协作边界。本文不是完整 PRD，也不定义验收标准、异常状态矩阵、数据字段、协议细节或技术实现方案；具体实现可行性、成本、功耗、结构限制和联调方案由后续硬件/固件/软件方案继续展开。

## 1. 产品范围

Eureka-3.0 硬件是一张具备 E-ink 显示、录音输入、蓝牙连接、NFC、麦克风自动切换和本地状态反馈能力的商务硬件卡片。

它延续 BizCard 的商务关系管理定位，但硬件名称升级为 Eureka 后，产品预期不再只是“一张可展示名片的卡片”，而是一个可被随身携带、低打扰触发、随时展示和记录商务上下文的硬件入口。

硬件端的核心任务不是独立完成复杂 AI 处理，而是：

1.  采集用户输入：闪念、会议录音、NFC 触发、按键操作。
    
2.  维护本地状态：录音中、离线暂存、低电量、麦克风来源、NFC 模式等。
    
3.  承载低功耗展示：默认展示页、动态反馈页、硬件本地状态页。
    
4.  与 App 协同：在线时上传输入并接收 App 渲染内容；离线时本地暂存，恢复在线后同步。

面向其他团队对齐时，可以把 Eureka-3.0 理解为四类能力的集合：

1.  **商务展示设备**：平时展示名片、二维码、个人主页、活动页或今日简报。
    
2.  **轻量输入设备**：用户通过按键快速记录闪念、发起 Ask Agent 或开始会议录音。
    
3.  **会议辅助设备**：会议中持续录音，并在 E-ink 上展示低打扰的录音状态、提词、提醒或后续动作。
    
4.  **近场触发设备**：通过 NFC 完成名片交换、个人主页打开、App deep link 或其他后续可评估的门禁/车钥匙能力。
    
本文后续内容重点描述这些能力在用户场景中的预期表现，不展开完整产品流程和验收规则。
    

## 2. 硬件功能模块

| 模块 | 核心能力 | 说明 |
| --- | --- | --- |
| E-ink 显示 | 常显、翻页、显示 App 反显内容、本地状态页 | 不承担复杂排版；App 在线时以 App 渲染内容为主，离线或关键硬件状态时显示本地状态 |
| 物理按钮 | 闪念长按、会议开始/结束、上下翻页、清空/返回 | 入口行为必须稳定、可盲操、反馈明确 |
| 音频采集 | 闪念短录音、会议长录音、离线录音暂存 | 闪念和会议的录音逻辑不同，不共用同一套交互 |
| 麦克风系统 | 环境麦、骨传导麦、耳机麦识别与切换 | 切换结果需要对用户可见 |
| App 连接 | 与 App 协同、状态同步、音频/内容同步、E-ink 图片下发 | 作为在线协同前提，不单独展开连接管理 |
| NFC | 名片链接、个人主页、App 动作触发、扩展门禁/车钥匙能力 | 门禁和车钥匙属于高安全/生态能力，后续单独评估 |
| 本地存储 | Base 图片缓存、Dynamic 最近内容、离线闪念、会议录音 | 硬件不做长期数据仓库，只保留必要内容 |
| 电源与状态 | 电量、低电提醒、充电状态、待同步状态 | 关键状态需要能在 E-ink 上反馈 |

## 3. E-ink 显示能力预期

### 3.0 屏幕区域

E-ink 需要同时承载用户长期展示内容和硬件状态。为避免用户 Profile、二维码等长显内容被蓝牙、电量、同步状态频繁打断，屏幕建议采用“状态区 + 主内容区”的展示结构。

建议：

*   顶部保留常驻状态区，用于展示蓝牙、电量、同步、录音、麦克风等少量状态。
    
*   下方作为主内容区，用于展示 Profile、二维码、Agent 结果、会议提示等内容。
    
*   状态区与主内容区的具体比例不在本文中写死，后续由 ID、UI 和屏幕尺寸共同确定。
    
*   默认展示名片或二维码时，状态区应尽量轻量，不影响主内容识别。
    
*   录音中、低电量、离线暂存、存储满等关键状态，可以扩大状态展示，必要时临时切换为整屏状态页。
    

状态区至少应支持展示：

*   蓝牙状态。
    
*   电量状态。
    
*   同步中或待同步状态。
    
*   录音中状态。
    
*   当前麦克风来源。
    
*   NFC 触发或当前 NFC 模式。
    

### 3.1 默认展示

硬件应支持一组默认展示内容，用于无事件发生时常显。

首版默认展示至少包括：

*   我的名片或 Profile。
    
*   二维码或个人链接。
    
*   用户在 App 中设置的默认页。
    

预期说明：

1.  用户可通过 App 配置默认展示内容。
    
2.  硬件应可在离线时展示已缓存的默认内容。
    
3.  无动态内容时，上下翻页只在默认展示内容中循环。
    
4.  用户清空动态内容后，应回到默认展示态。
    
5.  默认展示内容需要预留状态区，避免蓝牙、电量等状态遮挡 Profile 或二维码。
    
典型场景：

*   **商务会面前**：用户把 Eureka 放在桌面或递给对方，屏幕默认展示自己的姓名、公司、职位和二维码。对方不需要打开 App，也能扫码进入个人主页或数字名片。
    
*   **展会/活动场景**：用户可提前在 App 中设置活动专用展示页，例如展会二维码、品牌页、预约链接或产品介绍页。Eureka 在活动中作为低功耗常显展示物。
    
*   **临时身份展示**：用户在不同商务场景中切换不同身份，例如公司商务名片、个人顾问主页、招聘活动二维码。硬件只负责展示已缓存内容，具体内容配置由 App 完成。
    

### 3.2 动态反馈

硬件应支持 App 下发的动态反馈内容。

典型动态反馈包括：

*   闪念处理结果。
    
*   Ask Agent 结果摘要。
    
*   录音状态变化。
    
*   会议 Copilot 或实时翻译内容。
    
*   Reminder 或日程提醒。
    
*   NFC 触发结果。
    
*   麦克风来源变化。
    

预期说明：

1.  动态反馈可以临时覆盖默认展示内容。
    
2.  存在动态反馈时，上下翻页优先在动态内容中回看。
    
3.  清空/返回操作只清空屏幕上的动态内容，不删除 App 历史或硬件待同步内容。
    
4.  动态内容过期后，应回到默认展示态。
    
典型场景：

*   **闪念记录后**：用户长按按钮说“明天提醒我跟 Kevin 确认合同”，松开后屏幕显示“已记录”或“已创建提醒”的摘要反馈，让用户知道这次输入没有丢失。
    
*   **Ask Agent 查询后**：用户通过硬件问“Kevin 上次跟我聊了什么”，App/Agent 处理后可把一句话摘要推到 E-ink，例如“上次沟通重点：报价、交付周期、下周回访”。
    
*   **会议中提示**：会议录音过程中，App 可下发低打扰提示，例如“对方提到预算限制”“建议追问决策时间”“会后跟进：发送方案 PDF”。
    
*   **Reminder 到达**：重要跟进提醒到达时，屏幕可短暂覆盖默认名片页，提示用户“15:00 跟进 Kevin 报价”，过期后回到默认展示。
    

### 3.3 基础硬件状态

硬件状态只展示首版必要的基础状态，用于让用户知道设备当前在做什么。这里不展开复杂状态分支。

至少需要覆盖：

| 状态类型 | 典型内容 | 展示方式 |
| --- | --- | --- |
| 闪念状态 | 记录中、结束反馈 | 记录中可在状态区展示；结束后短暂反显 |
| 录音状态 | 录音中、录音标记反馈 | 状态区常驻，标记反馈短暂反显 |
| 蓝牙状态 | 未连接、已连接手机、已连接手机 + 耳机 | 状态区轻量展示 |
| 文件同步 | 同步文件中 | 状态区轻量展示 |
| 电源状态 | 电量百分比 | 状态区常驻 |
| 麦克风状态 | 环境麦、骨传导、耳机麦 | 状态区或录音状态中展示 |
| NFC 状态 | 当前 NFC 模式 | 状态区或短暂提示 |

预期说明：

1.  E-ink 同一时刻只显示一张完整画面。
    
2.  App 反显内容和硬件本地状态页不做复杂叠加。
    
3.  关键硬件状态可以临时覆盖 App 反显内容。
    
4.  App 恢复在线后，应能重新同步当前屏幕状态。
    
5.  本地状态页可以沿用状态区和主内容区结构，也可以在关键状态需要强调时使用整屏提示。
    
6.  蓝牙连接/断开只作为用户可见状态和在线协同前提，不在本文中展开连接管理功能。
    
场景边界：

*   用户需要在不打开手机的情况下知道设备是否正在录音、是否连接 App、当前大致电量、是否有待同步内容。
    
*   硬件状态展示的目标是建立用户信任，而不是替代 App 的完整设备管理页。
    
*   低频或复杂状态可以留在 App 中解释，E-ink 只展示会影响用户当前动作判断的关键信息。
    

## 4. 物理输入能力预期

Eureka-3.0 的物理输入目标是让用户在商务现场可以“盲操作”。用户不应该每次都需要拿起手机、打开 App、找入口；硬件按键需要承载最常用、最高频、最需要即时性的动作。

整体预期：

*   闪念是短输入，强调“想到就说，说完就走”。
    
*   会议是长输入，强调“开始后持续可靠，不因手机状态变化轻易中断”。
    
*   翻页和清空是屏幕控制，强调“快速回到我想展示的内容”。
    
*   具体按键形态不在本文展开，本文只表达希望硬件承载的功能入口和用户场景。

### 4.1 闪念按钮

闪念入口采用长按输入、松开结束。

预期说明：

1.  用户长按按钮时开始录制闪念。
    
2.  用户松开按钮时结束本次闪念。
    
3.  闪念只处理松开后形成的完整音频。
    
4.  闪念不做录制过程中的实时传输、实时 ASR 或实时 Agent 处理。
    
5.  松开时在线，则立即上传 App 处理，并在 E-ink 上展示处理反馈。
    
6.  松开时离线，则本地暂存，并在 E-ink 上展示“已暂存，连接 App 后同步”。
    
7.  录制过程中连接状态变化不改变本次闪念的处理方式，只在松开时判断在线或离线。
    
典型场景：

*   **会后走廊记录**：用户刚结束一次沟通，来不及打开 App，只需要长按 Eureka 说“Kevin 对价格敏感，下次先发基础版报价”，松开后即完成记录。
    
*   **灵感捕捉**：用户在通勤或活动现场想到一个跟进点，长按说一句话，屏幕给出“已暂存”或“处理中”反馈。
    
*   **轻量指令**：用户说“帮我明天下午三点提醒回访 Kevin”，App 在线时可由 Agent 转成 Reminder；硬件只负责录音入口和结果反馈。
    

### 4.2 会议按钮

会议入口采用点击开始、再次点击结束。

预期说明：

1.  用户点击会议按钮后开始会议录音。
    
2.  用户再次点击后结束会议录音。
    
3.  离线时，硬件继续本地会议录音，并展示基础录音状态。
    
4.  在线且用户开启实时增强时，硬件可将后续音频传给 App，用于 Copilot、实时翻译或会议提示。
    
5.  在线转离线时，实时增强停止，但会议录音继续。
    
6.  离线转在线时，后续音频可开始实时传输；离线阶段的历史音频在会后或恢复在线后同步。
    
典型场景：

*   **桌面会议**：用户把 Eureka 放在桌面，点击开始录音。屏幕显示“录音中”和基础状态，会议结束再次点击结束。
    
*   **客户拜访**：用户在客户现场不方便频繁看手机，Eureka 负责持续录音；App 在线时可把关键提词或客户背景推送到 E-ink。
    
*   **离线会议**：手机不在身边或 BLE 暂时不可用时，硬件仍然承担本地录音入口。用户需要看到这是“本地录音/待同步”，而不是误以为 App 已完成处理。
    
*   **会议增强**：在线状态下，App/Agent 可基于实时音频生成低打扰提示，例如“对方问到了合同周期”“建议确认预算范围”。这类增强是会议体验加分项，不是录音成立的前提。
    

### 4.3 翻页与清空

硬件应支持上下翻页和清空/返回操作。

预期说明：

1.  无动态内容时，上下翻页切换默认展示页。
    
2.  有动态内容时，上下翻页回看最近动态内容。
    
3.  清空/返回只影响屏幕显示，不删除录音、闪念、同步队列或 App 历史。
    
4.  关键状态提示可以被用户关闭，但相关记录仍需保留给 App。
    
典型场景：

*   **展示切换**：用户默认展示名片，向下翻到二维码，再翻到活动页。
    
*   **动态回看**：刚刚收到几条会议提示或 Agent 结果，用户可通过翻页回看最近几条。
    
*   **恢复展示态**：用户完成闪念或查看反馈后，一键清空动态内容，回到名片或二维码展示。
    

## 5. 音频与麦克风能力预期

Eureka-3.0 的音频能力不只服务“录音文件”，也服务后续的 Notes、Contacts、Reminders、Ask Agent 和会议 Copilot。硬件侧需要提供稳定、可理解的音频入口和麦克风来源，复杂语义处理交给 App/Agent。

### 5.1 闪念音频

闪念是短音频输入。

预期说明：

1.  闪念录音需要尽快启动，降低用户等待感。
    
2.  闪念结束后应明确反馈处理状态。
    
3.  闪念离线暂存时，应避免用户误解为已经保存到 App。
    
4.  暂存容量不足时，应阻止继续录入并提示用户连接 App。
    
内容场景：

*   **记录事实**：例如“客户说预算要等财务确认”。
    
*   **记录待办**：例如“明天上午给 Kevin 发合同附件”。
    
*   **记录关系上下文**：例如“Kevin 喜欢先看数据，不喜欢太销售化的表达”。
    
*   **触发查询**：例如“查一下我上次和 Kevin 见面是什么时候”。这类内容可能进入 Ask Agent，而不是普通 Note。
    

### 5.2 会议音频

会议是长时间录音。

预期说明：

1.  蓝牙连接中断不应中断会议录音，除非底层音频链路本身不可用。
    
2.  会议录音应支持离线持续录制。
    
3.  恢复在线后，历史音频需要同步到 App。
    
4.  在线实时增强不是会议录音成功的前置条件。
    
内容场景：

*   **完整转写**：会议结束后同步到 App，生成转写、总结和关键行动项。
    
*   **实时辅助**：会议中 App 可推送客户背景、议题提醒、异议应对或下一步建议到 E-ink。
    
*   **会议打点**：如果硬件形态支持，录音中可通过额外操作写入时间戳标记，用于会后快速定位重点片段。
    
*   **后续沉淀**：会议结果最终应回到 BizCard 的联系人、笔记、提醒和 Ask Agent 会话中，硬件只作为输入入口和反馈载体。
    

### 5.3 麦克风自动识别与切换

硬件应支持环境麦、骨传导麦、耳机麦的识别与切换。

预期说明：

1.  会议入口默认使用环境麦；耳机模式可用时，使用耳机麦。
    
2.  闪念入口可优先使用骨传导麦；骨传导信号不可用时，降级到环境麦。
    
3.  如果耳机麦被定义为强接管，则闪念和会议都优先使用耳机麦。
    
4.  自动切换结果需要在 E-ink 或 App 中可见。
    
5.  麦克风切换只影响音频输入源，不改变闪念和会议的入口规则。
    
典型场景：

*   **环境麦**：适合桌面会议、多人讨论、展位交流等需要收录外部声音的场景。
    
*   **骨传导麦**：适合用户低声记录自己的闪念，尤其是在嘈杂环境中更强调个人语音输入。
    
*   **耳机麦**：适合用户已经处在耳机通话、远程会议或希望更私密录音的场景。若产品定义为耳机强接管，则硬件和 App 都应让用户明确知道当前使用的是耳机麦。
    
*   **自动切换反馈**：用户不需要理解底层路由，但需要知道“当前收的是哪个麦”，避免录完后才发现声音来源不符合预期。
    

## 6. NFC 能力预期

NFC 是 Eureka-3.0 作为商务硬件卡片的重要扩展能力。它既可以服务对外展示和名片交换，也可以作为 App 或硬件动作的近场触发入口。本文先描述能力预期，不承诺所有高安全场景都在首版完成。

### 6.1 NFC Profile

硬件应支持可配置的 NFC Profile。

目标能力包括：

*   数字名片链接。
    
*   个人主页。
    
*   App deep link。
    
*   门禁卡。
    
*   车钥匙。
    

预期说明：

1.  名片链接、个人主页、App deep link 优先级更高，适合作为首批能力。
    
2.  门禁卡、车钥匙涉及安全芯片、系统权限或第三方生态，不应默认承诺全量支持。
    
典型场景：

*   **数字名片**：对方手机靠近 Eureka 后打开用户的个人主页或名片链接。
    
*   **活动主页**：展会或活动期间，NFC 指向特定活动页、预约页或资料下载页。
    
*   **身份切换**：用户可在 App 中配置不同 NFC Profile，例如商务名片、招聘主页、个人作品集。
    
*   **高安全扩展**：门禁卡和车钥匙可以作为硬件能力评估方向，但需要单独确认安全芯片、系统权限、生态合作和合规边界。
    

### 6.2 NFC Action

NFC 可触发基础动作。

目标能力包括：

*   Tap 后打开 App 指定页面。
    
*   Tap 后打开录音准备页。
    
*   Tap 后开始会议录音。
    
*   Tap 后打开名片或个人主页。
    

预期说明：

1.  如果动作可由硬件本地完成，硬件可以直接执行并显示本地状态。
    
2.  如果动作依赖 App，则通过 App 完成后续处理。
    
3.  NFC 触发录音时，后续仍按会议入口规则处理。
    
4.  NFC 不改变闪念入口的长按松开规则。
    
典型场景：

*   **靠近打开名片页**：对方手机 Tap Eureka 后直接打开用户名片或个人主页。
    
*   **靠近进入 App 指定页面**：用户自己的手机 Tap Eureka 后打开硬件管理页、录音页或某个联系人页面。
    
*   **靠近触发会议准备**：在固定会议室或特定场景中，NFC 可触发 App 进入会议录音准备态，后续是否开始录音仍遵循硬件或 App 的录音入口规则。
    
*   **靠近切换展示**：未来可评估通过 NFC 切换某个展示 Profile，例如从个人名片切换到活动二维码。
    

## 7. 本地存储与缓存能力预期

硬件需要具备必要的本地存储能力，但不作为长期数据仓库。

需要支持：

*   默认展示内容缓存。
    
*   最近动态内容缓存。
    
*   离线闪念暂存。
    
*   离线会议录音。
    
*   固件本地状态页。
    

预期说明：

1.  默认展示内容应能支持离线翻页。
    
2.  最近动态内容应能支持短期回看。
    
3.  离线闪念和会议录音应分别管理，不互相挤占关键空间。
    
4.  容量不足时，硬件必须给出明确提示。
    
5.  已确认同步的内容应允许清理。
    
场景预期：

*   **无手机展示**：即使 App 不在前台或手机暂时不在身边，Eureka 仍能展示已缓存的名片、二维码或固定展示页。
    
*   **离线闪念**：用户在离线状态下记录短语音，硬件暂存后等待下次连接 App 处理。
    
*   **离线会议**：会议过程中不能依赖网络或 App 实时可用，硬件需要承担基础录音保存能力。
    
*   **短期回看**：用户可在硬件上回看最近几条动态反馈，但长期历史、搜索、编辑和删除应回到 App 中完成。
    

## 8. App 协同预期

硬件端需要与 App 形成稳定协同。

App 需要能完成：

*   配置默认展示内容。
    
*   查看当前屏幕状态。
    
*   查看硬件连接、电量、存储和固件状态。
    
*   查看离线待同步内容。
    
*   配置 NFC Profile 和 NFC Action。
    
*   查看麦克风来源和切换记录。
    
*   接收硬件入口产生的闪念、会议、Ask Agent 请求。
    
*   下发 App 渲染后的 E-ink 内容。
    

硬件需要能完成：

*   接收 App 下发的显示内容。
    
*   上报本地状态和用户操作。
    
*   在离线时保留关键内容。
    
*   在恢复在线后配合同步。
    
*   在 App 不可用时展示本地状态。
    

协同场景：

*   **App 配置，硬件展示**：用户在 App 中选择默认展示页、二维码、NFC Profile 和展示顺序，硬件负责离线常显。
    
*   **硬件触发，App 处理**：用户通过硬件录入闪念、会议或 Ask Agent 请求，App 负责 ASR、Agent、保存和业务结果生成。
    
*   **App 渲染，硬件反显**：复杂内容由 App 渲染成适合 E-ink 的画面，硬件不做复杂排版。
    
*   **离线暂存，在线补处理**：硬件在离线时保留输入和状态，恢复在线后由 App 继续完成处理，并把结果同步回硬件屏幕。
    
*   **统一资产沉淀**：硬件产生的输入最终应尽量进入 BizCard 的 Notes、Contacts、Reminders、Meetings 和 Ask Agent Thread，而不是停留在孤立硬件日志中。
    

## 9. 典型端到端场景

### 9.1 商务展示与交换

用户参加展会或客户拜访前，在 App 中设置 Eureka 默认展示为个人名片、二维码和活动页。到现场后，Eureka 平时常显二维码；对方扫码或 NFC Tap 后打开个人主页。若后续发生名片交换或对方保存成功，E-ink 可以短暂显示“已交换/已保存”的轻反馈，然后回到默认展示。

这个场景中，硬件的价值是低功耗常显和近场触达；App 的价值是配置展示内容、管理联系人和沉淀后续关系。

### 9.2 闪念捕捉与提醒创建

用户在会后想到一个跟进点，长按 Eureka 说“明天下午三点提醒我给 Kevin 发报价”，松开后结束输入。在线时，App 处理语音并创建 Reminder，E-ink 显示“已创建提醒”；离线时，硬件显示“已暂存，连接 App 后同步”。

这个场景中，用户关注的是快速记录和确定性反馈，不关注底层是否立即完成 ASR 或 Agent 处理。

### 9.3 会议录音与低打扰辅助

用户进入客户会议，点击硬件开始录音。会议中，E-ink 显示录音状态、当前麦克风来源和必要提示。在线增强开启时，App 可下发 Copilot 提词，例如客户背景、关键问题、下一步建议；离线时，硬件继续承担基础录音，增强能力暂停。

这个场景中，会议录音是基础能力，实时提示是增强能力。硬件需要让用户确信“会议正在被记录”，同时不让屏幕提示打断会面。

### 9.4 耳机/骨传导/环境麦切换

用户在嘈杂展会中记录个人闪念时，设备可优先使用骨传导麦；用户在桌面会议中录音时，默认使用环境麦；用户已经连接耳机并进入耳机录音模式时，系统可优先使用耳机麦。E-ink 或 App 需要让用户看到当前麦克风来源。

这个场景中，麦克风切换的目标不是增加用户操作复杂度，而是让不同场景下的音频输入更可靠。

### 9.5 离线使用与恢复同步

用户全天参加活动，手机可能不在身边或 App 没有保持连接。Eureka 仍可展示已缓存名片，记录短闪念或会议录音，并显示待同步状态。活动结束后，用户打开 App，离线内容同步进入 App，完成识别、总结、提醒创建和屏幕结果回推。

这个场景中，硬件不需要独立完成复杂业务，但必须让用户知道哪些内容只是暂存，哪些内容已经完成处理。

## 10. 功能方向对齐点

以下内容用于后续和硬件、固件、App、算法、业务团队继续对齐功能预期，不涉及具体实现方案或验收标准：

1.  Eureka 的核心定位是否统一为“商务上下文第二屏 + 轻量输入入口”，而不是独立智能终端。
    
2.  默认展示内容需要覆盖哪些商务场景：个人名片、二维码、个人主页、活动页、今日简报或其他固定展示页。
    
3.  Dynamic 反馈需要覆盖哪些高价值内容：闪念结果、Ask Agent 摘要、会议提示、Reminder、NFC 触发结果、麦克风来源变化。
    
4.  闪念、Ask Agent、会议录音之间的用户心智是否足够清晰：短输入用于快速记录和查询，会议入口用于持续录音和会议辅助。
    
5.  会议场景中，E-ink 应优先展示哪些内容：录音状态、客户背景、实时提词、关键问题、下一步建议或会后跟进。
    
6.  NFC 更适合优先服务哪些商务动作：名片链接、个人主页、App deep link、会议准备、展示页切换，门禁/车钥匙作为后续高安全能力单独评估。
    
7.  麦克风来源需要支持哪些用户场景：桌面会议的环境麦、嘈杂环境的骨传导麦、耳机模式下的耳机麦，以及用户可见的来源提示。
    
8.  离线能力需要给用户怎样的确定性：能展示已缓存内容、能暂存输入、能让用户知道内容尚未完成 App 侧处理。
    
9.  硬件产生的输入最终应如何回到 BizCard 的 Notes、Contacts、Reminders、Meetings 和 Ask Agent Thread，避免形成孤立硬件记录。
