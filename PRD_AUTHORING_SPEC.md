# PRD 编写规格（PRD Authoring Spec）

> **文档性质**：Meta-Spec —— 约束 PM 与 AI 协作产出 PRD 的过程、结构与质量标准。  
> **适用范围**：所有 BizCard 产品需求文档的创建与定稿。  
> **设计理念**：Spec Engineering —— 通过双 Agent 对抗式协作，将非结构化讨论收敛为可约束的产品文档。

---

## 1. 核心原则

1. **讨论即输入，文档即输出**：对话中的每一个已确认结论必须落入文档；未落入文档的口头共识视为不存在。
2. **PRD 即 Spec**：对于单一功能，PRD 就是最终规格，不需要额外产出 SPEC 文档。PRD 写好了就是 spec。
3. **对抗出质量**：一个 Agent 负责写，另一个负责挑。PM 居中判断。经过多轮 debate 自然收敛为定稿。
4. **可测试性**：PRD 中的每条规则都应可转化为至少一条验收用例。无法测试的描述要么不够具体，要么不属于 PRD。
5. **渐进式精度**：允许从粗到细迭代，但每轮产物必须明确标注当前状态（草案 / 评审中 / 已定稿）。

---

## 2. 协作流程：双 Agent 对抗收敛

### 2.1 角色定义

| 角色 | 职责 | 立场 |
|------|------|------|
| **PM** | 提供需求输入、判断决策、裁决分歧 | 最终决策者 |
| **Agent 1（Proposal）** | 与 PM 讨论需求、输出 Proposal / PRD 草案、根据 Review 修改文档 | 建设性：尽量完善方案 |
| **Agent 2（Review）** | 逐章审查 Agent 1 的产出、发现问题、提出修改建议 | 批判性：尽量找出漏洞 |

### 2.2 流程

```
PM 提出需求
    ↓
Agent 1 × PM：讨论 → 输出 Proposal
    ↓
Agent 2：审查 Proposal → 输出 Review
    ↓
Agent 1：根据 Review 修改 → 输出 v2
    ↓
Agent 2：再审 → 输出 Review v2
    ↓
  … 重复直到 PM 判断收敛 …
    ↓
PM 确认定稿 → PRD（已定稿）
```

### 2.3 收敛判断

PM 在以下条件满足时可宣布定稿：

- Agent 2 最新一轮 Review 中无 Blocker 级 finding
- 连续两轮 Review 未产生新的 Major finding
- PM 对核心流程和规则无额外补充

### 2.4 轮次建议

| 功能复杂度 | 典型轮次 | 说明 |
|-----------|---------|------|
| 简单（单页面、少状态） | 1-2 轮 | Agent 1 直出草案，Agent 2 快速扫一遍 |
| 中等（多状态、有异常分支） | 2-3 轮 | 第一轮定结构，第二轮补边界，第三轮收敛术语 |
| 复杂（多页面、跨模块、硬件联动） | 3-5 轮 | 前两轮定核心架构，后续轮次逐步收紧异常和门禁 |

## 3. 文档分类与命名

### 3.1 文档类型

| 类型 | 命名格式 | 用途 |
|------|----------|------|
| **PRD** | `PRD_<主题>.md` | 功能的完整需求定义（= 最终规格） |
| **Review** | `<主题>_review.md` | Agent 2 的评审记录（过程文档） |
| **Demo** | `<主题>-demo.html` | 交互原型验证 |

> **不再单独产出 SPEC**。PRD 定稿后即为实现规格。如果历史原因已有散落的过程文档需整合，可一次性合入 PRD，合并后过程文档降级为归档参考。

### 3.2 目录结构

```
feature-module/
  PRD_<主题>.md           ← 需求文档（= 规格终稿）
  <主题>-demo.html        ← 交互原型（如有）
  <主题>_review.md        ← 评审记录（过程文档）
```

复杂功能可建 `reviews/` 子目录存放多轮评审记录。

## 4. PRD 结构模板

### 4.1 必选章节

```markdown
# PRD：<标题>

| 属性 | 内容 |
|------|------|
| 状态 | 草案 / 评审中 / 已定稿 |
| 版本 | v1.0 |
| 目标 | 一句话描述本 PRD 要解决的核心问题 |
| 关联文档 | 列出有依赖关系的其他文档 |

---

## 1. 目标与非目标

### 1.1 目标
（本 PRD 要实现什么）

### 1.2 非目标
（明确不在本 PRD 范围内的内容，减少歧义）

---

## 2. 核心模型 / 信息架构
（实体关系、数据结构、页面结构等）

---

## 3. 用户流程
（主路径 + 分支路径，建议用 Mermaid 流程图）

---

## 4. 状态与规则
（状态机、门禁、前置条件等，用表格）

---

## 5. 异常处理
（异常场景 + 策略 + 用户可见表现）

---

## 6. 验收标准
（可测试的条目，可直接转化为测试用例）

---

## N. 修订记录

| 版本 | 日期 | 说明 |
|------|------|------|
```

### 4.2 可选章节（按需添加）

| 章节 | 适用场景 |
|------|----------|
| **线框** | UI 密集型功能，用 ASCII 或 Mermaid 描述关键页面布局与状态 |
| **与现有功能的差异** | 迭代型功能（v2 vs v1），明确哪些行为变了、哪些不变 |

### 4.3 格式规范

| 规则 | 说明 |
|------|------|
| **表格优先** | 规则、状态、条件映射用表格，不用长段落 |
| **Mermaid 流程图** | 用户流程、状态机、页面路由用 Mermaid，不用纯文字描述 |
| **一条规则一行** | 避免在同一段落中混合多条互相独立的规则 |
| **结论前置** | 每个小节先给结论/规则，再解释原因 |
| **避免模糊词** | 禁止使用「适当」「可能」「某些情况下」——要么给出具体条件，要么标注为「待确认」 |

---

## 5. 质量门禁

### 5.1 Agent 1 输出 Proposal 后（可进入 Review）

| # | 检查项 | 标准 |
|---|--------|------|
| 1 | 目标与非目标 | 各至少一条，且不重叠 |
| 2 | 主流程 | 有完整的用户从入口到终态的路径描述 |
| 3 | 状态定义 | 核心实体的所有状态已列出 |
| 4 | 门禁规则 | 主要动作的前置条件已定义 |
| 5 | 异常覆盖 | 至少覆盖：网络失败、中途退出、并发冲突三类 |
| 6 | 无残留讨论 | 文档中不含「待讨论」「A or B」等未收敛表述 |

### 5.2 多轮 Debate 后（可宣布定稿）

| # | 检查项 | 标准 |
|---|--------|------|
| 1 | 阻塞项 | Agent 2 最新 Review 中无 Blocker |
| 2 | 一致性 | 同一口径在文档内部无冲突 |
| 3 | 可测性 | 每条规则可对应至少一条验收标准 |
| 4 | 术语 | 全文无同义混用 |
| 5 | 修订记录 | 包含从草案到定稿的所有变更摘要 |

---

## 6. Agent 1 行为约束（Proposal Agent）

### 6.1 讨论阶段

- **主动追问**：遇到模糊需求时，先提问再动笔，不基于假设输出大段内容。
- **结构化提问**：一次最多问 2-3 个关键问题，避免信息过载。
- **方案对比**：有多种可行方案时，用表格对比利弊，不替 PM 做选择。

### 6.2 输出阶段

- **不编造需求**：只输出讨论中确认的内容。不确定的标注「待确认」，不用自己觉得合理的内容填充。
- **结论前置**：先给结论或规则，再解释原因。不写「经过分析，我们发现…因此建议…」式的论证过程。
- **表格优先**：规则、映射、条件判断用表格。长段落只用于解释「为什么」，不用于定义「是什么」。

### 6.3 回应 Review

- 对 Blocker：必须修改，不可搁置。
- 对 Major：修改或给出不修改的理由，由 PM 裁决。
- 对 Minor：可采纳可不采纳，简要说明即可。
- 修改后更新版本号和修订记录。

### 6.4 禁止事项

- 禁止在 PRD 正文中写探索性讨论过程。
- 禁止使用「我觉得」「可能」「大概」。
- 禁止在同一份 PRD 中保留两个互斥方案而不标注哪个已拍板。
- 禁止修改文档时不更新修订记录。

---

## 7. Agent 2 行为约束（Review Agent）

### 7.1 审查维度

| 维度 | 检查内容 |
|------|----------|
| **完备性** | 主流程是否闭环？是否有入口无出口的状态？ |
| **一致性** | 同一口径在不同章节是否一致？ |
| **可测性** | 能否从规则直接写出测试用例？ |
| **边界** | 是否覆盖了空态、极端值、并发场景？ |
| **术语** | 有无同义混用或歧义命名？ |
| **语气** | 是否仍残留「倾向」「建议采用」等探索性表述？ |

### 7.2 输出格式

每轮 Review 按严重性分级：

```markdown
## 第 N 轮 Review

### Blocker（阻塞）
- [ ] Finding 描述 + 建议处理方式

### Major（重要）
- [ ] Finding 描述

### Minor（建议）
- [ ] Finding 描述
```

### 7.3 Review 规则

- 逐章检查，不跳过任何章节。
- 每个 Finding 附带具体修改建议，不只说「这里有问题」。
- Blocker 必须在下一版中关闭。
- 同一 Finding 连续两轮未关闭 → 升级为 Blocker。

---

## 8. 异常与边界分析框架

PRD 中的异常处理章节应至少覆盖以下维度：

| 维度 | 典型问题 |
|------|----------|
| **连接/网络** | 中途断网、超时、重连 |
| **状态不同步** | App 前后台切换、多端同时操作 |
| **数据完整性** | 必填字段缺失、关联对象被删除 |
| **并发** | 同一资源多人操作、同一设备多 App 操作 |
| **权限/门禁** | 不满足前置条件时的拦截与引导 |
| **硬件/环境** | 蓝牙断开、电池耗尽、系统权限被关闭 |

每条异常需定义：

```
触发条件 → 系统行为 → 用户可见表现 → 恢复路径
```

## 9. Demo 规范

### 9.1 何时需要 Demo

- UI 交互复杂度高（多状态、多页面跳转）
- 状态机有 5 个以上状态
- PRD 描述了「但看不出来长什么样」的交互

### 9.2 Demo 技术要求

| 要求 | 说明 |
|------|------|
| **单文件** | 一个 `.html` 文件，无外部依赖（除 Google Fonts 可选） |
| **状态切换** | 侧边面板列出所有状态，可一键跳转 |
| **交互可操作** | 核心流程可在 Demo 内点击走通 |
| **iPhone 仿真** | 使用 iPhone 尺寸外框，模拟真实设备观感 |
| **异常态覆盖** | 不仅展示正常流程，还展示关键异常状态 |

### 9.3 Demo 与 PRD 的关系

- Demo 是 PRD 的**可视化补充**，不替代文档
- 如 Demo 与 PRD 冲突，以 PRD 为准

## 10. 术语治理

### 10.1 何时需要术语表

- 引入了产品特有概念（如「Eureka-Audio」「闪念」）
- 同一概念有多种叫法风险
- 跨团队协作

### 10.2 格式

```markdown
| 使用 | 避免混用 | 说明 |
|------|----------|------|
| 设备控制台 | 设备详情 / 硬件管理页 | 页面命名 |
```

### 10.3 执行

术语表一旦定义，全文统一执行。Agent 2 在 Review 时将术语不一致作为 Finding 提出。
