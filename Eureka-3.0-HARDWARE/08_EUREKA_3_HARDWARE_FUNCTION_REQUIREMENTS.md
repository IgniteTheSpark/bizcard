# 08 Eureka-3.0 硬件功能需求

> 本文面向 Eureka-3.0 硬件与嵌入式开发对齐，定义硬件端需要支持的功能、状态和软硬件协同边界。本文不定义 App、固件或云端的数据字段、表结构、协议细节；具体实现由后续技术方案和联调文档展开。

## 1. 产品范围

Eureka-3.0 硬件是一张具备 E-ink 显示、录音输入、蓝牙连接、NFC、麦克风自动切换和本地状态反馈能力的商务硬件卡片。

硬件端的核心任务不是独立完成复杂 AI 处理，而是：

1. 采集用户输入：闪念、会议录音、NFC 触发、按键操作。
2. 维护本地状态：录音中、离线暂存、低电量、麦克风来源、NFC 模式等。
3. 承载低功耗展示：默认展示页、动态反馈页、硬件本地状态页。
4. 与 App 协同：在线时上传输入并接收 App 渲染内容；离线时本地暂存，恢复在线后同步。

## 2. 硬件功能模块

| 模块 | 核心能力 | 说明 |
|------|----------|------|
| E-ink 显示 | 常显、翻页、显示 App 反显内容、本地状态页 | 不承担复杂排版；App 在线时以 App 渲染内容为主，离线或关键硬件状态时显示本地状态 |
| 物理按钮 | 闪念长按、会议开始/结束、上下翻页、清空/返回 | 入口行为必须稳定、可盲操、反馈明确 |
| 音频采集 | 闪念短录音、会议长录音、离线录音暂存 | 闪念和会议的录音逻辑不同，不共用同一套交互 |
| 麦克风系统 | 环境麦、骨传导麦、耳机麦识别与切换 | 切换结果需要对用户可见 |
| App 连接 | 与 App 协同、状态同步、音频/内容同步、E-ink 图片下发 | 作为在线协同前提，不单独展开连接管理 |
| NFC | 名片链接、个人主页、App 动作触发、扩展门禁/车钥匙能力 | 门禁和车钥匙属于高安全/生态能力，后续单独评估 |
| 本地存储 | Base 图片缓存、Dynamic 最近内容、离线闪念、会议录音 | 硬件不做长期数据仓库，只保留必要内容 |
| 电源与状态 | 电量、低电提醒、充电状态、异常状态 | 关键状态必须能在 E-ink 上反馈 |

## 3. E-ink 显示需求

### 3.0 屏幕区域

E-ink 需要同时承载用户长期展示内容和硬件状态。为避免用户 Profile、二维码等长显内容被蓝牙、电量、同步状态频繁打断，屏幕建议采用“状态区 + 主内容区”的展示结构。

建议：

- 顶部保留常驻状态区，用于展示蓝牙、电量、同步、录音、麦克风等少量状态。
- 下方作为主内容区，用于展示 Profile、二维码、Agent 结果、会议提示等内容。
- 状态区与主内容区的具体比例不在本文中写死，后续由 ID、UI 和屏幕尺寸共同确定。
- 默认展示名片或二维码时，状态区应尽量轻量，不影响主内容识别。
- 录音中、低电量、离线暂存、存储满等关键状态，可以扩大状态展示，必要时临时切换为整屏状态页。

状态区至少应支持展示：

- 蓝牙状态。
- 电量状态。
- 同步中或待同步状态。
- 录音中状态。
- 当前麦克风来源。
- NFC 触发或当前 NFC 模式。

### 3.1 默认展示

硬件应支持一组默认展示内容，用于无事件发生时常显。

首版默认展示至少包括：

- 我的名片或 Profile。
- 二维码或个人链接。
- 用户在 App 中设置的默认页。

需求约束：

1. 用户可通过 App 配置默认展示内容。
2. 硬件应可在离线时展示已缓存的默认内容。
3. 无动态内容时，上下翻页只在默认展示内容中循环。
4. 用户清空动态内容后，应回到默认展示态。
5. 默认展示内容需要预留状态区，避免蓝牙、电量等状态遮挡 Profile 或二维码。

### 3.2 动态反馈

硬件应支持 App 下发的动态反馈内容。

典型动态反馈包括：

- 闪念处理结果。
- Ask Agent 结果摘要。
- 录音状态变化。
- 会议 Copilot 或实时翻译内容。
- Reminder 或日程提醒。
- NFC 触发结果。
- 麦克风来源变化。

需求约束：

1. 动态反馈可以临时覆盖默认展示内容。
2. 存在动态反馈时，上下翻页优先在动态内容中回看。
3. 清空/返回操作只清空屏幕上的动态内容，不删除 App 历史或硬件待同步内容。
4. 动态内容过期后，应回到默认展示态。

### 3.3 基础硬件状态

硬件状态只展示首版必要的基础状态，用于让用户知道设备当前在做什么。这里不展开复杂异常分支和失败态。

至少需要覆盖：

| 状态类型 | 典型内容 | 展示方式 |
|----------|----------|----------|
| 闪念状态 | 记录中、结束反馈 | 记录中可在状态区展示；结束后短暂反显 |
| 录音状态 | 录音中、录音标记反馈 | 状态区常驻，标记反馈短暂反显 |
| 蓝牙状态 | 未连接、已连接手机、已连接手机 + 耳机 | 状态区轻量展示 |
| 文件同步 | 同步文件中 | 状态区轻量展示 |
| 电源状态 | 电量百分比 | 状态区常驻 |
| 麦克风状态 | 环境麦、骨传导、耳机麦 | 状态区或录音状态中展示 |
| NFC 状态 | 当前 NFC 模式 | 状态区或短暂提示 |

需求约束：

1. E-ink 同一时刻只显示一张完整画面。
2. App 反显内容和硬件本地状态页不做复杂叠加。
3. 关键硬件状态可以临时覆盖 App 反显内容。
4. App 恢复在线后，应能重新同步当前屏幕状态。
5. 本地状态页可以沿用状态区和主内容区结构，也可以在关键状态需要强调时使用整屏提示。
6. 蓝牙连接/断开只作为用户可见状态和在线协同前提，不在本文中展开连接管理功能。

## 4. 物理输入需求

### 4.1 闪念按钮

闪念入口采用长按输入、松开结束。

需求约束：

1. 用户长按按钮时开始录制闪念。
2. 用户松开按钮时结束本次闪念。
3. 闪念只处理松开后形成的完整音频。
4. 闪念不做录制过程中的实时传输、实时 ASR 或实时 Agent 处理。
5. 松开时在线，则立即上传 App 处理，并在 E-ink 上展示处理反馈。
6. 松开时离线，则本地暂存，并在 E-ink 上展示“已暂存，连接 App 后同步”。
7. 录制过程中连接状态变化不改变本次闪念的处理方式，只在松开时判断在线或离线。

### 4.2 会议按钮

会议入口采用点击开始、再次点击结束。

需求约束：

1. 用户点击会议按钮后开始会议录音。
2. 用户再次点击后结束会议录音。
3. 离线时，硬件继续本地会议录音，并展示基础录音状态。
4. 在线且用户开启实时增强时，硬件可将后续音频传给 App，用于 Copilot、实时翻译或会议提示。
5. 在线转离线时，实时增强停止，但会议录音继续。
6. 离线转在线时，后续音频可开始实时传输；离线阶段的历史音频在会后或恢复在线后同步。

### 4.3 翻页与清空

硬件应支持上下翻页和清空/返回操作。

需求约束：

1. 无动态内容时，上下翻页切换默认展示页。
2. 有动态内容时，上下翻页回看最近动态内容。
3. 清空/返回只影响屏幕显示，不删除录音、闪念、同步队列或 App 历史。
4. 关键异常状态可以被用户关闭提示，但异常记录仍需保留给 App。

## 5. 音频与麦克风需求

### 5.1 闪念音频

闪念是短音频输入。

需求约束：

1. 闪念录音需要尽快启动，降低用户等待感。
2. 闪念结束后应明确反馈处理状态。
3. 闪念离线暂存时，应避免用户误解为已经保存到 App。
4. 暂存容量不足时，应阻止继续录入并提示用户连接 App。

### 5.2 会议音频

会议是长时间录音。

需求约束：

1. 蓝牙连接中断不应中断会议录音，除非底层音频链路本身不可用。
2. 会议录音应支持离线持续录制。
3. 恢复在线后，历史音频需要同步到 App。
4. 在线实时增强不是会议录音成功的前置条件。

### 5.3 麦克风自动识别与切换

硬件应支持环境麦、骨传导麦、耳机麦的识别与切换。

需求约束：

1. 会议入口默认使用环境麦；耳机模式可用时，使用耳机麦。
2. 闪念入口可优先使用骨传导麦；骨传导信号不可用时，降级到环境麦。
3. 如果耳机麦被定义为强接管，则闪念和会议都优先使用耳机麦。
4. 自动切换结果需要在 E-ink 或 App 中可见。
5. 麦克风切换只影响音频输入源，不改变闪念和会议的入口规则。

## 6. NFC 需求

### 6.1 NFC Profile

硬件应支持可配置的 NFC Profile。

目标能力包括：

- 数字名片链接。
- 个人主页。
- App deep link。
- 门禁卡。
- 车钥匙。

需求约束：

1. 名片链接、个人主页、App deep link 优先级更高，适合作为首批能力。
2. 门禁卡、车钥匙涉及安全芯片、系统权限或第三方生态，不应默认承诺全量支持。
3. 用户应能在 App 中看到当前 NFC 模式。
4. NFC 模式切换成功或失败时，硬件应能给出反馈。

### 6.2 NFC Action

NFC 可触发基础动作。

目标能力包括：

- Tap 后打开 App 指定页面。
- Tap 后打开录音准备页。
- Tap 后开始会议录音。
- Tap 后打开名片或个人主页。

需求约束：

1. 如果动作可由硬件本地完成，硬件可以直接执行并显示本地状态。
2. 如果动作依赖 App，则通过 App 完成后续处理。
3. NFC 触发录音时，后续仍按会议入口规则处理。
4. NFC 不改变闪念入口的长按松开规则。

## 7. 本地存储与缓存需求

硬件需要具备必要的本地存储能力，但不作为长期数据仓库。

需要支持：

- 默认展示内容缓存。
- 最近动态内容缓存。
- 离线闪念暂存。
- 离线会议录音。
- 固件本地状态页。

需求约束：

1. 默认展示内容应能支持离线翻页。
2. 最近动态内容应能支持短期回看。
3. 离线闪念和会议录音应分别管理，不互相挤占关键空间。
4. 容量不足时，硬件必须给出明确提示。
5. 已确认同步的内容应允许清理。

## 8. App 协同需求

硬件端需要与 App 形成稳定协同。

App 需要能完成：

- 配置默认展示内容。
- 查看当前屏幕状态。
- 查看硬件连接、电量、存储和固件状态。
- 查看离线待同步内容。
- 配置 NFC Profile 和 NFC Action。
- 查看麦克风来源和切换记录。
- 接收硬件入口产生的闪念、会议、Ask Agent 请求。
- 下发 App 渲染后的 E-ink 内容。

硬件需要能完成：

- 接收 App 下发的显示内容。
- 上报本地状态和用户操作。
- 在离线时保留关键内容。
- 在恢复在线后配合同步。
- 在 App 不可用时展示本地状态。

## 9. 分期建议

### P0：基础闭环

- E-ink 默认展示与动态反馈。
- E-ink 状态区与主内容区的基础展示结构。
- 闪念长按录入、松开结束。
- 会议点击开始、再次点击结束。
- 离线闪念暂存。
- 会议离线录音。
- App 渲染内容与固件本地状态页的切换边界。
- 基础录音状态、蓝牙状态、闪念反馈和异常提示。

### P1：协同增强

- 会议实时增强：Copilot、实时翻译、会议提示。
- 麦克风自动识别和切换反馈。
- NFC 名片链接、个人主页、App deep link。
- NFC 触发基础动作，例如打开录音或打开 App 页面。
- App 硬件管理页展示当前屏幕、离线同步、NFC、麦克风来源。

### P2：高级硬件能力

- 门禁卡、车钥匙等高安全 NFC Profile。
- 更智能的屏幕调度。
- 更丰富的 E-ink 模板。
- 多设备或多场景同步策略。
- 更完整的异常恢复和诊断能力。

## 10. 待确认问题

以下问题需要在硬件、固件、App 和后端进入实现前进一步确认：

1. E-ink 分辨率、刷新速度、全刷/局刷能力和功耗边界。
2. 本地可缓存的默认展示数量、动态内容数量、闪念数量和会议录音容量。
3. 闪念与会议按钮的物理形态、触发阈值、防误触策略。
4. 蓝牙状态、同步失败、传输失败时的用户可见反馈。
5. 环境麦、骨传导麦、耳机麦的检测能力和切换优先级。
6. NFC 支持范围，尤其门禁卡和车钥匙的安全与生态限制。
7. App 反显内容和固件本地状态页的切换恢复规则。
8. 录音、蓝牙、同步、电量、麦克风、NFC 等基础状态的展示优先级。
