# 06 硬件交互流与缓存管理机制 (Hardware Interaction Flow)

随着软硬协同方案的深入，我们对 E-ink 硬件的“语音输入 -> App 处理 -> E-ink 呈现”这一核心链路进行了详细的梳理与收敛。

## 1. 统一入口：All-in-One 路由至 Ask Agent
为了避免在 App 端维护多套 NLP 解析逻辑，我们将硬件传来的所有语音文本**统一透传给后端的 Ask Agent**，由 Agent 的意图识别与 Tools 调用能力来决定最终动作。

### 统一处理的优势
- **去中心化判断**：App 端不再需要硬编码正则表达式或关键词，Agent 会利用大模型自行判断这句话是需要“新建一个 Note”、“添加一个 Reminder”还是“执行查询/发邮件等 CRUD 操作”。
- **一致的正向反馈闭环**：无论 Agent 执行了什么操作，**必须**返回一个结果状态给到 E-ink 屏幕。
  - 如果是存闪念：E-ink 刷屏显示 `[✅ 已记录] 拓展 B2B 市场...`
  - 如果是加待办：E-ink 刷屏显示 `[⏰ 待办已加] 明天下午2点开会`
  - 如果是查人脉：E-ink 刷屏显示 `[👤 冯建国] CEO, 138-xxxx...`
  
这种设计确保了用户在完全不掏出手机的情况下，也能获得极强的“确定感”与操作闭环。

## 2. 屏幕呈现的“双轨循环 (Dual-Loop)”缓存管理逻辑
硬件设备受限于仅有的“上下翻页”和少量物理按键，如何在有限的交互下管理海量的展示内容是体验的关键。我们引入了**“双轨循环 (Static Base Loop vs Dynamic Overlay Loop)”**机制。

### 逻辑一：Base Loop (基础静态循环)
- **内容来源**：用户在手机 App 端主动配置并预先下发到卡片中的图片。
- **典型内容**：个人数字名片、微信二维码、收款码、或者固定的日程 Dashboard。
- **交互**：在平时待机状态下，用户按上下翻页键，仅在这几张基础图片中循环切换。

### 逻辑二：Dynamic Overlay Loop (动态覆盖循环)
- **内容来源**：由 App 端动态推送过来的即时性内容（即 Ask Agent 处理完成后的正向反馈，或是会议中 Copilot 实时提取的提词卡片）。
- **缓存机制**：硬件接收到动态图片后，将其压入一个临时的内存栈（Dynamic Cache）中，并立即切换显示。
- **交互演变**：
  - 一旦存在动态图片，硬件的“上下翻页键”将**被劫持**，此时翻页只会在“刚刚收到的这几张动态反馈图”中切换（例如会议中连续收到了 3 条 Copilot 提示，翻页就在这 3 条中流转）。
  - **一键清空 (Clear Action)**：用户通过一个特殊的物理操作（例如：**长按翻页键 2 秒**，或提供一个**专属的 Dismiss 按键**），可以一键清空当前硬件里的 Dynamic Cache。
  - 清空后，屏幕立刻回退并固定在 **Base Loop (基础静态循环)** 的第一张（如个人名片）。

**总结**：这保证了临时性的 AI 信息能够第一时间占据屏幕并方便回看，同时也赋予了用户极低成本的“恢复出厂展示态”的控制权。

## 3. 现阶段技术可行性：硬件唤醒手机 App
“通过硬件长按语音输入，唤醒休眠中的手机 App 并调用网络” 在现阶段的技术栈中是**完全可行**的，但需要针对 iOS 和 Android 采用不同的底层实现策略：

### iOS 端的破局方案
iOS 对后台应用的限制极其严格，普通的 BLE (低功耗蓝牙) 通信很难在 App 被杀掉或深度休眠时唤醒它去执行耗时的网络请求（Agent 调用）。现阶段成熟的解决方案有两种：
1. **蓝牙音频协议伪装 (HFP / A2DP)**：将硬件同时注册为 BLE（传信）和传统蓝牙耳机（传声）。当用户按下按钮时，硬件向手机发送一个“播放/录音”的媒体控制指令，这会以极高的系统优先级唤醒 App 的后台音频进程。App 在后台悄悄录制并上传音频，完成后再通过 BLE 通道把图片切片发回给硬件。
2. **苹果 ANCS 协议与静默推送**：利用 Apple Notification Center Service，结合 BLE 的 `bluetooth-central` background mode。硬件端发送一个特定的特征值变化，App 被系统短暂唤醒（通常有 10-30 秒的执行窗口期），利用这段时间快速完成语音转文本、大模型调用和位图渲染并下发。

### Android 端的实现
Android 端相对宽容，通过建立一个绑定的 **Foreground Service (前台服务，带有一个常驻通知栏)** 维持 BLE 长连接，当硬件按下时，立刻通过蓝牙回调激活 Service，进而调用各项 API。

**结论**：在软硬件联调阶段，这属于核心的“技术攻坚点”，但业内已有如各大 AI 录音笔（如 Plaud）等成熟的落地案例，逻辑上完全跑得通。
