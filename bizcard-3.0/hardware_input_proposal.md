# 硬件长按输入方案评估：入口区分与 App 端呈现

在 BizCard 2.0 中，E-ink 硬件的“长按录音”承载了两种核心的交互场景：
1. **闪念记录 (Voice Memo)**：用户旨在“保存”一段信息或灵感。
2. **AI 语音指令 (AI User Input/Ask Agent)**：用户旨在向系统下达执行命令（如“查询某人”、“发个邮件”）。

为了在 App 端优雅地呈现并做好兜底，我们面临两个核心的架构选择。

---

## 问题一：App 端如何呈现“指令历史”？

### 方案 A：独立新建 Command History 页面 (当前实现方案)
将来自硬件的所有纯指令（如查询名片）单独存放在一个类似系统日志的时间线页面中。

- **优势**：逻辑清晰，与 App 端原有的 Ask Agent 聊天流解耦。作为“硬件专属的影子日志”，非常适合排查问题和追溯纯硬件操作。
- **劣势**：打破了用户“只有一个全能 AI 助理”的认知。用户如果想在 App 上接着刚才在硬件上的查询继续深挖，会发现必须跳出历史记录页重新对 Ask Agent 发问。

### 方案 B：延用 Ask Agent 的 Chat Thread (推荐方案)
将硬件发起的语音指令，直接视作用户在 App 端的 `Ask Agent` 中发出的语音消息。直接将其无缝插入到现有的 Agent Chat Thread 中。

- **工作流设想**：
  1. 用户在硬件长按说：“查询冯总名片”。
  2. 硬件获取并在 E-ink 上刷屏展示冯总名片。
  3. 后台默默将这条请求（`user:`）与结果卡片（`agent:`）压入 App 的 `Ask Agent` 聊天历史中。
  4. 当用户掏出手机打开 BizCard App，进入中间的 Ask Agent Tab，能看到完整的聊天上下文，甚至可以直接通过键盘补充一句：“帮我把这份资料通过微信发给他”。
- **结论**：**极度推荐采用方案 B**。这实现了真正意义上的“多端统一智能体”。不管是从硬件入口进来的声音，还是软件入口进来的文字，都汇聚在同一个 Chat Thread 中，兜底体验最好。

## 问题二：在意图识别上，用 AI 判别还是硬件按键隔离？

目前采用的是通过 ASR 结合 LLM 的意图路由器 (`Intent Router`) 来动态判定用户是想“存闪念”还是想“下指令”。这会带来一定的时间延迟和误判率。

### 探讨：是否要在硬件上物理区分？

#### 物理区分方案：
通过交互手势直接定调：
- **短按 / 单击侧键**：触发 **Voice Memo 模式**。录完直接存为 Note，不走指令判断。
- **长按侧键 (类似对讲机)**：触发 **Ask Agent 模式**。直接对接后端的系统级 Tools 流。

#### 优劣势对比：
1. **交互心智**：硬件物理区分对于重度商务用户来说非常清晰（类似苹果的 Action Button 区分长短按），可以做到极高的确定性，**消除了 AI 误判导致“本想存灵感结果去查了人”的尴尬**。
2. **系统延迟**：省去了大模型前置的 Intent 分析耗时（至少节省 1-2 秒），下指令的响应速度会直线上升，更符合 Copilot 的即时性要求。
3. **软硬协同**：这要求硬件固件 (Firmware) 能够识别并发送不同的 BLE 头部标识（如 `FLAG_MEMO` vs `FLAG_CMD`）给手机。

### 综合建议策略
1. **短期 (当前版本)**：维持 `Intent Router` 软分发逻辑，因为硬件可能还无法提供复杂的按键手势，先跑通闭环。
2. **长期 (硬件量产前)**：在硬件层面物理区分（如**单按存闪念，长按唤醒AI**），让输入性质在发生的第一秒就绝对明确。在 App 端，闪念落为 `Note`，AI唤醒落入 `Ask Agent Chat Thread`。这样体验最干脆、最丝滑。
