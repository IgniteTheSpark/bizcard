# 实体兜底关联 (Entity Fallback) 交互与流程说明

在会议或闪念记录经过大模型（LLM）提取待办事项后，往往会包含原话中的“称呼指代”（例如：“下午发文件给**冯总**”、“明天找**Kevin**确认进度”）。

为了减轻用户必须进入“逐字稿关联页”手动打标的负担，我们引入了 `EntityFallbackBinder`。它会在数据展示给前端前，做一次拦截与预判，尝试将这些指代与用户的 CRM 系统进行静默绑定。

## 实体匹配流转逻辑图

以下流程图展示了从大模型提取文本，到后端计算置信度，再到前端采取不同策略兜底的完整过程。

```mermaid
flowchart TD
    A[大模型 LLM] -->|返回 JSON<br>包含: related_mentions: ['冯总']| B(Meeting Processor)
    
    subgraph Entity Fallback Binder [实体兜底引擎 (Backend / App Core)]
        B --> C{遍历 related_mentions}
        C -->|提取 '冯总'| D[比对用户 CRM 联系人库]
        
        D -->|计算匹配得分| E{置信度分层 Score}
        
        E -->|Score >= 90<br>精确匹配/唯一联系人| F[HIGH 置信度]
        E -->|70 <= Score < 90<br>同姓氏多人/职称模糊| G[MEDIUM 置信度]
        E -->|Score < 70<br>无记录| H[LOW 置信度]
        
        F --> I[静默绑定: 将 '冯总' 替换为真实的 Contact_ID]
        G --> J[保持原占位符, 附加 suggested_id 推荐字段]
        H --> K[保持原占位符, 无推荐]
    end

    I --> L(落库保存 Note & Actions)
    J --> L
    K --> L
    
    L --> M[App 前端界面渲染]
    
    subgraph UI Presentation [UI 呈现与交互]
        M -->|HIGH 匹配| N[UI: 直接渲染蓝色高亮链接<br>[👤 冯建国]]
        N --> O(无需用户操作，已完美关联)
        
        M -->|MEDIUM 匹配| P[UI: 渲染黄色问号标签<br>[❓ 冯总]]
        P -->|用户点击| Q[弹出认领界面: 顶部默认选中推荐人'冯建国']
        Q -->|用户一键确认| R(完成关联)
        
        M -->|LOW 匹配| S[UI: 渲染红色问号标签<br>[❓ 冯总]]
        S -->|用户点击| T[弹出认领界面: 空白状态]
        T -->|手动搜索 / 新建联系人| R
    end
```

## 关键交互收益
1. **HIGH 置信度 (静默绑定)**：对于具有极高辨识度的名字或小池子（仅20个联系人）的用户，系统做到了**完全无感**的智能化体验，待办被自动分发到对应联系人的动态下。
2. **MEDIUM 置信度 (半自动兜底)**：不越俎代庖（避免把机密文件错误关联给了同姓的另一个人）。UI 依然提示用户这存在不确定性，但**极大地缩短了确认路径**，从“打字搜索”变为了“一键勾选确认”。
3. **闭环演进**：如果用户在 LOW/MEDIUM 状态下进行了手动修正关联，系统会记录这次映射（例如 `冯总` -> `ID:102`），以提升下一次相同称呼的打分权重。
