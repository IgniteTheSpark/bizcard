# BizCard 2.0 软硬件协同与 App 重构执行计划

本计划基于前期的需求梳理与产品设计文档，梳理了 BizCard 2.0 的核心开发任务。各任务围绕“资产降噪、结构泛化、硬件协同、多方集成”展开，为团队对齐和任务分配提供清晰的指引。

## 一、核心数据与页面重构

### 1. `Notes` 泛化模型重构
*   **目标**：将原有的 `Meeting` 数据模型泛化升级为 `Notes` 体系，以兼容由 E-ink 硬件及 App 端带来的多样化记录输入。
*   **核心功能点与逻辑**：
    *   构建基础的 `BaseNote` 结构（包含通用属性如时间、地点、标题等）。
    *   实现基于 `BaseNote` 的多态拓展，在底层和接口层面清晰区分为 `MeetingNote`（包含会议录音、纪要等特有字段）、`VoiceMemoNote`（包含闪念语音、AI总结等字段）和 `ManualNote`（手动输入文本）。
    *   在前端列表及详情页中，统一使用泛化的接口拉取数据，并基于类型标识动态渲染对应的 Icon 与特有展示区块。
*   **参考文档**：`02_NOTES_ARCHITECTURE.md`

### 2. 首页 (Home) 全新布局升级
*   **目标**：弱化个人名片的视觉层级，将首页的主视觉让渡给用户的三大核心工作资产：Contacts（人脉）、Notes（记录）和 Reminders（待办）。
*   **核心功能点与逻辑**：
    *   将原本占据大面积的个人信息与硬件状态，收敛至顶部的极简悬浮模块中，并新增系统的全局通知入口（小铃铛）。
    *   在页面核心区域依次排布横向滚动的近期 Contacts 列表、垂直的信息流（Notes & Meetings）以及近期的 Reminders 列表，帮助用户进入 App 即可快速聚焦关键信息。
    *   重组底部导航栏结构，移除冗余的 Messages 入口，并将 `Ask Agent (🤖)` 进行居中视觉强化。
*   **参考文档**：`01_HOME_REDESIGN.md`, 演示: `home-demo.html`

## 二、统一日程与时间线管理

### 3. 日历页面 (Calendar) 聚合重构与多方集成
*   **目标**：将 Calendar 页面从单纯的日期网格，升级为一个聚合用户“全生命周期数据”以及“多方工作日程”的统一化时间管理工具 (Unified Calendar)。
*   **核心功能点与逻辑**：
    *   **页面交互重构**：顶部采用单行的周视图日历条替代传统的整月网格。选中具体日期后，下方以“Daily Digest (每日摘要)”的形式展示当日的纵向信息流，同时包含当日要处理的**待办 (Due Reminders)**和当日产生的**记录 (Happened Notes)**。
    *   **Unified 第三方日程集成**：打通外部工作生态，开发授权接入模块。支持用户绑定并同步诸如钉钉 (DingTalk)、飞书 (Feishu)、Google Calendar、Outlook 等第三方日历系统的日程数据。
    *   **数据聚合展示**：在 Daily Digest 中，原生资产（Notes/Reminders）与第三方拉取的外部日程将被统合在同一个时间线中，让 Calendar 真正成为用户全局工作的掌控中枢。
*   **参考文档**：`07_CALENDAR_RESTRUCTURE.md`, 演示: `calendar-demo.html`

## 三、E-ink 硬件交互与智能链路

### 4. 硬件语音意图分发 (Intent Router)
*   **目标**：精准识别 E-ink 硬件长按输入的语音内容，自动将其分发到对应的处理工作流（记录沉淀或指令执行），减少用户的二次确认操作。
*   **核心功能点与逻辑**：
    *   在完成基础的语音转文字 (ASR) 之后，串联大模型进行意图判定。系统需自动区分当前输入是 `COMMAND`（如：查询某人电话、安排行程）还是 `MEMO`（如：记录灵感、生成代办）。
    *   **闪念流水线**：若判定为 `MEMO`，则走总结与抽取逻辑，自动生成 `VoiceMemoNote` 存入数据库，并解析其中包含的潜在待办事项。
    *   **指令流水线**：若判定为 `COMMAND`，则直接将文本发送至 Ask Agent 系统，由 Agent 执行查询或操作，并准备结果。
*   **参考文档**：`03_INTENT_ROUTER_AND_HISTORY.md`, `hardware_input_proposal.md`

### 5. 统一的 AI 指令交互历史
*   **目标**：确保用户通过硬件发起的语音查询及 AI 的答复，能够在 App 端得到妥善沉淀和回溯，保持多端交互体验的一致性。
*   **核心功能点与逻辑**：
    *   复用并拓展 App 现有的 `Ask Agent` 聊天界面，使其成为所有 AI 交互的统一承载池。
    *   当 Intent Router 判定硬件输入为指令、且完成处理后，系统需将该轮对话以特定的视觉样式静默同步插入到 Ask Agent 的聊天流中，标明其来源为“硬件输入”，方便用户在手机端翻阅或继续追问。
*   **参考文档**：`hardware_input_proposal.md`

### 6. E-ink 离屏渲染与蓝牙同步
*   **目标**：实现“App 渲染排版 -> 蓝牙传输 -> 硬件展示”的完整技术闭环，使硬件屏幕能根据 AI 指令或系统通知实时更新。
*   **核心功能点与逻辑**：
    *   **App 端离屏渲染引擎**：利用 Canvas 技术在后台静默将需要展示的内容（如排版好的文本、联系人名片码）绘制为图片，并通过抖动算法 (Dithering) 转化为符合 E-ink 硬件规格的 1-bit 黑白位图。
    *   **蓝牙分片传输协议**：封装与硬件的 BLE 通信通道，能够将生成的位图数据切分为合适大小的 Chunk 依次发送，处理好重传与状态管理，并在数据传输完毕后发送屏幕刷新指令。
*   **参考文档**：`04_EINK_RENDER_AND_BLE_SYNC.md`, `eink_logic.md`, 演示: `hardware-manager-demo.html`

## 四、智能化业务场景增强

### 7. 会议实体的自动兜底绑定
*   **目标**：大幅降低用户在会议或录音结束后，需要手动将记录关联至特定 CRM 联系人的操作成本。
*   **核心功能点与逻辑**：
    *   接收会议 Processor 中由 LLM 抽取的会议参与人/提及人物列表 (`related_mentions`)，与用户手机端已有的 Contacts 数据库进行多维度模糊匹配（全名匹配、拼音匹配、公司名/职位组合比对）。
    *   建立基于置信度的动作分级机制：对于极高置信度的实体（如全名完全吻合）执行后台静默关联；对于中等置信度的实体（如只有姓氏或职位吻合），在 UI 面板提供“一键建议绑定”的快捷操作卡片供用户确认。
*   **参考文档**：`05_ENTITY_FALLBACK.md`, `entity_fallback_flow.md`
