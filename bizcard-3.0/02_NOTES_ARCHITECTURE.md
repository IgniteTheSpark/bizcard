# 02 泛化实体：Notes 体系架构 (Notes Architecture)

## 设计背景
在早期的 BizCard 设计中，系统以 `Meeting` 作为核心的数据载体。然而，随着 E-ink 硬件长按“闪念”功能的加入，以及用户手动记录诉求的增加，单一的 `Meeting` 模型已无法满足多模态、多场景的信息沉淀需求。

为了统一管理所有信息输入来源，我们将底层的 `Meeting` 实体泛化升级为 `Note` 体系。

## 数据模型演进

我们将原有的 `meetings` 数据表重构为广义的 `notes` 实体，并通过 `type` 字段进行场景区分。同时，所有关联外键（如 Action/Reminder 上的关联 ID）也从 `meetingId` 统一变更为 `noteId`。

### Note 的四大核心类型

#### 1. `MEETING` (会议/交谈)
- **触发场景**：下拉 E-ink 硬件屏幕，露出全向麦克风进行的双边或多边长录音。
- **关联人 (Contacts)**：通常关联多个参与者（从日历拉取或会后识别绑定）。
- **后续动作**：由 AI 提取针对多人的 Action Items / Reminders，并生成会议摘要。
- **UI 标识**：蓝色 `🎙️` 图标。

#### 2. `VOICE_MEMO` (闪念)
- **触发场景**：上推 E-ink 硬件屏幕（骨传导/定向麦模式），长按快捷键进行的单边短语音输入。
- **关联人 (Contacts)**：默认关联为 `self` (自己)。用户也可以手动添加提及的人。
- **后续动作**：由 AI 快速总结想法，提取专属个人的待办。
- **UI 标识**：紫色 `💭` 图标。

#### 3. `MANUAL` (手动记录)
- **触发场景**：用户在 App 端直接打字或无录音环境下的随手记。
- **关联人 (Contacts)**：手动绑定。
- **后续动作**：无 ASR 与自动摘要流程。
- **UI 标识**：橙色 `⌨️` 图标。

#### 4. `IDEA` (想法)
- **触发场景**：用户基于一条或多条 Note（会议/闪念）主动发起，由 AI 生成的衍生产出（方案、报告、调研等）。
- **关联人 (Contacts)**：可继承来源 Note 的关联人，或手动绑定。
- **后续动作**：可编辑、再生成；可选「从 Idea 创建 Reminder」。
- **UI 标识**：绿色 `💡` 图标。Ideas 在首页与列表中以 Note 的一种类型展示，类型筛选为「全部 / 会议 / 闪念 / 想法」。
