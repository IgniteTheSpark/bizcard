# Agent 主动简报与首页承载设计（讨论稿）

## 0. 术语澄清（重要）

此前文档曾把「谁更容易拓展 Agent 主动能力」与 **PRD 里的方案1 / 方案2** 混在了一起，需要拆开：

| 名称 | 指什么 | 不要混淆 |
|------|--------|----------|
| **首首页 UI 方案（截图）** | 你提供的**两套高保真首页视觉/布局**，代表两种**首屏信息优先级** | 见下文「设计 A / 设计 B」 |
| **PRD 方案1（Notes First）** | `PRD_OPTION_1_NOTES_FIRST.md`：主区域仅 Notes，Calendar 独立页 | 是**信息架构**，不等于某一张截图 |
| **PRD 方案2（Tab Workspace）** | `PRD_OPTION_2_TAB_WORKSPACE.md`：首页 Notes / Calendar **双 Tab** | 同上 |

**结论**：截图里的两套设计 = **两种首页 UI 范式**；PRD 方案1/2 = **Calendar 是否在首页用 Tab 承载**。二者可以**组合**（例如：Notes First + 截图 A 的 Ask 大标题，或 Notes First + 截图 B 的简洁 Notes 首屏）。

下文 **§2** 专门讨论：**在「截图的两套首页」下**，谁更适合承载今日简报 / EOD 总结 / 待办预准备。  
**§3** 仍保留「Agent 简报区」的通用设计建议（位置、形态），可落在任一套 UI 上再微调。

---

## 1. 背景与目标

Ask-agent 不仅支持**被动**「Ask me anything」，还将支持**主动**向用户推送：

| 类型 | 说明 | 典型场景 |
|------|------|----------|
| **今日简报 (Daily Briefing)** | 当天要关注的事：日程、待办数量、会前提要等 | 早晨打开 App 时 |
| **当日总结 (EOD Summary)** | 今天完成了什么、新增了哪些笔记/待办 | 傍晚或睡前回看 |
| **待办预准备 (Todo Prep)** | 针对某条待办或某场会议的准备建议、上下文摘要 | 会前或处理某条待办前 |

## 2. 两套「截图首页」与 Agent 主动内容的拓展性

下面用**设计 A / 设计 B** 对应你提供的两张图（描述来自设计稿，非 PRD 方案编号）。

### 2.1 设计 A：Ask / Agent 占首屏核心区（「How Can I help you?」+ 大输入条）

- **特征**：顶部大块 **Agent 入口**（标题 + 高亮搜索/语音条），下方才是 Notes（All / Meeting / Memo + 列表）；底部可能为 Record FAB 等。
- **与主动简报的关系**：首屏**已经划给「人与 Agent 对话」**。主动内容（今日简报、EOD、会前预准备）可以：
  - **融入同一块「Agent 面」**：例如在大标题与输入条之间插入**简报卡**、或输入条占位文案随场景变为「今日 3 条待办 · 点我追问」；
  - **少开一个「新模块名」**：用户心智是「上面整块都是 AI」，被动问与主动推**共用同一视觉领地**。
- **拓展上的特点**  
  - **利**：主动推送与 Ask **同区生长**，不必在 Notes 区再抢一块显著位置；动效/品牌上容易做成「一个 Agent 中枢」。  
  - **注意**：需区分 **「系统推给你的简报」** 与 **「你来提问」**，避免用户以为只有搜索、看不到今日摘要；可用**简报卡 + 保留输入条**的结构明确双层（推 / 问）。

### 2.2 设计 B：Notes 为视觉主角（大标题 Notes + 分段列表 + 底部 Record / Ask）

- **特征**：首屏**主标题是 Notes**，类型筛选为横向 chips；列表按「本周 / 更早」分组；**Ask** 多在底部与 Record 并列，首屏**不占大块 Agent 区**。
- **与主动简报的关系**：主动内容**不会自然出现在现有结构里**，需要**明确新增一块承载区**，典型位置：
  - **Notes 标题与筛选行下方**、**「本周」分组上方**；或  
  - **Top Bar 与 Notes 标题之间**的一条简报带（需控制高度）。
- **拓展上的特点**  
  - **利**：**笔记流与「AI 推送给你的信息」边界清晰**——列表仍是「我录下来的东西」，简报是「助理今天让我先看什么」；产品规则、埋点、空态都更好讲。  
  - **注意**：多了一块 UI，要定义**折叠/高度**，避免挤占首屏 Note 预览；与底部 **Ask anything** 要文案上闭环（「展开详情 → 进对话」）。

### 2.3 谁「更容易」拓展？（在截图语义下）

- **不是简单的「A 比 B 容易」或反过来**，而是**拓展方式不同**：
  - **设计 A**：更像**扩展既有 Agent 英雄区**——工程与叙事上「少一个新区域」，但要做好**推 vs 问**的层次。  
  - **设计 B**：更像**新增一个标准区块**——多一次布局决策，但**Notes 与 Agent 推送的职责分离**更干净，和 PRD 里「首页主路径是记录与回看」也更容易对齐（简报是「加料」，不替代列表主角地位）。

- **若产品强诉求是「进 App 先看 AI 今天要说什么」**：设计 A 的骨架更顺手。  
- **若产品强诉求是「进 App 先看笔记，简报是辅助」**：设计 B + 明确「简报区」更一致。

## 3. Agent 简报区 — 通用设计建议（两套 UI 均可落地）

以下与 PRD 方案1/2 **独立**：无论最终选截图 A 还是 B，都可参考。

### 3.1 位置（按设计微调）

- **设计 B（Notes 主角）**：Notes 标题行（含筛选）**下方**、**「本周」上方**，与列表同一条纵向滚动流。  
- **设计 A（Ask 主角）**：放在 **「How Can I help you?」与输入条之间**，或**输入条下方、Notes 区块上方**；与输入条组成「简报 → 追问」动线。

### 3.2 内容形态（按场景）

- **今日简报**：1–2 行 + 「查看详情」；可含待办数、下一场会议、会前提要入口。  
- **当日总结**：晚间场景；今日完成/新增摘要。  
- **待办预准备**：可并入简报一条，或会议将开始时单独小卡。

同一时间只突出**一种**为主，避免首屏多块 Agent 卡片。

### 3.3 交互与空间

- 默认**可折叠**（一行标题 + 关键数字）；无内容时**弱占位或隐藏**。  
- 与 Ask 闭环：卡片上「追问 / 在对话里展开」与底部 Ask 入口一致。

### 3.4 与 PRD「方案1 / 方案2」的关系（补充）

- **PRD 方案2（双 Tab）**若采用：Agent 简报仍建议**全局出现在 Tab 之上**或**固定在 Notes Tab 内**，避免 Calendar Tab 用户完全看不到简报——这与**截图 A/B 是正交问题**，需单独拍板。

## 4. 结论与下一步

1. **术语**：你提供的**两张截图 = 两套首页 UI（设计 A / 设计 B）**，**不等于** PRD 里的 Notes First vs Tab Workspace；后续讨论请分开说「截图方案」与「PRD 方案」。  
2. **主动简报**：两套截图**都能承载**；A 偏向**在现有 Agent 英雄区里长出来**，B 偏向**在 Notes 流上增加一块明确简报区**。  
3. **下一步**：在定稿「采用截图 A 还是 B（或融合）」后，把 §3 的位置与文案落到具体线框图；若需要，可在 `PRD_OPTION_1_NOTES_FIRST.md` 中增加一句「首页视觉可参考设计 A/B，与 Calendar 是否 Tab 无关」。
