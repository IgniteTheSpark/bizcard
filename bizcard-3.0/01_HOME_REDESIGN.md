# 01 首页重构与信息层级倒转 (Home Redesign)

**修订记录**：与当前高保真线框对齐 — Top bar 下增加**导航型 Ask 条**（进入 Ask 专页）；首页**底部悬浮仅保留 Add note + 上传「+」**；在 **Ask 条与 Notes 之间预留可折叠 Panel** 以扩展 Agent 主动推送。历史表述中「列表末位 Add New 为唯一录音入口」「首页无悬浮录音」已废止，以 §2、§3 为准。

## 1. 设计背景与核心痛点
在 BizCard 的早期版本中，首页的视觉重心完全倾向于“展示自我”。首屏超过 1/3 的面积被巨大的数字名片（Profile Card）和硬件状态连接图（Hardware Card）所占据。

随着产品的演进，尤其是 E-ink 硬件的大量引入，用户的核心诉求发生了转变：
1. **名片的展示频率降低**：用户更多通过硬件去“碰一碰”或者展示 E-ink 屏幕上的二维码来交换名片，App 内打开大屏名片的频率显著下降。
2. **核心资产难以触达**：用户录制的大量会议（Meetings）、产生的闪念（Notes）以及 AI 提取的待办事项（Reminders）被折叠或挤压在屏幕下方，导致用户进入 App 后第一眼无法立刻获知“我今天该干什么”、“我最近见过谁”。

因此，我们对首页进行了彻底的**信息层级倒转 (Information Hierarchy Inversion)**，将“名片与硬件”收敛。同时，因**用户来源以硬件为主**，用户购买硬件后首先用其**录音**，录音产生**总结（Notes）**后才关联人与待办；若不主打「名片 App」，用户对名片无强感知，心智上这是**硬件的录音卡片附属 App**。故首页与主功能的**主要锚点以 Notes 为主**，Contacts 与 Reminders 为配套展示。

**首页信息层级原则**：**首页以 Notes 列表为主区**；顶部为极简控制区；**Top bar 下方提供 Ask Agent 入口（导航型，非内联输入）**；**底部仅保留一条悬浮操作条**（录音主入口 + 上传录音文件）；不在首页内嵌完整对话或 chat history。Reminders / Contacts 等不作为首页主列表区块（与信息架构方案一致时可放在其他 Tab 或详情内）。

## 2. 首页交互：纵向结构（Header → Ask 条 → 可扩展区 → Notes）+ 底部悬浮双入口

### 2.0 整体布局（与当前线框一致）

自上而下：

1. **Top bar（极简 Header）**  
   身份入口（头像 / 占位）、硬件连接状态（如「+ Connect」/「Connected」）、以及次要图标入口（如文档/卡片类入口）。具体文案与图标以实现稿为准。

2. **Ask Agent 入口条（导航型）**  
   - **位置**：紧挨 Top bar **下方**，全宽或近全宽的**胶囊形态**（类似搜索条），左侧可配品牌色 AI 图标，主文案如 **「Ask me anything…」**。  
   - **关键行为**：**不是**在首页展开键盘或内联输入；用户**点击整条（或主区域）即跳转/ present 至独立的 Ask Agent 页面**（与 `bizcard-askagent` PRD 中「进入即新对话 / 侧边 history」的专页一致）。  
   - **与底部 Tab**：若底部仍有居中的 Ask Tab，则首页条为**快捷第二入口**，动线一致（都进同一套 Ask 体验），避免两套不同对话栈。

3. **（预留）Agent 主动推送区 — 可折叠 Panel**  
   - **位置**：**Ask 入口条与 Notes 区块标题之间**。  
   - **v1**：可先不展示或仅占位高度，避免挤压 Notes 首屏。  
   - **扩展**：后续在此插入**可折叠 Panel**，承载今日简报、会前摘要、EOD 等**主动推送**（见 [`AGENT_PROACTIVE_BRIEFING_DESIGN.md`](./AGENT_PROACTIVE_BRIEFING_DESIGN.md)）。折叠态一行标题 + 关键数字/角标，展开态列表或卡片；与 Ask 专页闭环为「简报内 CTA → 带上下文进入 Ask」。

4. **Notes 区块**  
   - **主滚动区**：大标题 **「Notes」** + 筛选（如 All / Meeting / Memo），列表按「这周 / 更早」等分组。  
   - **展示**：按时间倒序的 Notes，标题、时间、时长、类型与 Reminders 等标签。  
   - **交互**：点击某条 → 进入详情。  
   - **「查看全部」类入口**：以实现为准（如区块右上角进入历史/日历）。

**底部悬浮条（仅此一条，双按钮）**

- **不设**：独立的「Scan to Add」大按钮、与录音并列的第三枚悬浮主按钮（旧版双胶囊若已弃用则不再写进需求）。  
- **保留两项**：  
  1. **Add note（主按钮）**：大胶囊、带麦克风图标，**主录音入口**——点击即**开启录音**（与硬件或 App 内录逻辑一致），录完生成新 Note。  
  2. **「+」**：圆形次要按钮，**添加入口**——用于**上传已有录音文件**（导入后走同一套转写/生成 Note 流程）。  

可选：若产品仍希望列表内弱引导新建，可在列表底部保留轻量「新记录」行，但**主路径以底部悬浮「Add note」为准**（与当前设计对齐）。

### 2.1 Notes 列表

- **展示**：按时间倒序的 Notes（会议 / 闪念 / 手输等），每条带类型标签、标题、时间、Reminders 数量等。
- **交互**：点击某条 Note → 进入该条详情（或展开）。

### 2.2 录音与上传的主路径

- **录音**：优先通过底部悬浮 **Add note** 触发。  
- **上传**：通过悬浮条 **「+」** 选择本地录音文件导入。  
- （可选）列表末位 **Add New** 卡片：若保留，则与 Add note 等价或作为弱入口，需在实现上与底部主按钮二选一或并存时避免重复打扰。

## 3. 核心模块改动点详解（与上述布局对应）

### 模块一：Top bar / Minimal Header (极简控制中枢)
将身份与硬件状态收敛至屏幕最顶部一行，避免早期「大 Profile + 大硬件卡」占首屏。

- **左侧 (Identity)**：头像（未上传时为占位图）、点击进入 Profile / 个人相关页面。
- **中部 / 右侧**：**硬件连接状态**（如未连接时「+ Connect」药丸按钮，已连接时「Connected」+ 状态点）；以及文档/卡片等次要入口（以实现稿为准）。
- **通知**：若仍保留消息/通知，可放在 Top bar 末端或并入 Me，与 `DISPLAY_AND_CONTRACT.md` 中「Messages 收敛到 Me」的规划对齐。

### 模块二：Ask Agent 入口条（Top bar 下方）
- **形态**：全宽胶囊条 + AI 图标 + 占位文案（如「Ask me anything…」）。
- **行为**：**仅负责导航**——点击进入 **Ask Agent 专页**（非首页内联输入框）。与底部 Tab 中的 Ask 进入同一套会话体验。

### 模块三：（预留）可折叠 Panel — Agent 主动推送
- **位置**：**Ask 入口条与 Notes 大标题之间**（见 §2.0）。
- **用途**：后续承载简报、会前准备、EOD 等；默认可折叠，避免与 Notes 抢首屏。
- **与 Ask 闭环**：Panel 内「追问 / 查看详情」跳转 Ask 专页并携带上下文（deep link 或预填意图）。

### 模块四：📝 Notes & Meetings (泛化资产流) — **首页主列表**
**主锚点**，在 Ask 条（及预留 Panel）之下，占据滚动区主体。

- **视觉呈现**：区块标题 **Notes** + 筛选 chips（All / Meeting / Memo 等）+ 按时间分组的列表。
- **展示逻辑**：列表可限制首屏条数并提供「查看全部」跳转 Calendar / 历史（条数以产品为准）。
- **内容类型**：如 [02_NOTES_ARCHITECTURE](./02_NOTES_ARCHITECTURE.md)，混合展示会议 / 闪念 / 手输等，用图标或标签区分。
- **信息强化**：时间、时长、**Reminders** 等标签（如 `2 Reminders`）提醒未处理待办。
- **空数据 / Mock**：无数据时策略见 §4；引导文案优先指向底部 **Add note** 录音。

### 模块五：底部悬浮条 + Bottom Navigation
- **悬浮条（首页内）**：**仅两项** — **Add note**（主录音）与 **「+」**（上传录音文件）。叠在列表底部安全区上方，不增加第三枚并列主操作。
- **Tab Bar**：
  - **Messages**：建议收敛到 Me 或 Top bar，空位给 **Ask Agent**（中间突出）。
  - **Ask Agent**：底部居中主入口之一；与首页 Ask 条互为补充动线。
  - 其余 Tab：Home、Contacts、Calendar、Me 等按产品规划保留。

## 4. 首次进入/无数据体验（减轻沉闷）与运营位

### 4.1 问题与目标

- **问题**：用户刚购买硬件、首次进入 App 时没有任何 Notes/Reminders/Recent Contacts，首屏若仅展示三个「空区块」会显得**沉闷、冷清**，且无法建立「这个 App 能帮我做什么」的预期。
- **目标**：在无数据状态下，首屏仍应**有温度、有引导、有一句价值感**，并预留**运营位**用于新用户引导、活动或技巧推送，避免「空白列表 + 空洞标题」的体验。

### 4.2 无数据时：直接展示 Mock 笔记、Mock 联系人、Mock 待办

**与当前首屏模块对齐**：若首页已收敛为 **Header + Ask 条 + Notes**（无首屏 Contacts / Reminders 整区），则无数据 Mock **至少覆盖**：示例 Notes 列表、Ask 条可保留默认文案；**Recent Contacts / Reminders 的 Mock 可放到对应 Tab 或首次引导页**，避免文档表格与真实首屏不一致。

**结论**：新用户刚进来时**不展示空状态**，而是**直接展示 Mock（示例）数据**：示例 Notes、（若仍在首屏则）示例 Contacts、示例 Reminders，与有数据时的形态一致。用户一眼就能看到「有数据时首页长什么样」，更直接、减少冷清感；再配合一句轻量说明（如「以下为示例，录下第一条或添加联系人后将显示你的内容」），明确预期。

#### 无数据时的首屏结构（推荐）

| 区域 | 内容 | 说明 |
|------|------|------|
| **Minimal Header** | 保持现有 | 身份 + 硬件状态 + 通知。 |
| **轻量欢迎 + 示例说明** | 无数据时在 Header 下放一条**轻量**文案：欢迎 + 一句价值 + **「以下为示例数据，录下第一条或添加联系人后将显示你的内容」**（或「让你先看看有数据时的样子」）。高度约 60–80px。 | 建立预期，并标明当前为示例。 |
| **👥 Recent Contacts** | Section 仍存在，内容区展示 **Mock 联系人**（2–4 条示例卡片，如示例姓名、时间戳），最后一张可为「Add New」入口。 | 形态与有数据时一致，用户直接看到联系人流的样式。 |
| **📝 Notes & Meetings** | Section 仍存在，内容区展示 **Mock 笔记**（2–3 条示例 Note：会议/闪念/手输，带类型图标、标题、时间、Reminders 标签）。 | 用户直接看到「记录 → 人 → 待办」的卡片长什么样。 |
| **✅ Reminders** | Section 仍存在，内容区展示 **Mock 待办**（2–3 条示例待办，带溯源会议名）。 | 用户直接看到待办列表与溯源信息的样式。 |

**原则**：无数据 = **同布局 + Mock 数据**，不出现空状态卡片。用户首次进入即看到「满的」首屏；通过顶部一句说明知悉为示例，录下第一条或添加联系人后自动切换为真实数据（可由后端根据「是否已有任意真实数据」切换展示 Mock 或真实内容）。

### 4.3 运营位补充

运营位用于新用户引导、活动、技巧、功能推荐等，需**位置固定、数量可控**，避免冲淡核心资产。

| 位置 | 形式 | 适用场景 | 注意 |
|------|------|----------|------|
| **Header 下方、主内容顶部** | 单条横幅或步骤卡片（可关闭/一次展示） | 新用户：设备连接引导、首次录音引导；大促/活动：限时能力、活动入口。 | 高度控制在 56–80px（横幅）或 80–100px（步骤卡）；老用户可配置为「不再展示」或按条件隐藏。 |
| **Notes 与 Reminders 之间** | 单条信息卡片（如「小技巧」「新功能」） | 技巧类：「长按设备可记闪念」；功能类：「试试对助手说：明天要跟进什么」。 | 一条为宜，可轮播；不喧宾夺主。 |
| **Reminders 下方（列表后）** | 软推广或品牌故事 | 如「更多用法见帮助中心」、会员/高级能力入口。 | 与主内容有分隔，不干扰核心待办。 |

**数量与节奏**：首屏**同时出现的运营位建议不超过 2 个**（例如：顶部 1 条新用户引导 + 中间 1 条技巧/活动）。运营位需支持**后台配置**（文案、跳转、展示条件、目标人群），便于运营替换而不发版。

### 4.4 小结

- **无数据时的展示方式**：**直接展示 Mock 数据**——示例 Notes、示例 Contacts、示例 Reminders，与有数据时布局和形态一致；顶部加一句轻量说明（如「以下为示例，录下第一条或添加联系人后将显示你的内容」）。不展示空状态卡片，新用户一进来就看到「有数据时」的完整样子。
- **运营位**：在 Header 下、区块间、列表后预留固定位置，单屏不超过 2 条，支持配置化，兼顾新用户引导与日常运营。

## 6. 与「进 App 后感知」的对齐自检（参考 CURRENT_KEY_POINTS）

以下对照 [bizcard-business-expansion/CURRENT_KEY_POINTS.md](../bizcard-business-expansion/CURRENT_KEY_POINTS.md) 中「核心卖点与进 App 后的感知」做自检，便于判断首页 demo 是否达到「用户有所感知」的效果。

### 6.1 已具备的效果 ✅

| 落点 | 当前 demo 表现 |
|------|----------------|
| 信息层级倒转 | 名片/硬件已收敛为顶部胶囊，首屏不再被大 Profile 占据。 |
| Notes 可见且带类型 | Notes 区域有会议/闪念/手输三种图标与「2 Reminders」红色 Tag，能传达「记录 → 待办」关联。 |
| Reminders 溯源 | 每条待办下有 `📅 Jan 22 · Product Design Sync` 等溯源锚点，点击可回到会议，体现「待办 ↔ 记录」。 |
| Ask Agent 入口 | **Top bar 下 Ask 条（进专页）+ 底部 Tab 居中 Ask** 双动线一致。 |
| 核心操作外露 | **底部悬浮 Add note（录音）+「+」（上传文件）**；硬件连接在 Top bar 状态。 |

### 6.2 尚不足或可加强的点 ⚠️

| 落点 | 差距 | 建议 |
|------|------|------|
| **首屏顺序** | 当前顺序为：Contacts → Notes → Reminders。共识是「以 Notes 为关键锚点，其次 Reminders」。 | 考虑将 **Notes & Meetings** 提到 **Recent Contacts** 之上**，**使「录下来的变成笔记」第一眼占位更重。 |
| **Note 上「人」的显性** | Note 卡片有标题（如 "Exchange with John Mitchell"），但**未在卡片上显式露出关联联系人**（头像/姓名）。「每条记录都连到对的人」不够一眼可见。 | 在每条 Note 卡片增加一行：关联联系人（如小头像 + 姓名），或至少在有绑定联系人时显示「👤 John Mitchell」。 |
| **一句价值感** | 首屏没有任何一句「这个 App 是干什么的」的文案（如「录下的 · 连到人 · 变成待办」）。新用户 10 秒内可能只看到三个区块，难以形成一句话认知。 | 在 Minimal Header 下方或 Notes 区块上方增加**一句短文案**（可配置/可 A/B）：如「你的记录与待办」或「录下的会变成笔记，自动连到人与下一步」。 |
| **空状态** | Demo 仅有有数据状态，未做空状态。 | 当 Notes/Reminders 为空时，使用**预期设定型**空状态文案（见 CURRENT_KEY_POINTS），例如：「用设备录下的会议与闪念会出现在这里，并自动关联联系人与待办」。 |
| **首页「人+记录+待办」同屏** | 「笔记—人—待办」三条线在首页是**分块**的（Notes 一块、Reminders 一块、Reminders 里有溯源）。没有单张卡片同时展示「这条 Note + 关联人 + 从它出的 Reminders」。 | 首页可保持现状；**Note 详情页 / Contact 详情页** 必须同时露出：该记录 + 关联人 + 从该记录出的 Reminders，以满足「至少一处同屏」。 |

### 6.3 自检三问（当前 demo）

- **用户打开 App 第一眼，10 秒内能否用一句话说出「这个 App 是干什么的」？**  
  → 目前偏难；建议加一句价值文案 + Notes 优先顺序后会有改善。
- **若用户还没有任何录音/笔记，空状态是否传达「将来会有什么、为什么值得留」？**  
  → Demo 未覆盖空状态，需在真实实现中补足预期设定型空状态。
- **用户第一次看到「带 Reminder、带联系人的 Note」时，是否有「原来是这样连在一起的」的感知？**  
  → 部分有（Reminder 溯源到会议名）；若 Note 卡片上显式露出关联人，感知会更强。

**结论**：在**当前线框**下，首页已具备：**Header 硬件态、Ask 导航条、Notes 主列表、底部录音/上传双入口**；主动推送可在 Ask–Notes 间 Panel 扩展。要**完整达到**「进 App 后有所感知」，仍建议补齐：**Note 卡片关联人显性、价值文案、空状态预期、详情页人+记录+待办同屏**；若首屏不再放 Contacts/Reminders 整块，相应 Mock 与运营位应迁移到实际出现的模块或 Tab。
