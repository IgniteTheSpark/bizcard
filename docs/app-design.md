## Problem Statement

录音转录类产品帮助用户把语音变成文字，但文字停留在 App 内供回看，没有变成可执行的行动。用户仍然需要手动把 action item 复制到 Notion、手动创建 Calendar event、手动在 Apple Todo 里建提醒。这套土方法费时费力，失败率高——大量有价值的内容被录下来之后消失在"以后再整理"的黑洞里。

---

## Status Quo（用户现在怎么做）

| 场景 | 当前做法 | 痛点 |
|---|---|---|
| 会议结束后 | 打开转录文本 → 手动复制 action item → 粘贴到 Notion / Todo App | 上下文切换多，容易遗漏 |
| 有想法时 | 语音备忘 / 文字备忘 → 停留在备忘录，不再回看 | 想法沉没，无法执行 |
| 零散记录 | 微信自己发消息、Apple 备忘录 | 分散，找不到，无结构 |

---

## Target Users

学生、内容生产者、商务人士。当前阶段不做差异化，使用统一体验。核心共性：**有大量语音输入需求，需要把口头表达转化为可执行的结构化数据。**

---

## 已确认前提

1. **Session 是主轴，资产有独立入口** — App 以 session 时间流为核心叙事，但 Todo / Idea / Expense / Contact 有独立的 Library 视图供快速访问
2. **硬件是刚需** — 没有硬件录音入口，产品就是另一个 AI 笔记 App。硬件是护城河，不是功能之一
3. **不做用户群差异化** — 当前阶段统一体验，pipeline 和 UI 不按用户类型分流
4. **最大风险是习惯养成和 AI 信任** — 技术已跑通，核心挑战是让用户愿意对着硬件说话，并信任 AI 提炼的结果

---

## 选定方向：Hybrid B+C（流 + 库 + 统一 AI 上下文）

### 核心设计原则

三种输入类型有不同的"形状"，不能强行混在同一个容器里：

| 输入类型 | 特征 | 隐喻 |
|---|---|---|
| 闪念录音 | 短、碎、高频、个人化 | 语音 Twitter |
| 会议录音 | 长、结构化、有明确事件上下文 | 独立的会议室 |
| 文字追问 | 对话式、即时、无录音 | 随时可问的 AI |

解法：**三种输入有视觉上可区分的卡片形态，在同一个时间流里共存，背后共享同一个 AI Agent 上下文和资产库。**

## 核心交互流程

### 闪念输入
```
用户按硬件录音键（"闪念"标签）
    ↓
本地/廉价 ASR 转录
    ↓
Flash Note Pipeline（Dispatcher → 并行 sub-skills → Session Writer）
    ↓
今日流出现新的 [闪念] 卡片，展示提炼结果
    ↓
用户可文字追问修改（"第一个待办改成后天"）
```

### 会议录音
```
硬件录音（"会议"标签）
    ↓
专业 ASR 转录（带说话人分离）
    ↓
Meeting Pipeline（提取行动项 / 参会人 / 决策 / 摘要）
    ↓
今日流出现 [会议] 卡片，可展开看完整内容
    ↓
用户可追问（"刚才那个行动项谁负责？"）
```

### 文字追问
```
用户在任意 session 内打字
    ↓
Conversational Agent（有完整 session 历史）
    ↓
直接调用 MCP 工具执行 / 回答
    ↓
今日流出现 [问答] 条目，或资产被静默更新
```

## 技术架构对应

| App 层概念 | 技术层对应 |
|---|---|
| 今日流中的 [闪念] 卡片 | Flash Note Pipeline 输出的 cards JSON |
| 今日流中的 [会议] 卡片 | Meeting Pipeline 输出（待设计） |
| 今日流中的 [问答] 条目 | Conversational Agent 输出 |
| 资产库 | query_asset / query_contact MCP 工具 |
| 文字追问 | Conversational Agent + session history |
| 卡片跳转 | cards[].action.route（已在 session writer 中定义）|

## 数据模型（已确认）

```
File（音频元数据）→ Transcript（ASR 结果）→ 触发 Pipeline
                                              ↓
Session（App 层容器）←── 聚合 ──── Asset（todo/idea/expense/note）
                                   Contact（独立实体）
```

- File → Transcript：纯技术流程，不依赖 Session
- Session：App 层组织概念，持有 file/transcript/asset 的引用
- 三种 session 类型：`flash_note`（持续累积）/ `meeting`（每次独立）/ `chat`（跨 session 对话层）

## Open Questions

1. **今日流的时间范围** — 只看今天，还是支持"昨天"/ 按周回看？入口怎么设计？
2. **会议卡片展开后的内容深度** — 完整摘要 + 行动项列表 + 参会人？还是需要进入独立的会议详情页？
3. **资产库的主视图** — 默认展示 Todo（最高频），还是让用户自定义？
4. **AI 主动推送** — 是否支持"今天有 3 个待办到期"这类主动提醒？这是 C 方向的核心能力，加入时机待定
5. **Meeting Pipeline** — 还未设计，是接下来最大的技术工作
6. **Conversational Agent** — 还未设计，system prompt 和 session history 管理方案待定

## The Assignment（下一步最重要的一件事）

**找 5 个真实用户，让他们用现有竞品录一段话，然后坐在旁边看他们怎么处理录音结果。** 不要 demo 你们的产品，只观察他们的现有行为。你需要看到：他们有没有手动整理？整理了什么？跳过了什么？这个观察会决定 pipeline 提炼哪些信息是最有价值的，以及今日流卡片应该默认展示什么。
